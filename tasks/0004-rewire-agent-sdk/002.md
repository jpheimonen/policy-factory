# 002: Convert file tools to MCP format

## Context

The four file tools (`list_files`, `read_file`, `write_file`, `delete_file`) are currently defined as Anthropic API format dicts with JSON Schema input definitions, dispatched manually by `AgentSession._execute_tool()`. They need to be converted to `@tool`-decorated async MCP handlers served via `create_sdk_mcp_server()`, following the cc-runner pattern. The tool implementations themselves (the actual file I/O logic) stay unchanged — only the definition format and invocation mechanism change.

This step also introduces the tool context infrastructure (contextvars) needed for MCP handlers to access `data_dir`, and the MCP server factory function that `AgentSession` will call in step 004.

## Reuse Opportunities

- **Tool implementations — 100% reuse**: The four functions (`list_files`, `read_file`, `write_file`, `delete_file`) keep their exact signatures and logic. The MCP handlers call them directly.
- **Sandbox validation — 100% reuse**: `validate_path()` and `SandboxViolationError` stay exactly as they are.
- **`make_result()` pattern from cc-runner**: The cc-runner's `common.py` has a `make_result(success, data, error)` helper that formats results as MCP content blocks. Adopt the same helper to replace `_error_result()` and `_success_result()`.
- **Tool context pattern from cc-runner**: The cc-runner's `_set_tool_context()` / `get_tool_context()` contextvars pattern is adopted directly. For policy-factory, the context needs fewer fields — primarily just `data_dir`.
- **Existing tool tests**: The `TestPathValidation` and implementation test classes (`TestListFiles`, etc.) are 100% reusable since the underlying functions don't change.

## Deliverables

### Tool context infrastructure (in `tools.py` or a new `tool_context.py`)

Add a contextvars-based tool context following the cc-runner pattern:

- A `ContextVar` that holds a dict with at minimum a `data_dir` key
- A `_set_tool_context(data_dir)` function that stores the context for the current asyncio task
- A `get_tool_context()` function that retrieves the context, falling back to a default dict with `data_dir=None` if none has been set
- This context is isolated per asyncio task — concurrent sessions setting different `data_dir` values do not interfere with each other

### MCP result helper

Add a helper function (similar to cc-runner's `make_result()`) that formats tool results into MCP content format. This replaces the old `_error_result()` and `_success_result()` helpers.

The MCP content format is a dict with this exact structure:

```
{"content": [{"type": "text", "text": "<result text here>"}]}
```

- On success: wraps the result data as a text string in the MCP content format shown above
- On error: wraps the error message in the same MCP content format (the error text is the `"text"` value)
- Both success and error results are returned (not raised) — sandbox violations and I/O errors are caught inside the handler and returned as error results

### Four MCP tool handlers

Each file tool gets an async handler decorated with `@tool` from `claude_agent_sdk`:

**list_files handler:**
- Name: `"list_files"`
- Description: preserved from the current `LIST_FILES_TOOL` description
- Input schema: takes a `path` parameter (string type)
- Behavior: retrieves `data_dir` from tool context, calls the existing `list_files()` implementation, returns result in MCP format
- Errors: catches `SandboxViolationError` and other exceptions, returns error results in MCP format

**read_file handler:**
- Name: `"read_file"`
- Description: preserved from the current `READ_FILE_TOOL` description
- Input schema: takes a `path` parameter (string type)
- Behavior: retrieves `data_dir` from tool context, calls the existing `read_file()` implementation, returns result in MCP format

**write_file handler:**
- Name: `"write_file"`
- Description: preserved from the current `WRITE_FILE_TOOL` description
- Input schema: takes `path` (string) and `content` (string) parameters
- Behavior: retrieves `data_dir` from tool context, calls the existing `write_file()` implementation, returns result in MCP format

**delete_file handler:**
- Name: `"delete_file"`
- Description: preserved from the current `DELETE_FILE_TOOL` description
- Input schema: takes a `path` parameter (string type)
- Behavior: retrieves `data_dir` from tool context, calls the existing `delete_file()` implementation, returns result in MCP format

### MCP server factory function

A function (e.g., `create_tools_server()`) that:

- Accepts `data_dir` as a parameter and a tool set identifier to determine which tools to include
- Calls `_set_tool_context(data_dir)` to set the contextvar before server creation
- Selects the appropriate set of MCP tool objects based on the tool set identifier:
  - Full file tools: all four handlers
  - Read-only tools: only `list_files` and `read_file` handlers
  - No tools: returns an appropriate empty/null result
- Raises `ValueError` if an unrecognized tool set identifier is passed (consistent with the `ValueError` pattern used in `resolve_tools()` for unknown roles in `config.py`)
- Calls `create_sdk_mcp_server()` with `name="policy-factory-tools"`, `version="1.0.0"`, and the selected tools
- Returns a dict mapping the server name to the server config (same format cc-runner uses for `ClaudeAgentOptions.mcp_servers`)
