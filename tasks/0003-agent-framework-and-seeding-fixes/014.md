# 014: Add seeding integration tests

## Context

The seeding endpoints orchestrate complex flows: running agents, parsing output, writing files, committing to git, and triggering cascades. Integration tests verify these flows work end-to-end with the agent mocked at the session level, testing the endpoint logic without making actual API calls.

## Related Code

Existing files/patterns to reference or extend:

- `tests/test_seed_api.py` — Existing seed endpoint tests. These may need updating for the new endpoint behaviors and can serve as a pattern.
- `tests/test_integration_cascade.py` — Integration tests for cascade flows. Shows pattern for testing multi-component flows with mocks.
- `tests/test_integration_first_run.py` — Tests for data initialization. Shows pattern for testing startup behavior.
- `tests/test_data_init.py` — Unit tests for data initialization functions.
- `tests/conftest.py` — Contains fixtures for test client, temporary data directories, and auth tokens.

## Reuse Opportunities

- Use existing test client fixtures for making API requests
- Use existing auth token fixtures for authenticated requests
- Follow the integration test patterns from test_integration_cascade.py
- Mock at the AgentSession.run() level to avoid API calls while testing endpoint logic

## Deliverables

- Update or create integration tests for seeding flows:

  - Values seeding flow tests:
    - POST /api/seed/values requires authentication
    - Values seeding creates markdown files in data/values directory
    - Values seeding clears existing values before writing new ones
    - Values seeding returns count of values created
    - Values seeding commits changes to git
    - Values seeding with agent error returns appropriate error response

  - SA seeding flow tests:
    - POST /api/seed/ no longer returns 409 when SA layer has items
    - SA seeding clears existing SA items before writing new ones
    - SA seeding accepts optional context in request body
    - Context is incorporated into the agent prompt when provided
    - SA seeding works without context (backward compatible)
    - SA seeding triggers cascade after completion
    - SA seeding returns item count and cascade ID

  - Seed status tests:
    - Status reflects empty layers after initialization
    - Status reflects populated layers after seeding
    - Status returns correct counts for both layers

  - Data initialization tests:
    - Initialization creates all layer directories
    - Initialization does not write pre-seeded values
    - Values directory contains only README.md after init
    - Initialization is idempotent
