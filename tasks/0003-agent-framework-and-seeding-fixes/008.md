# 008: Fix SA seed endpoint

## Context

The existing SA seed endpoint has two problems:

1. **409 guard prevents re-seeding**: The endpoint returns 409 Conflict if the SA layer already has items, treating seeding as a one-time operation. This made sense for initial setup but blocks iterative refinement. Users should be able to re-seed when the world changes or they want fresh data.

2. **No context input**: The endpoint doesn't accept human-provided context. Users should be able to provide situational context ("Here's how Finland's situation looks right now") to inform the agent's research.

## Related Code

Existing files/patterns to reference or extend:

- `src/policy_factory/server/routers/seed.py` — The file to modify. Contains the `trigger_seed()` endpoint with the 409 guard at lines 92-100 and the `_is_seeded()` helper function.
- `src/policy_factory/data/layers.py` — Contains `list_items()` and `delete_item()` for clearing existing items before re-seeding.

## Reuse Opportunities

- Keep the existing endpoint structure, authentication, agent execution flow, cascade triggering, and response model
- Use existing `list_items()` and `delete_item()` to clear SA layer before re-seeding
- The values seed endpoint (step 007) follows a similar "clear then write" pattern

## Deliverables

- Remove the 409 guard from `POST /api/seed/`:
  - Delete or bypass the check that returns 409 when SA layer has items
  - Before running the agent, clear all existing SA items using list_items/delete_item

- Add context parameter to the endpoint:
  - Accept an optional request body with a `context` field (string)
  - When context is provided, prepend it to the seed prompt so the agent incorporates it alongside web research
  - When context is not provided, behavior is unchanged (backward compatible)

- Update the request model to accept the optional context field

- Keep existing behavior:
  - Authentication requirement
  - Agent execution with web search and file tools
  - Git commit after seeding
  - Cascade trigger after successful seeding
  - Response with success status, item count, and cascade ID

## Acceptance Criteria

- [ ] POST /api/seed/ no longer returns 409 when SA layer already has items
- [ ] Endpoint clears existing SA items before running the agent
- [ ] Endpoint accepts optional context field in request body
- [ ] When context is provided, it is incorporated into the agent prompt
- [ ] When context is not provided, endpoint works as before (backward compatible)
- [ ] Endpoint still requires authentication
- [ ] Endpoint still triggers cascade after successful seeding
- [ ] Endpoint still returns item count and cascade ID
- [ ] Endpoint still commits changes to git
- [ ] Multiple successive calls succeed (re-runnable)
