# 011: Update seed status endpoint

## Context

The existing seed status endpoint (`GET /api/seed/status`) only reports SA layer seeding status. With the new values seeding endpoint, the status should report both layers so the UI can display whether values and SA have been seeded and how many items each contains.

## Related Code

Existing files/patterns to reference or extend:

- `src/policy_factory/server/routers/seed.py` — Contains the existing `get_seed_status()` endpoint and `SeedStatusResponse` model. The endpoint currently checks only SA layer via `_is_seeded()`.
- `src/policy_factory/data/layers.py` — Contains `list_items()` for counting items in a layer.

## Reuse Opportunities

- Use existing `list_items()` function to count items in both layers
- Follow the existing `_is_seeded()` helper pattern but for both layers
- Extend the existing `SeedStatusResponse` model rather than creating a new one

## Deliverables

- Update `SeedStatusResponse` model to include:
  - values_seeded: boolean indicating whether values layer has items
  - values_count: integer count of items in values layer
  - sa_seeded: boolean indicating whether SA layer has items
  - sa_count: integer count of items in SA layer

- Update `get_seed_status()` endpoint to:
  - Check values layer using list_items
  - Check SA layer using list_items (existing behavior)
  - Return combined status for both layers

- Consider removing the `_is_seeded()` helper or generalizing it since the logic is simple (count > 0)

## Acceptance Criteria

- [ ] GET /api/seed/status returns values_seeded boolean
- [ ] GET /api/seed/status returns values_count integer
- [ ] GET /api/seed/status returns sa_seeded boolean
- [ ] GET /api/seed/status returns sa_count integer
- [ ] Status correctly reflects empty layers (both false, counts 0) after initialization
- [ ] Status correctly reflects populated values layer after values seeding
- [ ] Status correctly reflects populated SA layer after SA seeding
- [ ] Endpoint continues to require authentication (existing behavior)
- [ ] Response model is properly documented for API consumers
