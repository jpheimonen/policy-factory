# 006: Observability backend API

## Context

The policy factory already stores full agent output text in the `agent_runs` table (`output_text` column) and links heartbeat tier entries to agent runs via `agent_run_id`. However, none of this data is exposed through the API — the cascade detail endpoint explicitly omits `output_text` from its agent run serialization, and the heartbeat endpoints return tier entries with `agent_run_id` but no way to fetch the associated output.

Without this data in the API, the frontend cannot show what agents actually wrote, making it impossible to diagnose prompt quality, heartbeat behavior, or cascade output. This step adds the backend API changes needed before the UI can be built in steps 007 and 008.

Two changes are needed:
1. Add `output_text` to the existing cascade detail endpoint's agent run serialization (one field addition)
2. Add a new endpoint to fetch a single agent run's full record including `output_text` (for on-demand transcript loading in the heartbeat log viewer)

## Related Code

The `list_agent_runs()` store method already returns full `AgentRun` objects with `output_text` populated — the data is available, just not serialized.

The `_heartbeat_run_to_dict()` helper serializes heartbeat runs including `structured_log` tier entries that contain `agent_run_id`.

- `src/policy_factory/store/agent_run.py` — The `AgentRunStoreMixin` with `get_agent_run(agent_run_id)` returning a full `AgentRun` dataclass including `output_text`. This is the store method the new endpoint will use.

- `src/policy_factory/store/agent_run.py` — The `AgentRun` dataclass with fields: `id`, `cascade_id`, `agent_type`, `agent_label`, `model`, `target_layer`, `started_at`, `completed_at`, `success`, `error_message`, `cost_usd`, `output_text`.

- `tests/test_heartbeat_api.py` — Existing heartbeat API tests. Uses the fixture pattern: `_configure_auth` (autouse), `data_dir` (tmp directories), `client` (TestClient with scheduler disabled), `auth_headers` (admin user + JWT token). Tests are organized by endpoint in test classes. The new endpoint tests should follow this same pattern and can be added to this file.

- `tests/test_cascade_api.py` — Existing cascade API tests. Contains `TestCascadeDetail` with `test_includes_agent_runs` that creates agent runs via `store.create_agent_run()` and verifies they appear in the detail response. - `src/policy_factory/server/deps.py` — Dependency injection for `get_store`, `get_current_user`, etc. Used by all API endpoints.

## Reuse Opportunities

- The `store.get_agent_run(agent_run_id)` method already exists and returns the full `AgentRun` dataclass. The new endpoint just needs to call it and serialize the result — no store changes needed.

- The cascade detail endpoint's agent run serialization dict (lines 336-350) already serializes 10 fields. Adding `output_text` is a single additional key-value pair in the same dict comprehension.

- The heartbeat API test file's fixture pattern (`_configure_auth`, `data_dir`, `client`, `auth_headers`) is directly reusable for the new endpoint tests. The test can create agent runs via `store.create_agent_run()` and `store.complete_agent_run(output_text=...)` and then hit the endpoint.

- The heartbeat API test pattern of testing 401 for unauthenticated requests and 404 for missing resources should be followed for the new endpoint.

## Deliverables

### 1. Add `output_text` to cascade detail API response

Modify the agent run serialization in the `get_cascade_detail()` endpoint to include `output_text`. The field should be included in the dict for each agent run alongside the existing 10 fields.

The value is `r.output_text` from the `AgentRun` dataclass — it will be `null` for agent runs that haven't completed or that completed without storing output text, and a string for completed runs that stored output.

No store changes, no schema changes, no new imports needed.

### 2. Add `GET /api/heartbeat/agent-run/{agent_run_id}` endpoint

Add a new endpoint to the heartbeat router that returns the full agent run record for a given ID. This endpoint enables the heartbeat log viewer UI (step 007) to fetch agent output text on demand when a user expands a tier entry.

**Behavior:**
- Accepts an `agent_run_id` path parameter
- Requires JWT authentication (same `get_current_user` dependency as all other endpoints)
- Uses `store.get_agent_run(agent_run_id)` to fetch the record
- Returns 404 if the agent run ID does not exist
- Returns the full agent run record serialized as a JSON dict with all fields: `id`, `cascade_id`, `agent_type`, `agent_label`, `model`, `target_layer`, `started_at`, `completed_at`, `success`, `error_message`, `cost_usd`, `output_text`. Note: `cascade_id` may be `null` for agent runs that originated from heartbeat tiers rather than cascade steps — the `AgentRun` dataclass allows this
- Datetime fields are serialized as ISO format strings (consistent with existing serialization patterns)
- `output_text` is included as-is (string or null)

**Why this endpoint lives on the heartbeat router:** The primary consumer is the heartbeat log viewer, which fetches transcripts for heartbeat tier entries. The agent run data itself is generic, but placing it under the heartbeat prefix keeps the API organized by feature. If future features need agent run lookup for non-heartbeat purposes, the endpoint can be moved to a more generic location later.

### 3. Add tests for the new heartbeat agent-run endpoint

Add a test class to the existing `tests/test_heartbeat_api.py` following the established patterns:

- Test that the endpoint returns 200 with the full agent run record when the agent run exists and has output text
- Test that `output_text` is returned as `null` when the agent run was created but not completed with output text
- Test that the endpoint returns 404 when the agent run ID does not exist
- Test that the endpoint returns 401 without authentication
- Test that the response includes all expected fields (id, cascade_id, agent_type, agent_label, model, target_layer, started_at, completed_at, success, error_message, cost_usd, output_text)

### 4. Add tests for `output_text` in cascade detail response

Add tests to the existing `tests/test_cascade_api.py` or extend the existing `TestCascadeDetail` class:

- Test that the cascade detail response includes `output_text` in each agent run dict
- Test that `output_text` is `null` for agent runs that have no stored output
- Test that `output_text` contains the full stored text for agent runs that completed with output text
- Test that existing fields in the agent run dict are unchanged (no regressions)

### 5. Verify existing heartbeat endpoints are unaffected (regression tests)

The existing heartbeat endpoint tests already cover the history, latest, status, and trigger endpoints. Verify these all continue to pass with the new endpoint added to the router. No new regression tests needed — the existing test suite provides this coverage as long as it passes.
