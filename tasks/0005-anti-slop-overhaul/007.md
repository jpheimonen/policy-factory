# 007: Heartbeat log viewer page

## Context

There is currently no way to see what the heartbeat system is doing. The AdminPage has a basic status card showing whether the scheduler is active, when the last run happened, and what tier it reached — but nothing about what the agents actually processed, flagged, or wrote. Without logs, it's impossible to diagnose problems like missed events, over-sensitive escalation, or poor writing quality.

This step builds a new HeartbeatLogPage that shows heartbeat run history with expandable tier-by-tier detail and on-demand transcript viewing. It follows the CascadePage's expandable-detail pattern and consumes the API endpoints from step 006.

## Deliverables

### 1. Create `ui/src/pages/HeartbeatLogPage.tsx`

A new page component following the CascadePage expandable-detail pattern. The page has three levels of detail:

**Level 1 — Run list (always visible):**
- Fetches heartbeat run history from `GET /api/heartbeat/history` on mount
- Shows a list of heartbeat runs in reverse chronological order
- Each run entry shows:
  - When the run started (relative time via `formatRelativeTime`)
  - Trigger type: "scheduled" or "manual" (displayed as a badge or label)
  - Highest tier reached (1-4), displayed prominently since this is the key metric
  - Duration if the run is completed (difference between `started_at` and `completed_at`)
  - Brief outcome derived from the structured log based on the highest tier reached and its escalation result:
    - Tier 1 reached, did not escalate → "nothing noteworthy"
    - Tier 2 reached, did not escalate → "nothing noteworthy" (triage determined no update needed)
    - Tier 3 completed successfully → "update made"
    - Tier 4 reached → "cascade triggered"
    - Any tier failed (error) → "failed" (display with error styling)
    - Fallback for unexpected states → show the raw highest tier and status
- Clicking a run entry expands it to show Level 2

**Level 2 — Tier detail (shown when a run is expanded):**
- Shows tier-by-tier breakdown from the run's `structured_log` array
- Each tier entry shows:
  - Tier number with a descriptive label (Tier 1: Skim, Tier 2: Triage, Tier 3: SA Update, Tier 4: Cascade)
  - Whether this tier escalated to the next tier
  - Outcome text
  - Duration (difference between `started_at` and `ended_at` from the tier entry)
- Each tier entry that has an `agent_run_id` has an expand/collapse control for the transcript

**Level 3 — Agent transcript (shown on demand per tier):**
- When the user clicks to expand a tier's transcript, the page fetches the agent run from `GET /api/heartbeat/agent-run/{agent_run_id}` (the endpoint from step 006)
- The full `output_text` is displayed in a preformatted or markdown-rendered block
- Loading state while fetching, error state if the fetch fails (404 or network error)
- The transcript is cached after first fetch — subsequent expand/collapse does not re-fetch

**Pagination:**
- Initial load fetches 20 runs
- "Load more" button at the bottom when more runs may exist (following the HistoryPage pattern)
- Each click fetches 20 more runs and appends them to the list
- **Important:** The existing `GET /api/heartbeat/history` endpoint only accepts a `limit` query parameter — it does NOT support `offset`. However, the underlying store method (`list_heartbeat_runs`) already accepts both `limit` and `offset` parameters. The endpoint must be extended to also accept an `offset` query parameter so the frontend can paginate. This is a one-parameter addition to the endpoint signature — the store call just needs the `offset` passed through

**Page states:**
- Loading: `LoadingState` molecule during initial fetch
- Error: `ErrorState` molecule if the fetch fails
- Empty: `EmptyState` molecule with the existing `heartbeat.logEmpty` i18n key when no runs exist
