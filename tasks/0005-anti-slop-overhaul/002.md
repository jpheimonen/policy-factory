# 002: Values seed prompt redesign

## Context

The values seed prompt (`seed/values.md`) is the root cause of committee-speak in the values layer. It pre-specifies 10 safe domain categories ("Sovereignty and Self-Determination", "Environmental Responsibility", "Technological Leadership", etc.) and asks for "cross-partisan" values that "reflect broad Finnish consensus." This guarantees bland output — the model produces exactly the safe, uncontroversial platitudes the prompt asks for.

Real values should address the tensions that actually matter when making difficult policy decisions: ethnic cohesion vs. open immigration, military sovereignty vs. alliance dependence, welfare generosity vs. work incentives. If nobody would seriously argue the other side, it's not a tension worth including.

The values seed prompt must be completely rewritten around controversial tension-pairs while maintaining output format compatibility with the existing parsing pipeline (`_parse_values_output()` and `_slugify()` in `server/routers/seed.py`).

## Related Code

- `src/policy_factory/prompts/seed/values.md` — The current values seed prompt. This is the file to rewrite. Currently 77 lines covering value definition, source material, output format, target coverage (the 10 pre-specified categories), and critical instructions.
- `src/policy_factory/server/routers/seed.py` — Contains `_parse_values_output()` (line 140) which parses `---` delimited YAML frontmatter blocks from agent output, and `_slugify()` (line 101) which converts titles to filename-safe slugs. The parsing regex expects the pattern: `---\n<yaml>\n---\n<body>` repeated, with `title` as the only required frontmatter field. The `tensions` field in frontmatter is optional (used but not required by the parser).
- `src/policy_factory/server/routers/seed.py` lines 304-338 — The code that processes parsed values: slugifies the title, handles duplicate slugs, and writes each value as a markdown file via `write_item()`. No hardcoded filenames or title expectations — fully dynamic based on parser output.
- `src/policy_factory/prompts/generators/values.md` — The values generator prompt (cascade-time, not seed-time). This file is rewritten in step 003, but its existence is relevant context: the seed prompt produces the initial values, and the generator prompt later maintains them during cascades. Both must share the same conceptual framing of values as tension-pairs.

## Reuse Opportunities

- The existing output format section in the current prompt can be adapted rather than written from scratch. The frontmatter structure (`title`, `tensions` list, body content) is already established and the parser handles it. The new prompt just needs to change what content goes into this structure.
- The `_slugify()` function already handles arbitrary title strings including special characters, multi-word titles, and unicode normalization. No code changes needed to handle tension-pair titles like "Ethnic & Cultural Cohesion vs. Open Immigration" — the `&` is stripped, "vs" is preserved, and the result is a valid slug.
- The `_parse_values_output()` regex and YAML parsing are format-driven, not content-driven. As long as the new prompt instructs the model to produce the same `---`/frontmatter/`---`/body structure, the pipeline works unchanged.

## Deliverables

### 1. Rewrite `src/policy_factory/prompts/seed/values.md`

Complete rewrite of the values seed prompt. The new prompt must cover:

**Value definition — tension-pairs, not domain categories:** Values are controversial tensions between two legitimate but conflicting priorities where reasonable people genuinely disagree. Each value represents a genuine dilemma that Finnish policymakers must navigate — not a settled consensus that everyone already agrees on. If nobody would seriously argue the other side, it's not a value worth including. Remove the pre-specified 10 domain categories entirely — the model identifies the tensions that actually matter.

**Examples of the calibration expected:** The prompt should include 2-3 examples of good tension-pair titles to calibrate the model's understanding, such as "Ethnic & Cultural Cohesion vs. Open Immigration" or "Military Sovereignty vs. Alliance Dependence" or "Welfare Generosity vs. Work Incentives." These are examples of the format and controversy level expected, not mandated outputs.

**Body content requirements for each tension-pair:** For each value, the body must analyze:
- Where Finland actually sits today on this tension (not where it claims to sit)
- How Finland's position compares to other Nordic countries, Western countries more broadly, and global norms
- What specific policy questions this tension governs — which real decisions does it affect?
- Why it's genuinely controversial — what are the strongest arguments on each side?
- What makes this tension specifically Finnish rather than generic

**Explicit anti-consensus directive:** The prompt must explicitly prohibit consensus items. "Democracy is good" is not a tension. "Rule of law matters" is not a tension. If both sides of the Finnish political spectrum would nod along, it doesn't help make any actual policy decision and should not be included. The bar is: would articulating Finland's position on this tension make at least one major political faction uncomfortable?

**Output format — preserve compatibility:** The output format section must instruct the model to produce the same `---` delimited YAML frontmatter block structure that `_parse_values_output()` expects:
- Each value starts with `---` followed by YAML frontmatter containing at least `title` (required) and `tensions` (list of other tension-pair titles this one interacts with)
- The closing `---` is followed by the body content
- Values are separated by the opening `---` of the next value
- The `title` field contains the tension-pair title (e.g., "Ethnic & Cultural Cohesion vs. Open Immigration")

**Target count:** Instruct the model to produce 8-12 tension-pairs (same range as the current prompt).

**Source material:** The model draws on its training knowledge of Finnish policy, politics, demographics, geopolitics, economics, and social dynamics. No tool use — this is a single-shot text generation task. Remove the tech-centric source material list and replace with general Finnish policy context: constitutional principles, geopolitical position, demographic trends, economic structure, Nordic model, EU membership, NATO membership, immigration patterns, welfare state design, energy policy, etc.

**General policy scope:** All references to "tech policy" or "technology policy" must be replaced with general policy framing. The system covers all policy domains: immigration, defense, taxation, drugs, energy, social welfare, education, trade, EU relations, etc.

### 2. Add tests for `_slugify()` with tension-pair titles

The `_slugify()` function in `server/routers/seed.py` has no existing tests. Add unit tests verifying it correctly handles tension-pair style titles:
- Titles containing "vs." produce valid, non-empty slugs
- Titles containing ampersands (`&`) produce valid slugs (the `&` is stripped by the regex)
- Long multi-word titles produce reasonable slugs
- Titles with mixed case are normalized to lowercase
- Empty string input produces an empty slug (or the function handles it gracefully)

These tests belong in a new test file or alongside existing seed API tests, at the implementer's discretion. Note: both `_slugify()` and `_parse_values_output()` are module-private functions (prefixed with `_`). They must be imported directly from `server.routers.seed` for testing — this is acceptable for unit-testing internal parsing logic.

### 3. Add tests for `_parse_values_output()` with tension-pair format

The `_parse_values_output()` function also has no existing tests. Add unit tests verifying it correctly parses the tension-pair output format:
- A well-formed output with multiple tension-pair values (each with `---`/frontmatter/`---`/body structure) is parsed into the correct number of (frontmatter_dict, body_string) tuples
- The `title` field is correctly extracted from frontmatter with tension-pair format titles
- The `tensions` list in frontmatter is correctly parsed
- Malformed blocks (missing title, invalid YAML) are skipped without crashing
- An output with no valid blocks returns an empty list
