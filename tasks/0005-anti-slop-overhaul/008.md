# 008: Cascade transcript UI and cascade trigger verification

## Context

The cascade detail panel (`CascadeDetailPanel` in `CascadePage.tsx`) currently shows basic summary information when a history entry is expanded — trigger source, timing, error message, and processed layer badges. It does NOT show individual agent runs or their output text, even though the API response (after step 006) will include `output_text` for each agent run.

This step adds expandable agent run entries with output text transcripts to the cascade detail panel. It also updates the `AgentRunInfo` TypeScript interface to include the new `output_text` field from the modified API response.

Separately, the requirements specify that the layer detail page should have a visible cascade trigger control. The existing `LayerDetailPage.tsx` already has a "Refresh Layer" button that calls `POST /api/cascade/refresh` with the current layer slug. This button satisfies the requirement — this step verifies that it works correctly and adds E2E test coverage for it.

Finally, this step adds E2E tests for the heartbeat log page (built in step 007), the cascade transcript expansion (built in this step), and the layer detail refresh button.

## Deliverables

### 1. Update `AgentRunInfo` interface in `ui/src/pages/CascadePage.tsx`

Add the `output_text` field to the `AgentRunInfo` interface — nullable, representing the full agent output text. This aligns the TypeScript interface with the modified API response from step 006 where `output_text` was added to the cascade detail agent run serialization.

### 2. Add agent run entries with expandable transcripts to `CascadeDetailPanel`

Modify the `CascadeDetailPanel` component to display individual agent run entries below the existing summary information (trigger source, timing, layers processed). The current component already receives agent runs but uses them only for layer badge extraction.

**Agent run list:**
- Below the existing summary content, add a section heading if there are any agent runs
- List each agent run as a distinct entry showing:
  - Agent label — the human-readable name of the agent that ran
  - Target layer displayed as a layer badge
  - Success/failure status as a badge (success = green, failure = red)
  - Duration (computed from started_at and completed_at if both are present)
  - Error message if present

**Expandable transcript per agent run:**
- Each agent run entry that has output text has an expand/collapse control
- Clicking the control toggles visibility of the transcript section
- The transcript displays the output text in a preformatted block with appropriate styling for long text content (monospace font, horizontal scroll for long lines, max-height with vertical scroll)
- Agent runs without output text do not show the expand control
- Multiple transcripts within the same cascade detail can be expanded simultaneously

**No additional API calls needed.** The output text is included in the cascade detail response (after step 006). The data is already available when the panel renders.

### 3. Add styled components to `ui/src/pages/CascadePage.styles.ts`

Add styled components for the agent run entries and transcript display within the cascade detail panel:

- A container for the list of agent run entries, with a top border separator from the summary section
- Individual agent run entry wrapper with padding and flex layout for label + meta info
- A header row for agent label, layer badge, status badge, and duration
- A preformatted text block for output text display — monospace font, max-height with vertical scroll for long output, horizontal scroll for long lines, subtle background for visual distinction
- A clickable control for expand/collapse with cursor pointer styling

### 4. Add i18n translation keys in `ui/src/i18n/locales/en.ts`

Add keys to the existing `cascade` namespace for agent run display:

- Section heading for agent run list
- Success and failure status labels
- Expand and collapse transcript control labels
- Label for agent runs without output text

### 5. Verify LayerDetailPage refresh button satisfies cascade trigger requirement

The `LayerDetailPage` already has a "Refresh Layer" button that calls `POST /api/cascade/refresh` with the current layer slug. This satisfies the requirement in the requirements spec:

> "The layer detail page in the UI has a visible button/control to trigger a cascade refresh starting from that layer"

**No code changes needed.** The button exists, uses appropriate i18n keys, shows loading state during the request, and calls the correct API endpoint. The verification is documented by the E2E test added in deliverable 6.

### 6. Add E2E tests

Add E2E tests across three areas:

**A. Heartbeat log page tests — add to `ui/e2e/heartbeat-log.spec.ts` (new file):**

Tests for the heartbeat log page built in step 007. Uses the same `setupAndLogin` helper pattern as existing E2E test files.

- Test that the heartbeat log page is accessible at `/heartbeat` after login
- Test that the navigation bar includes a link to the heartbeat log page (click nav link → arrives at `/heartbeat`)
- Test that the page shows an empty state when no heartbeat runs exist
- Test that when heartbeat runs exist, the page shows run entries with trigger type and tier information. **Data setup:** Use `POST /api/heartbeat/trigger` (authenticated, admin-only) to create a heartbeat run. The heartbeat will execute quickly against Yle RSS and produce at least a tier-1 entry. Alternatively, if the trigger endpoint requires external dependencies that aren't available in E2E, the test may need to verify only the empty state and skip data-dependent tests with a note explaining why
- Test that clicking a run entry expands it to show tier detail
- Test that the expanded detail shows tier information (tier number, escalation status, outcome)
- Test that clicking the expanded entry again collapses it

**B. Cascade transcript tests — add to `ui/e2e/cascade-viewer.spec.ts` (extend existing file):**

Tests for the expandable agent run transcript in the cascade detail panel.

- Test that the cascade detail panel shows agent run entries when a completed cascade with agent runs exists
- Test that agent run entries display agent label and target layer
- Test that agent runs with output text have an expand control
- Test that expanding an agent run's transcript shows the output text content

These tests require a completed cascade with agent runs that have `output_text` stored. **Data setup:** The existing E2E pattern (seen in `layer-detail.spec.ts` and `version-history.spec.ts`) creates test data via authenticated API calls using `page.request.post()` with a JWT token. For cascade data, use `POST /api/cascade/refresh` with a layer slug to trigger a cascade — but this requires the layer to have content and agents to be available. If triggering a real cascade is not feasible in E2E (due to agent/model dependencies), these tests should be marked as requiring a seeded database or skipped with a clear comment. The cascade-viewer E2E tests currently test only idle state, suggesting real cascade data hasn't been reliably producible in E2E before.

**C. Layer detail refresh button tests — add to `ui/e2e/layer-detail.spec.ts` (extend existing file):**

- Test that the layer detail page displays a refresh button
- Test that clicking the refresh button changes the button state (shows loading/refreshing text)
