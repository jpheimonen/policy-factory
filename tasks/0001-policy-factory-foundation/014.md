# 014: Agent framework and prompt system

## Context

All prior steps have built the data layer (markdown files with YAML frontmatter, git-backed), the REST API, the SQLite store, the event system with WebSocket broadcasting, and the frontend integration. What has been missing is the core capability that makes Policy Factory an AI-powered system: the ability to run Claude Code sessions that read and write the policy files, stream their output in real-time, and do so with a carefully constructed prompt that begins with a bias meditation ritual.

This step builds the agent framework — a simplified adaptation of cc-runner's session wrapper — that all later steps depend on. The cascade orchestrator (step 015) uses it to run layer generators and synthesis agents. The critic runner (step 016) uses it for all 6 critic archetypes. The input classifier (step 017), idea pipeline (step 018), heartbeat (step 019), and seed agent (step 017) all create agent sessions through this framework.

The framework has four distinct concerns:

1. **Session wrapper**: Wraps the `claude-agent-sdk` to run Claude Code against the `data/` directory. Each agent session is configured with a specific model, working directory, and prompt. The session streams SDK messages, emits text chunks as typed events through the EventEmitter (step 012), handles transient errors with retries, and returns a structured result.

2. **Prompt system**: A PromptLoader that reads markdown template files from a `prompts/` directory, substitutes dynamic variables, and supports composing prompts from multiple sections (meditation preamble + agent-specific template). Every agent prompt begins with the meditation preamble.

3. **Meditation filter**: A streaming output filter that detects the meditation section (a countdown from 10 to 1 with bias self-reflection) in the agent's streamed output and suppresses it from the WebSocket broadcast. Users see only the post-meditation analysis content. The full unfiltered output including meditation is captured in the agent run log for auditability.

4. **Prompt template files**: The actual markdown prompt files for every agent category — generators (one per layer), critics (one per archetype), synthesis, heartbeat tiers, input classifier, idea generation/evaluation, and the shared meditation preamble.

Policy Factory's agent framework is significantly simpler than cc-runner's because it does not need: the Ralph Wiggum looping system, HITL prompts, tool permission hooks, custom MCP servers, guardrails/settings management, multiple agent type support (Codex), or session resume. Policy Factory agents are fire-and-forget tasks — they run a prompt to completion, streaming output along the way. The only complex behaviours retained are streaming event emission, retry with exponential backoff, transient error classification, and meditation filtering.

### Session Configuration

- Fields: working directory (Path, defaults to the data directory), model name (optional string — when omitted, the SDK uses its default), max turns (optional integer), system prompt (optional string), permission mode (string literal, defaults to a permissive mode suitable for file read/write operations since agents need to modify markdown files).

When an environment variable is not set, the default for that role is used.

### Session Wrapper

A module at `src/policy_factory/agent/session.py` providing the agent session wrapper:

**Initialisation:**
- Accepts a session configuration dataclass, an EventEmitter instance (from step 012), and an optional cascade ID or context identifier (used for attributing emitted events to a specific cascade run or operation).
- Stores references to the config and emitter for use during the run.

**Run method:**
- An async method that accepts a prompt string and runs it to completion.
- Builds the SDK client options from the session config: working directory, model, max turns, system prompt, permission mode. Agents need native web search capability (Claude Code's built-in web search), so the allowed tools configuration includes web search.
- Creates a `ClaudeSDKClient` with the options, sends the prompt via `client.query()`, and enters the streaming receive loop (`async for message in client.receive_response()`).
- For each message in the stream:
  - Extracts text content from `AssistantMessage` content blocks (blocks with a `text` attribute).
  - Passes each text chunk through the meditation filter (described below). If the filter indicates the chunk should be streamed (post-meditation content), emits an `AgentTextChunk` event through the EventEmitter with the cascade/context ID, an agent label (human-readable string like "Values layer generator" or "Realist critic"), and the text chunk.
  - Regardless of meditation filtering, all text chunks are accumulated in a full output buffer for logging and return value.
  - When the message stream yields a result message (has `is_error` attribute), captures the result.
- Returns a structured result dictionary containing: whether an error occurred (boolean), the result text, the total cost (if available from the SDK), the number of turns, and the full unfiltered output text (including meditation content).

**Retry logic:**
- Wraps the run in a retry loop with a maximum of 3 attempts.
- On each failed attempt, checks whether the error is transient using a classification function.
- Transient errors: Anthropic API 500, 502, 503, 529 status codes; "overloaded" or "rate limit" messages in the error string. These trigger a retry after exponential backoff (base delay of 2 seconds, doubled each retry: 2s, 4s, 8s).
- Non-transient errors: authentication failures, context overflow ("prompt is too long"). These are raised immediately without retry.
- Context overflow detection: if the SDK returns an `is_error` result with text containing "prompt is too long", a `ContextOverflowError` is raised.
- If all retry attempts are exhausted, the last error is raised to the caller.
- Between retries, logging captures the error details and retry attempt number.

**Error types:**
- `ContextOverflowError` — raised when the agent's context window is exceeded. Carries the error message.
- `AgentError` — a general agent error raised for non-transient failures. Carries the error message and optionally the agent role and cascade ID for diagnostics.

### Meditation Filter

A class or stateful function at `src/policy_factory/agent/meditation_filter.py` that processes streamed text chunks and determines whether each chunk should be broadcast to clients:

**Meditation detection:**
- The meditation preamble instructs the AI to count down from 10 to 1, reflecting on biases at each step. The filter detects the start of the meditation by looking for the countdown pattern (the number "10" appearing at or near the beginning of the output, followed by text, then "9", "8", etc.).
- The filter maintains internal state tracking whether it is currently inside the meditation section.
- The meditation section ends when the countdown reaches "1" and the subsequent content transitions to the actual analysis. The filter detects this transition and begins allowing content through.

**Chunk processing:**
- Each incoming text chunk is passed to the filter.
- If the filter is in the "meditation" state (between detecting the countdown start and detecting the end), the chunk is suppressed from broadcasting — the filter returns an indication that this chunk should not be emitted as an event.
- If the filter is in the "post-meditation" state (after the countdown completes), the chunk is allowed through for broadcasting.
- If no meditation pattern is ever detected (the AI skipped or reformatted the meditation), the filter takes a conservative approach and allows all content through. The filter does not wait indefinitely for a meditation pattern — if the first significant chunk of text does not match the expected meditation start pattern, the filter assumes no meditation is present and streams everything.

**Full capture:**
- Regardless of whether chunks are suppressed or allowed, the filter provides access to the complete unfiltered text for logging purposes. The caller accumulates the full output separately from the filtered output.

### Prompt Loader

A module at `src/policy_factory/prompts/loader.py` adapted from cc-runner's PromptLoader:

**PromptLoader class:**
- Initialised with a prompts directory path. Defaults to the `src/policy_factory/prompts/` directory (resolved relative to the package).
- `load(category, name, **variables)` — loads a markdown file from `prompts/{category}/{name}.md` and substitutes template variables using `str.format()`. Double braces `{{ }}` in the template are escaped and become literal braces in the output. Raises `FileNotFoundError` if the prompt file does not exist.
- `load_section(name)` — loads a single section file from `prompts/sections/{name}.md`. Returns the raw content without variable substitution (sections are static preambles). Raises `FileNotFoundError` if not found.
- `load_sections(names)` — loads multiple sections by name and concatenates them with double-newline separators. Returns an empty string for an empty list. Preserves the order of sections as specified.

**Module-level convenience functions:**
- `get_prompt_loader()` — returns the module-level singleton PromptLoader instance, creating it on first call.
- `load_prompt(category, name, **variables)` — convenience wrapper calling `get_prompt_loader().load()`.
- `load_section(name)` — convenience wrapper calling `get_prompt_loader().load_section()`.
- `load_sections(names)` — convenience wrapper calling `get_prompt_loader().load_sections()`.

**Public API:**
- The prompts package (`src/policy_factory/prompts/__init__.py`) exports the PromptLoader class and all convenience functions.

### Prompt Construction Helper

A function at `src/policy_factory/agent/prompts.py` that assembles complete agent prompts from components:

- Accepts an agent role (e.g., "generator", "critic", "synthesis"), a category subdirectory name, a template name, and a dictionary of dynamic template variables.
- Loads the meditation preamble section using `load_section("meditation")`.
- Loads the agent-specific prompt template using `load_prompt(category, name, **variables)`.
- Concatenates the meditation preamble and the agent-specific template with a clear separator between them.
- Returns the assembled prompt string ready to be passed to the session wrapper.

This helper ensures every agent prompt consistently includes the meditation preamble without each caller needing to remember to include it.

### Prompt Template Files

All prompt template files are markdown files stored in the `src/policy_factory/prompts/` directory, organised by category. Each file uses `{variable_name}` placeholders for dynamic content that gets substituted at runtime.

**Meditation preamble — `prompts/sections/meditation.md`:**
- A section file containing the bias meditation ritual instructions.
- Instructs the AI to begin its response by counting down from 10 to 1.
- At each number, the AI reflects on a different dimension of potential bias: ideological leanings, cultural assumptions, recency bias, confirmation bias, anchoring effects, Western-centric framing, techno-optimism or techno-pessimism, status quo bias, elite perspective bias, and any other biases it recognises in itself.
- The AI is never told what its biases are — it discovers and acknowledges them itself.
- The meditation concludes with a commitment to approach the analysis with awareness of these biases.
- After the meditation, the AI proceeds to the actual analysis task.

**Generator prompts — `prompts/generators/`:**
- One file per layer: `values.md`, `situational-awareness.md`, `strategic.md`, `tactical.md`, `policies.md`.
- Each generator prompt instructs the AI to read the existing layer content (or create it if empty), consider the content of upstream layers, incorporate any pending feedback memos from downstream layers, and produce updated markdown content for the layer.
- Template variables include: the layer slug, the layer's current content summary, pending feedback memos from other layers, and cross-layer context (summaries of related layers).
- Generator prompts emphasise that the output must be in the specific markdown format with YAML frontmatter that the data layer expects.
- The values layer generator has a special variant for Finland's cross-party tech group context, with emphasis on non-partisan, evidence-based analysis.

**Critic prompts — `prompts/critics/`:**
- One file per critic archetype: `realist.md`, `liberal-institutionalist.md`, `nationalist-conservative.md`, `social-democratic.md`, `libertarian.md`, `green-ecological.md`.
- Each critic prompt instructs the AI to adopt the specified ideological perspective and critically assess the layer content.
- The critic produces a structured assessment: areas of agreement, areas of disagreement with reasoning, alternative framings or policy directions from this ideological perspective, and a numerical score.
- Template variables include: the layer slug, the layer content being assessed, and context from other layers.
- Each archetype prompt provides a brief description of the ideological framework's core principles and analytical lens, without caricaturing the perspective.

**Synthesis prompt — `prompts/synthesis/synthesis.md`:**
- Instructs the AI to integrate all 6 critic assessments into a coherent synthesis.
- Identifies points of consensus across archetypes, areas of genuine tension where perspectives conflict, and suggests how the layer content could be refined to address the strongest criticisms.
- Template variables include: the layer slug, the original layer content, and all 6 critic assessment outputs.
- The synthesis explicitly identifies tensions rather than seeking false consensus.

**Heartbeat prompts — `prompts/heartbeat/`:**
- `skim.md` — Tier 1: instructs the AI to skim a news source (yle.fi) for developments relevant to Finnish tech policy. Uses web search. Determines whether any development warrants deeper investigation. Template variables: the current date, a summary of the current Situational Awareness layer.
- `triage.md` — Tier 2: instructs the AI to search the web more broadly for the flagged development, assess its significance for the policy stack, and determine whether the Situational Awareness layer should be updated. Template variables: the flagged development from Tier 1, current SA layer summary.
- `sa-update.md` — Tier 3: instructs the AI to update the Situational Awareness layer with the new information. Template variables: the triage assessment from Tier 2, the full current SA layer content, pending feedback memos.

**Classifier prompt — `prompts/classifier/classifier.md`:**
- Instructs the AI to classify free-text user input to determine which layer(s) it most directly affects.
- Returns a structured classification: the primary target layer, optionally secondary affected layers, a confidence assessment, and an explanation.
- Template variables: the user's input text, a brief description of each layer and its current state.

**Idea prompts — `prompts/ideas/`:**
- `evaluate.md` — Instructs the AI to evaluate a submitted idea against the full policy stack. Scores the idea on 6 axes (feasibility, alignment with values, political viability, evidence basis, implementation complexity, innovation). Template variables: the idea text, summaries of all 5 layers.
- `generate.md` — Instructs the AI to generate policy ideas based on the current state of the policy stack. Optionally scoped to a specific strategic or tactical objective. Template variables: summaries of all 5 layers, optional scoping context.

**Seed prompt — `prompts/seed/seed.md`:**
- Instructs the AI to create the initial Situational Awareness layer for Finland's tech policy landscape.
- Uses web search to gather current information about Finland's digital policy environment, EU regulatory landscape, Nordic cooperation, and technology trends.
- Template variables: the current date, the values layer content (which is pre-seeded in step 007).
