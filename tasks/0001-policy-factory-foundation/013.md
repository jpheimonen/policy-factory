# 013: Frontend WebSocket integration

## Context

The backend event system and WebSocket infrastructure are in place (step 012). The backend can emit typed events, persist them to SQLite, and broadcast them via WebSocket to all authenticated clients. The frontend now needs to connect to this WebSocket, receive events, and dispatch them to the appropriate zustand stores so that UI components react to backend activity in real-time.

Currently, the cascade store (step 009) and all other frontend data are updated only via REST polling on page mount. After this step, the cascade store, idea store, and a new activity store are updated live via WebSocket events. The cascade status indicator in the navigation (step 009) animates when a cascade starts, without requiring a page refresh. The idea inbox (step 021) will show evaluation progress in real-time. The activity feed (step 022) will display new events as they arrive.

This step also handles the critical reconnection scenario: if the WebSocket connection drops (network interruption, server restart), the hook auto-reconnects with exponential backoff. On successful reconnection, it replays missed events from the REST replay endpoint (`GET /api/activity/replay?since_id=N`) to fill any gaps. Events are deduplicated by their database-generated ID, so replayed events and live events never cause double-renders.

The key architectural decision is **centralised dispatch** rather than per-component WebSocket connections. Unlike cc-runner where each page creates its own `useWebSocket(runId)` hook for a specific run, Policy Factory uses a single global WebSocket connection (since all events are relevant to all authenticated users) with a central dispatcher that routes incoming events to the correct zustand stores based on event type.

**Backend endpoints (step 012):**

- `ws://host/ws?token=<JWT>` — WebSocket endpoint. Events arrive as JSON objects with an `id` field (database-generated, for deduplication), `event_type`, `timestamp`, and type-specific fields.
- `GET /api/activity/replay?since_id=N` — Returns events with ID > N in chronological order (for reconnection replay). Limited to 500 events.

## Reuse Opportunities

- The **useWebSocket hook** from cc-runner is the foundation. Policy Factory adapts it with: JWT token in query parameter (instead of session cookies), no `runId` scoping (global connection), and store-based dispatch instead of returning an events array.
- The **exponential backoff constants** from cc-runner (1s initial, 15s max, 8 max attempts) are directly reusable and well-tuned.
- The **mergeEvents utility** from cc-runner (Map-based deduplication with ID sorting) is reusable for the replay merge.
- The **event type definitions** from cc-runner's `types/events.ts` provide the structural pattern — a base interface plus a discriminated union. Policy Factory defines its own event types matching the backend events from step 012.
- The **globalHITLStore dispatch pattern** from cc-runner (switch on event_type → update specific store) is the model for Policy Factory's centralised event dispatcher.
- The **cascade store** from step 009 already has the right state shape (status, current layer/step, queue depth, error info). This step adds the event handlers that update that state — the store's shape doesn't need to change, only its update mechanisms.

## Deliverables

### Frontend Event Type Definitions

A types file at `ui/src/types/events.ts` defining TypeScript types for all backend events:

**Base event interface:**
- `id` — number (database-generated, used for deduplication and replay cursor).
- `event_type` — string literal union of all valid event types.
- `timestamp` — string (ISO 8601).

**Event-specific interfaces** extending the base, one per event type from step 012:
- Cascade lifecycle: `CascadeStartedEvent`, `CascadeCompletedEvent`, `CascadeFailedEvent`, `CascadePausedEvent`, `CascadeResumedEvent`, `CascadeCancelledEvent`, `CascadeQueuedEvent`.
- Layer processing: `LayerGenerationStartedEvent`, `LayerGenerationCompletedEvent`, `CriticStartedEvent`, `CriticCompletedEvent`, `SynthesisStartedEvent`, `SynthesisCompletedEvent`.
- Agent streaming: `AgentTextChunkEvent`.
- Heartbeat: `HeartbeatStartedEvent`, `HeartbeatTierCompletedEvent`, `HeartbeatCompletedEvent`.
- Ideas: `IdeaSubmittedEvent`, `IdeaEvaluationStartedEvent`, `IdeaEvaluationCompletedEvent`, `IdeaGenerationStartedEvent`, `IdeaGenerationCompletedEvent`.
- System: `UserLoginEvent`, `UserCreatedEvent`, `CascadeLockAcquiredEvent`, `CascadeLockReleasedEvent`.

A discriminated union type (e.g., `PolicyEvent`) covering all event interfaces, discriminated by `event_type`. This enables type-safe event handling in the dispatcher.

### useWebSocket Hook

A custom hook at `ui/src/hooks/useWebSocket.ts` managing the global WebSocket connection:

**Connection:**
- Constructs the WebSocket URL: auto-detects protocol (`ws:` for HTTP, `wss:` for HTTPS), appends `?token=<JWT>` using the JWT from the auth store.
- Connects on mount (when the user is authenticated). Does not connect if no JWT is available.
- Disconnects cleanly on unmount (sets an intentional-close flag to prevent reconnection).

**Reconnection with exponential backoff:**
- On unexpected close (not intentional), increments a reconnect attempt counter.
- Schedules a reconnection attempt after a delay: initial delay of 1 second, doubling on each attempt, capped at 15 seconds.
- Maximum of 8 reconnect attempts. After 8 failed attempts, stops trying and exposes a "disconnected" state that the UI can display.
- If the JWT has been refreshed since the last connection (e.g., due to token refresh), uses the new JWT for the reconnection.
- Resets the attempt counter on successful reconnection.

**Event replay on reconnection:**
- On successful reconnection (detected by `reconnectAttempt > 0` when `onopen` fires), calls `GET /api/activity/replay?since_id=N` where N is the ID of the last event received before disconnection.
- Merges replayed events with any events received from the new WebSocket connection using the same `mergeEvents()` deduplication logic as cc-runner: Map-based by event ID, replayed events take precedence, sorted by ID.
- If the replay endpoint returns more than 500 events (indicating a long disconnection), the hook triggers a full refresh of all stores instead of incremental replay.
- If the replay request fails, the hook continues with live events only (graceful degradation — the user may miss some events but will get future ones).

**Tracking the last received event ID:**
- Maintains a ref tracking the highest event ID received (from both live and replayed events).
- This value is used as the `since_id` parameter for replay on reconnection.

**Exposed state and actions:**
- `connected` — boolean indicating whether the WebSocket is currently open.
- `reconnecting` — boolean indicating an active reconnection attempt (for UI display of "Reconnecting..." state).
- `disconnected` — boolean indicating that all reconnection attempts have been exhausted.

**Inbound message handling:**
- Parses each incoming message as JSON.
- Validates that the parsed object has an `id` and `event_type` field (basic sanity check).
- Deduplicates by ID: maintains a Set of recently seen event IDs (rolling window of the last ~1000 IDs to prevent unbounded growth). If an event ID has already been seen, it is silently dropped.
- Passes the validated, deduplicated event to the central dispatcher.

**Outbound messaging:**
- A `send(message)` function for sending JSON messages to the backend. Not currently needed (the WebSocket is primarily server-to-client), but included for future use. Fire-and-forget — no response expected.

### Central Event Dispatcher

A dispatcher function or module at `ui/src/hooks/useEventDispatch.ts` (or integrated into the useWebSocket hook) that routes incoming events to the appropriate zustand stores:

**Dispatch rules based on event_type:**

- **Cascade events** (`cascade_started`, `cascade_completed`, `cascade_failed`, `cascade_paused`, `cascade_resumed`, `cascade_cancelled`, `cascade_queued`) → cascade store. Each event type maps to a specific store mutation (e.g., `cascade_started` sets status to "running" and records the starting layer).
- **Layer processing events** (`layer_generation_started`, `layer_generation_completed`, `critic_started`, `critic_completed`, `synthesis_started`, `synthesis_completed`) → cascade store. These update the current step within the cascade progress.
- **Agent text chunk events** (`agent_text_chunk`) → cascade store. Text chunks are appended to a streaming text buffer for the live cascade viewer (step 020).
- **Idea events** (`idea_submitted`, `idea_evaluation_started`, `idea_evaluation_completed`, `idea_generation_started`, `idea_generation_completed`) → idea store. These update idea statuses and trigger list refreshes.
- **Heartbeat events** (`heartbeat_started`, `heartbeat_tier_completed`, `heartbeat_completed`) → activity store (for display in the activity feed). No dedicated heartbeat store needed.
- **System events** (`user_login`, `user_created`, `cascade_lock_acquired`, `cascade_lock_released`) → activity store.

- **All events** → activity store. Every event (regardless of category) is also pushed to the activity store's event list so the activity feed page (step 022) has a complete chronological stream.

**Side effects on certain events:**
- When `cascade_completed` is received, the dispatcher also triggers a refresh of the layer store (since a completed cascade means layer content has changed). This is an async side effect — it calls `layerStore.getState().refresh()`.
- When `cascade_started` is received, the dispatcher updates the cascade store's status and may trigger the cascade status indicator animation in the navigation.

### Cascade Store Upgrade

Extend the cascade store from step 009 (`ui/src/stores/cascadeStore.ts`) with WebSocket event handlers:

**New state fields:**
- `currentAgentLabel` — string or null. The human-readable label of the currently active agent (e.g., "Realist critic analyzing Strategic Objectives"). Updated from layer processing events. Cleared when the cascade completes or is cancelled.
- `streamingText` — string. The accumulated agent reasoning text being streamed live. Appended to on each `agent_text_chunk` event. Cleared when a new agent starts (new layer processing event). Used by the live cascade viewer (step 020).

**Event handler methods** (called by the dispatcher):

- `handleCascadeStarted(event)` — sets status to "running", records cascade ID, starting layer, and trigger source. Clears any previous error info and streaming text.
- `handleCascadeCompleted(event)` — sets status to "idle". Clears cascade ID, current layer/step, agent label, and streaming text.
- `handleCascadeFailed(event)` — sets status to "failed". Records error message, failed layer, and failed step.
- `handleCascadePaused(event)` — sets status to "paused". Records error message and paused position.
- `handleCascadeResumed(event)` — sets status to "running". Clears error info.
- `handleCascadeCancelled(event)` — sets status to "idle". Clears all cascade state.
- `handleCascadeQueued(event)` — increments queue depth. Records the queued cascade's position.
- `handleLayerProcessingStarted(event)` — updates current layer and step (generation, critics, or synthesis based on the event type). Clears streaming text for the new step.
- `handleLayerProcessingCompleted(event)` — updates step progress.
- `handleCriticStarted(event)` — sets the current agent label (e.g., "Realist critic").
- `handleCriticCompleted(event)` — clears the agent label for that critic.
- `handleAgentTextChunk(event)` — appends the text chunk to `streamingText`. Updates `currentAgentLabel` if the event carries an agent label.

The cascade store's existing REST-based `fetchStatus` action from step 009 is retained as a fallback. On initial load, the store fetches status via REST. After that, WebSocket events keep it current. If the WebSocket disconnects and reconnects, the replay mechanism fills the gap.

### Idea Store Shell

A zustand store at `ui/src/stores/ideaStore.ts` with the initial shape needed for WebSocket event handling:

**State:**
- A list of ideas (initially empty — the full idea list will be fetched via REST in step 021 when the ideas page is built).
- Loading and error states.

**Event handler methods** (called by the dispatcher):
- `handleIdeaSubmitted(event)` — adds the idea to the list (or marks it for refresh).
- `handleIdeaEvaluationStarted(event)` — updates the idea's status to "evaluating".
- `handleIdeaEvaluationCompleted(event)` — updates the idea's status to "evaluated" and triggers a detail refresh if the idea is currently being viewed.
- `handleIdeaGenerationStarted(event)` — could set a flag indicating generation is in progress.
- `handleIdeaGenerationCompleted(event)` — triggers a list refresh.

This is a minimal shell — step 021 (Frontend idea inbox) will add REST-based CRUD actions, filtering, and sorting. The event handlers are the only parts needed now for real-time updates to flow.

### Activity Store

A zustand store at `ui/src/stores/activityStore.ts` holding the chronological event stream:

**State:**
- A list of recent events (the last N events, where N is configurable — e.g., 200 — to prevent unbounded memory growth).
- Filter state: selected event category (cascade, heartbeat, idea, system, or all), selected layer slug (or all).

**Actions:**
- `addEvent(event)` — appends a new event to the list. If the list exceeds the maximum size, trims the oldest events. Maintains the list in chronological order (newest at the end).
- `setFilter(category, layerSlug)` — updates the filter criteria.
- `fetchHistory(limit, offset)` — calls `GET /api/activity/` to load historical events (for when the user first opens the activity feed page). Populates the store.

**Computed values:**
- `filteredEvents` — the event list filtered by the current category and layer slug selections. The component reads this derived value.

This store is populated by two sources: the dispatcher pushes every incoming WebSocket event via `addEvent()`, and the activity feed page fetches historical events via `fetchHistory()` on mount.

### WebSocket Hook Initialization

The useWebSocket hook is initialised at the app level, inside the protected route area:

- The hook is called in a component rendered inside the protected route wrapper (so it only activates when the user is authenticated).
- One option is to call it in the app shell layout component that wraps all protected pages. Another is to create a small `<WebSocketProvider />` component that calls the hook and renders nothing (a "connector" component). The implementer chooses the cleanest approach.
- The hook reads the JWT from the auth store. If the user logs out, the hook disconnects. If the user logs in, the hook connects.
- The dispatcher is connected to the hook — every validated event from the hook is passed to the dispatcher function.

### Connection Status UI

The navigation component (step 009) is extended with a subtle connection status indicator:

- When the WebSocket is connected, nothing extra is shown (connected is the default state).
- When the WebSocket is reconnecting (disconnected with reconnection attempts remaining), a subtle "Reconnecting..." indicator is shown in the navigation area. This could be a small text label, a pulsing icon, or a subtle banner.
- When the WebSocket has exhausted all reconnection attempts (fully disconnected), a "Connection lost" indicator is shown with a manual "Reconnect" button. Clicking it resets the attempt counter and triggers a new connection.
- All status labels use i18n translation keys.
