# 020: Frontend live cascade viewer

## Context

The cascade infrastructure is complete: the orchestrator (step 015) drives layer-by-layer updates, the critic runner (step 016) runs 6 perspectives in parallel, the cascade router (step 017) exposes trigger/status/control endpoints, and the heartbeat (step 019) can initiate cascades autonomously. The frontend WebSocket integration (step 013) already delivers cascade events and agent text chunks to the cascade store in real-time. The cascade store (step 013) accumulates streaming text, tracks the current layer and step, records the active agent label, and manages cascade status. All the data is flowing — but there is no page to display it.

This step builds the live cascade viewer page — the centrepiece of the real-time experience. When a cascade is running, users see exactly what is happening: which layer is being processed, which step within that layer (generation → critics → synthesis), which specific agent is active (e.g., "Green/Ecological critic analyzing Tactical Objectives"), and the agent's actual reasoning text streaming in live. The meditation preamble is already filtered out by the backend (step 014); what streams to the page is pure analysis content.

The viewer also handles the non-running states: when no cascade is running, it shows the most recently completed cascade's summary and the idle state. When a cascade is paused due to an error, it shows the error details with prominent resume and cancel buttons. When cascades are queued, the queue is visible so users know what is waiting.

This page is purely a frontend effort — no backend changes are needed. All data comes from:
1. The cascade store (updated in real-time via WebSocket events from step 013) — provides status, progress, streaming text, active agent label, queue depth, error info.
2. The cascade REST endpoints (step 017) — `GET /api/cascade/status` for initial load, `GET /api/cascade/:cascade_id` for cascade detail, `GET /api/cascade/history` for recent cascade runs.

The page route (`/cascade`) was already registered as a placeholder in step 009. This step replaces the placeholder with the real implementation.

- `GET /api/cascade/history` — Recent cascade runs in reverse chronological order. Fetched for the history section.

## Deliverables

### Cascade Page Component

Replace the placeholder `CascadePage` at `ui/src/pages/CascadePage.tsx` (route `/cascade`) with the full live cascade viewer. The page has three major visual sections that render conditionally based on the cascade state:

**Page header:**
- Page title using the `cascade.title` translation key.
- Cascade status badge showing the current state (idle, running, paused, failed, queued) with appropriate colour coding (status colours from the theme).
- If a cascade is running or paused: the trigger source (what started this cascade — user input, layer refresh, heartbeat, seed) and the started timestamp as relative time.

**Primary content area — changes based on cascade state:**

When a cascade is **running**:
- The progress indicator (described below) showing the cascade's journey through the layers.
- The active agent label displayed prominently (e.g., "Realist critic analyzing Strategic Objectives"). Updated in real-time from the cascade store. When no specific agent is active (between steps), shows the current step name (e.g., "Running synthesis for Tactical Objectives").
- The streaming text panel (described below) showing the agent's live reasoning output.

When a cascade is **paused** (due to error):
- The error details: which layer and which step failed, and the error message.
- Prominent resume and cancel action buttons. Resume calls `POST /api/cascade/:cascade_id/resume`. Cancel calls `POST /api/cascade/:cascade_id/cancel`. Both show loading state while the request is in flight and update the UI based on the result.
- The progress indicator showing where in the cascade the pause occurred (all completed layers shown as done, the failed layer/step highlighted in error state).

When a cascade is **idle** (no cascade running):
- A summary of the most recently completed cascade: which layers were processed, when it completed, what triggered it.
- The cascade history section (described below).
- An empty state message if no cascades have ever run, with guidance on how to trigger one (e.g., submit input on the home page, or use the seed endpoint).

**Queue section (visible whenever queue depth > 0, regardless of cascade state):**
- Listed below the primary content area.
- Shows all queued cascade requests in FIFO order, each with: trigger source, starting layer, and queued timestamp.
- Each queued entry has a cancel button that calls `DELETE /api/cascade/queue/:queue_id`.
- If the queue is empty, this section is hidden entirely (not shown with "0 items" text).

### Cascade Progress Indicator

A visual component showing the cascade's progression through the layer hierarchy:

**Layout:**
- Displays the cascade's layers as a horizontal or vertical sequence of steps, ordered from the starting layer to the topmost layer (policies). For example, a cascade starting from SA shows: SA → Strategic Objectives → Tactical Objectives → Policies.
- Each layer in the sequence shows three sub-steps: generation, critics, synthesis.

**State representation per layer:**
- **Completed**: the layer's step group is visually marked as done (e.g., checkmark icon, muted colour).
- **In progress**: the layer is highlighted with its identity colour. The current sub-step (generation, critics, or synthesis) is indicated — the active sub-step shows an animated indicator (spinner or pulse), completed sub-steps within the layer show as done, pending sub-steps show as upcoming.
- **Upcoming**: layers not yet started are shown in a dimmed/muted state.
- **Failed/paused**: the layer where the error occurred is highlighted with the error/warning colour. The failed sub-step is specifically marked.

**Data source:**
- Reads from the cascade store: `currentLayer`, `currentStep`, `status`, `startingLayer`.
- The list of layers from starting layer to policies is derived from the layer hierarchy (step 007's layer definitions, available via the layer store).

**Responsive behaviour:**
- At wider viewports: a horizontal progress bar or step sequence.
- At narrower viewports: collapses to a compact representation (e.g., "Layer 2/4: Strategic Objectives — Critics" with a progress bar).

### Streaming Text Panel

A scrollable panel that displays the agent's reasoning text as it streams in:

**Rendering:**
- Reads `streamingText` from the cascade store, which is a string that grows as `AgentTextChunk` events arrive.
- Renders the text in a monospace or slightly stylised readable font within a bordered, scrollable container. The container has a fixed height (e.g., 60-70% of the available vertical space) and scrolls vertically.
- While an agent is actively streaming (cascade status is running and text is being appended), shows a blinking cursor or subtle animation at the end of the text to indicate activity.
- When no text is available yet (generation just started, waiting for first chunk), shows a subtle loading indicator.

**Auto-scroll:**
- Uses the `useAutoScroll` hook (from cc-runner or Policy Factory's adapted version) to automatically scroll to the bottom as new text arrives.
- If the user scrolls up to read earlier content, auto-scrolling pauses. It resumes when the user scrolls back to the bottom.

**Text clearing:**
- The streaming text is cleared by the cascade store when a new agent starts (step 013's dispatcher handles this — when a new `LayerGenerationStarted`, `CriticStarted`, or `SynthesisStarted` event arrives, the store clears `streamingText`). The panel reflects this: when the agent switches, the old text disappears and new text begins streaming.

**Empty states:**
- When the cascade is idle (no running cascade), the streaming panel is not shown. The page shows the idle/history view instead.
- When the cascade is paused, the streaming panel shows the last text that was streaming before the pause (the store preserves it), along with the error overlay.

### Cascade History Section

A section shown when no cascade is running (idle state) displaying recent cascade activity:

**Most recent cascade summary:**
- If a cascade completed recently, show a summary card at the top: trigger source, starting layer, all layers processed, completion timestamp, overall success/failure status.
- The summary is fetched from `GET /api/cascade/status` (which includes the most recently completed cascade ID) and optionally `GET /api/cascade/:cascade_id` for the full detail.

**History list:**
- Below the summary, a scrollable list of recent cascade runs fetched from `GET /api/cascade/history`.
- Each entry shows: trigger source icon/label, starting layer with identity colour badge, status (completed/failed/cancelled) with colour-coded badge, and timestamps (started, duration).
- Clicking a history entry could expand it to show more detail (layers processed, agent runs) — loaded lazily from `GET /api/cascade/:cascade_id`.
- The list is limited to a reasonable number (e.g., 20 entries) matching the API default.

**Data fetching:**
- History data is fetched on page mount when the cascade is idle.
- If the cascade is running, the history section is not visible (the live view takes over), so history data is not fetched unnecessarily.
- When a cascade completes (detected via cascade store status transitioning from running to idle), the history is re-fetched to include the newly completed cascade.

### Cascade Control Actions

Action buttons integrated into the page for controlling cascades:

**Pause button:**
- Shown when a cascade is running.
- Calls `POST /api/cascade/:cascade_id/pause`.
- Shows loading state while the request is in flight.
- On success, the cascade store transitions to paused via the WebSocket event (the button doesn't need to manually update the store).

**Resume button:**
- Shown when a cascade is paused.
- Calls `POST /api/cascade/:cascade_id/resume`.
- Visual emphasis — this is a primary action when the cascade is in error state.

**Cancel button:**
- Shown when a cascade is paused.
- Calls `POST /api/cascade/:cascade_id/cancel`.
- Styled as a destructive action (danger variant).
- Shows a brief confirmation step before executing (e.g., a "Confirm cancel" state on the button, or a small inline confirmation). Cancelling releases the cascade lock and discards remaining work, so it should not be accidental.

**Queue entry cancel:**
- Each queued cascade entry has a cancel/remove button.
- Calls `DELETE /api/cascade/queue/:queue_id`.
- Removes the entry from the displayed queue on success.

All control actions use the authenticated HTTP client for API calls.
