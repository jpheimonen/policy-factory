# 019: Tiered heartbeat system

## Context

The heartbeat system is Policy Factory's autonomous monitoring capability. While cascades respond to explicit triggers (user input, manual refresh, initial seeding), the heartbeat system monitors the outside world for changes that the policy stack should respond to — without any human intervention. It ensures the situational awareness layer stays current and that the entire policy stack adapts to new developments automatically.

The heartbeat is a four-tier escalation chain where each tier only fires when the previous tier determines that further analysis is warranted. A quiet day costs a single cheap Haiku call (Tier 1 scans yle.fi, finds nothing noteworthy, and stops). A major policy-relevant event triggers the full chain: Tier 1 flags items → Tier 2 triages with web search → Tier 3 updates the SA layer with a full agent → Tier 4 triggers a cascade and idea generation. Cost scales with significance.

Prior steps have established everything the heartbeat needs:
- The agent framework (step 014) provides session wrappers with per-role model selection. Heartbeat roles are: heartbeat-skim (Haiku), heartbeat-triage (Sonnet), heartbeat-sa-update (Opus).
- Three heartbeat prompt templates (step 014): `prompts/heartbeat/skim.md`, `prompts/heartbeat/triage.md`, `prompts/heartbeat/sa-update.md`.
- The event system (step 012) defines heartbeat events: `HeartbeatStarted`, `HeartbeatTierCompleted`, `HeartbeatCompleted`.
- The cascade trigger function (step 015) handles lock acquisition, queueing, and background execution — Tier 4 just calls this function.
- The idea generation function (step 018) generates and auto-evaluates ideas — Tier 4 calls this after the cascade.
- The data layer utilities (step 007) read layer content for building agent context.
- The agent run mixin (step 015) records each agent invocation.

This step introduces APScheduler, which has no precedent in cc-runner. APScheduler runs in-process within the FastAPI application — no external cron jobs, no separate scheduler service. This makes deployment simple: the heartbeat runs wherever the FastAPI app runs, including Docker containers.

**From APScheduler (external library — no cc-runner precedent):**

- APScheduler is a Python scheduling library that runs jobs within the application process. Policy Factory uses APScheduler's async scheduler to integrate with FastAPI's event loop. The scheduler is started during the app lifespan startup and shut down during shutdown. The heartbeat job is registered as a recurring interval trigger (e.g., every 4 hours). APScheduler handles missed job executions gracefully — if the server was down during a scheduled time, the job runs on next startup.

## Deliverables

### Heartbeat Mixin

A store mixin at `src/policy_factory/store/heartbeat.py` added to the composed `PolicyStore`:

**Heartbeat runs table:**
- Each heartbeat run tracks: unique ID, started timestamp, completed timestamp (nullable — populated when the run finishes), highest tier reached (integer 1–4, updated as each tier completes), trigger (scheduled or manual — a string indicating how this heartbeat was initiated), and a structured log as JSON capturing the outcome of each tier that ran.

**Structured log format:**
- The structured log is a JSON object with one entry per tier that ran. Each tier entry includes: the tier number, whether it escalated to the next tier (boolean), the outcome description (a brief text summarizing what the tier found — e.g., "Flagged 3 items related to EU AI Act developments" or "No noteworthy items found"), the agent run ID (linking to the agent run record for this tier's agent invocation), and the start and end timestamps for the tier.
- Tiers that did not run (because a lower tier did not escalate) are absent from the structured log.
- This structured log is what the heartbeat history page and activity feed display. It provides a complete audit trail of each heartbeat run at a glance.

**Heartbeat mixin methods:**
- `create_heartbeat_run(trigger)` — creates a heartbeat run record with the started timestamp and initial highest tier of 0 (no tier completed yet). Returns the heartbeat run ID.
- `update_heartbeat_tier(run_id, tier, escalated, outcome, agent_run_id)` — appends a tier entry to the structured log and updates the highest tier reached. Called after each tier completes.
- `complete_heartbeat_run(run_id)` — sets the completed timestamp. Called when the heartbeat finishes (either because a tier did not escalate, or after Tier 4 completes).
- `get_heartbeat_run(run_id)` — returns the full heartbeat run record by ID.
- `list_heartbeat_runs(limit, offset)` — returns recent heartbeat runs in reverse chronological order with all fields. Used by the heartbeat history endpoint and activity feed. Default limit of 20.
- `get_latest_heartbeat_run()` — returns the most recent heartbeat run record. Used by the UI to show the last heartbeat status.

### Heartbeat Orchestrator

An async function at `src/policy_factory/heartbeat/orchestrator.py` that drives the four-tier escalation chain:

**Function signature:**
- Accepts: the trigger type (scheduled or manual), the store instance, the EventEmitter, a callable for creating agent sessions (the session factory from step 014), the cascade trigger function (from step 015), and the idea generation function (from step 018). Dependencies are passed as arguments, not imported directly — keeping the function testable and decoupled.

**Orchestration flow:**

1. Create a heartbeat run record in the store. Emit a `HeartbeatStarted` event with the heartbeat run ID.

2. **Tier 1 — News Skim:**
   a. Gather context: read the current SA layer's narrative summary using data layer utilities. Include the current date (so the agent knows what "recent" means).
   b. Build the Tier 1 prompt: load `prompts/heartbeat/skim.md`, substitute the current date and SA summary.
   c. Create an agent session with the heartbeat-skim model (Haiku), working directory set to `data/`, and the EventEmitter for streaming. The agent label is "Heartbeat — news skim".
   d. Record an agent run (type: heartbeat-skim).
   e. Run the agent session.
   f. Record agent run completion.
   g. Parse the agent's output to determine the result:
      - The skim prompt instructs the agent to output a clear decision: either a statement that nothing noteworthy was found, or a structured list of flagged items.
      - The parser looks for indicators of "nothing noteworthy" (the agent's output contains a clear nothing-found marker as instructed by the prompt). If found, the tier result is "no escalation".
      - If the output contains flagged items, extract them as a text block for passing to Tier 2. The tier result is "escalate" with the flagged items.
   h. Update the heartbeat run record (tier 1 completed, escalated or not, outcome summary).
   i. Emit a `HeartbeatTierCompleted` event (tier 1, outcome, escalated boolean).
   j. If no escalation: complete the heartbeat run, emit `HeartbeatCompleted` (highest tier: 1), return.

3. **Tier 2 — Triage Analysis:**
   a. Build the Tier 2 prompt: load `prompts/heartbeat/triage.md`, substitute the flagged items from Tier 1 and the current SA summary.
   b. Create an agent session with the heartbeat-triage model (Sonnet), working directory set to `data/`, and the EventEmitter. The agent label is "Heartbeat — triage analysis". The triage agent uses web search to find international sources and context for each flagged item.
   c. Record an agent run (type: heartbeat-triage).
   d. Run the agent session.
   e. Record agent run completion.
   f. Parse the agent's output to determine the result:
      - The triage prompt instructs the agent to assess each flagged item and produce a clear recommendation: either no items warrant SA updates, or a list of items that warrant updates with reasoning.
      - The parser checks for the "no updates warranted" signal. If found, the tier result is "no escalation".
      - If updates are warranted, extract the triage assessment (which items should be updated and why) as a text block for passing to Tier 3. The tier result is "escalate" with the triage assessment.
   g. Update the heartbeat run record (tier 2 completed, escalated or not, outcome summary).
   h. Emit a `HeartbeatTierCompleted` event (tier 2, outcome, escalated boolean).
   i. If no escalation: complete the heartbeat run, emit `HeartbeatCompleted` (highest tier: 2), return.

4. **Tier 3 — SA Update:**
   a. Gather the full SA layer content using the layer content gathering helper (step 016) — all items and the narrative summary as a text blob.
   b. Retrieve any pending feedback memos targeting the SA layer using the feedback memo mixin (step 017).
   c. Build the Tier 3 prompt: load `prompts/heartbeat/sa-update.md`, substitute the triage assessment from Tier 2, the full SA layer content, and the pending feedback memos (formatted as a list of memo texts, or a note that there are no pending memos).
   d. Create an agent session with the heartbeat-sa-update model (Opus), working directory set to `data/`, and the EventEmitter. The agent label is "Heartbeat — SA update".
   e. Record an agent run (type: heartbeat-sa-update).
   f. Run the agent session. This is a full Claude Code agent that reads and writes SA markdown files in the `data/` directory, creates or modifies items, and updates the narrative summary (README.md).
   g. Record agent run completion.
   h. After the agent completes successfully: auto-commit the data git repo with a descriptive message (e.g., "Heartbeat SA update — [brief summary of what changed]"). The commit message can incorporate the outcome summary from the triage assessment.
   i. Update the heartbeat run record (tier 3 completed, always escalates to tier 4 after successfully updating SA, outcome summary).
   j. Emit a `HeartbeatTierCompleted` event (tier 3, outcome, escalated: true).

5. **Tier 4 — Full Cascade + Idea Generation:**
   a. Trigger an upward cascade from the SA layer by calling the cascade trigger function with trigger source "heartbeat" and starting layer "situational-awareness". The cascade will process strategic-objectives → tactical-objectives → policies with full critic passes at each layer.
   b. The cascade trigger function returns immediately (the cascade runs asynchronously). Tier 4 does not wait for the cascade to complete — the cascade runs on its own, and its progress is tracked through its own events and status endpoints.
   c. Trigger AI idea generation by calling the idea generation function (step 018). This generates ideas responding to the newly discovered developments. Like the cascade, idea generation runs asynchronously as a background task.
   d. Update the heartbeat run record (tier 4 completed, outcome: "triggered cascade and idea generation").
   e. Emit a `HeartbeatTierCompleted` event (tier 4, outcome, escalated: false — tier 4 is the final tier).
   f. Complete the heartbeat run. Emit `HeartbeatCompleted` (highest tier: 4).

**Error handling:**

- If a tier's agent session fails (after the session wrapper's internal retries), the heartbeat run stops at that tier. The run is completed with the highest tier that succeeded (the failed tier is not counted as successful). The error is logged. The heartbeat does not retry failed tiers — the next scheduled heartbeat will run fresh.
- If Tier 3 fails (SA update agent error), Tier 4 does not run. The SA layer was not updated, so there is nothing new to cascade.
- If the cascade trigger (Tier 4) returns a queue entry (because a cascade is already running), this is not an error — the heartbeat's cascade request is queued and will execute when the lock is released. The heartbeat run is still completed normally with highest tier 4.
- All errors are captured in the heartbeat run's structured log (the tier entry includes the error information) and in the agent run record.

**Concurrency guard:**
- Only one heartbeat run should execute at a time. If the scheduler fires while a previous heartbeat is still running, the new execution is skipped (with a log message). This is handled via APScheduler's built-in job coalescing: if a job execution is missed or overlapping, it coalesces into a single execution.
- Alternatively, a simple in-memory flag (or checking for a heartbeat run with no completed timestamp in the store) can serve as a concurrency guard.

### APScheduler Integration

Integration of APScheduler with FastAPI's lifecycle:

**Scheduler setup:**
- The APScheduler async scheduler is instantiated and managed through the deps module or directly in the app lifespan.
- During app lifespan startup: create the scheduler, add the heartbeat job with an interval trigger (configurable interval, default 4 hours), and start the scheduler.
- During app lifespan shutdown: gracefully shut down the scheduler, waiting for any running heartbeat to complete (with a reasonable timeout).
- The heartbeat interval is configurable via an environment variable (e.g., `POLICY_FACTORY_HEARTBEAT_INTERVAL_HOURS`). The default is 4 hours. Setting the interval to 0 disables the heartbeat scheduler (useful for development or testing).

**Scheduler management in deps.py:**
- A module-level reference to the scheduler instance.
- `get_scheduler()` — returns the scheduler instance.
- `init_scheduler(store, emitter, session_factory, cascade_trigger, idea_generator)` — creates and configures the scheduler with the heartbeat job. All dependencies the heartbeat orchestrator needs are bound at this point (via a closure or partial function) so the scheduler job function has access to them.
- `shutdown_scheduler()` — stops the scheduler gracefully.

**Heartbeat job function:**
- The job function registered with APScheduler is a thin wrapper that calls the heartbeat orchestrator with trigger type "scheduled" and the bound dependencies (store, emitter, session factory, cascade trigger, idea generator).
- The job function catches and logs any unexpected exceptions to prevent the scheduler from silently breaking.

**App lifespan additions:**
- In the existing `create_app()` lifespan context manager, after the existing initializations (store, event emitter, etc.), add: initialize the scheduler with `init_scheduler()`, start it, and yield. On shutdown, call `shutdown_scheduler()` before the existing teardown steps.

### Heartbeat Router

A router at `src/policy_factory/server/routers/heartbeat.py` with prefix `/api/heartbeat` and tag `heartbeat`. All endpoints require JWT authentication.

**POST `/api/heartbeat/trigger`** — Manually trigger a heartbeat run.
- Initiates a heartbeat run with trigger type "manual".
- Launches the heartbeat orchestrator as a background asyncio task (the user doesn't wait for the full heartbeat to complete).
- Returns immediately with the heartbeat run ID and a status indicating the heartbeat has started.
- If a heartbeat is already running, returns 409 (conflict) indicating a heartbeat is in progress. The check uses the concurrency guard (in-memory flag or store query for uncompleted runs).

**GET `/api/heartbeat/history`** — Get recent heartbeat run history.
- Returns heartbeat runs in reverse chronological order.
- Accepts optional query parameters: `limit` (default 20, max 100).
- Each entry includes: run ID, trigger type (scheduled/manual), started timestamp, completed timestamp, highest tier reached, and the full structured log (tier outcomes, escalation decisions, agent run IDs).

**GET `/api/heartbeat/latest`** — Get the most recent heartbeat run.
- Returns the most recent heartbeat run record with all fields.
- Returns null/empty if no heartbeat has ever run.
- Used by the frontend to display a "last heartbeat" indicator (e.g., "Last heartbeat: 2 hours ago, Tier 1, nothing noteworthy").

**GET `/api/heartbeat/status`** — Get heartbeat system status.
- Returns: whether the scheduler is active (running), the configured interval, the next scheduled run time, whether a heartbeat is currently executing, and the most recent heartbeat run summary.
- Used by the admin panel to show heartbeat configuration status.

### Tier Output Parsing

The heartbeat orchestrator needs to parse agent output to determine escalation decisions. The parsing approach is consistent across tiers:

**Tier 1 parsing:**
- The skim prompt instructs the agent to end its response with a clear signal: either a specific "nothing noteworthy" marker or a structured list of flagged items.
- The parser looks for the nothing-noteworthy marker in the agent's output (case-insensitive substring matching on the marker phrase defined in the prompt).
- If the marker is not found, the parser assumes there are flagged items and extracts the full agent output (after meditation filtering) as the flagged items text to pass to Tier 2.
- The parser does not need to extract individual items as structured data — Tier 2 receives the full skim output and interprets it.

**Tier 2 parsing:**
- The triage prompt instructs the agent to end with a clear recommendation: either "no updates warranted" or a structured assessment of which items warrant SA updates.
- The parser follows the same approach: look for the no-updates marker, otherwise treat the output as a triage assessment to pass to Tier 3.

**Tier 3 parsing:**
- Tier 3 does not need output parsing for escalation — it always escalates to Tier 4 if successful (every SA update should cascade). The agent's output is the actions it took (files modified), which is captured in the agent run log.

This lenient parsing approach (marker-based detection, full-text fallback) is consistent with the critic output parsing strategy in step 016 — when in doubt, preserve the raw text and proceed.
