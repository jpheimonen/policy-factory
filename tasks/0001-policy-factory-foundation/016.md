# 016: Critic runner and synthesis

## Context

The cascade orchestrator (step 015) drives the layer-by-layer update sequence. At each layer, after the generation agent finishes, the orchestrator calls two functions: the critic runner and the synthesis runner. Step 015 deliberately left these as black-box callables — the orchestrator passes the layer slug, cascade ID, store, emitter, and an agent session factory, and expects structured results back. This step implements those two functions.

The critic runner is the system's mechanism for ensuring no single perspective dominates the policy analysis. Six ideological archetype critics — Realist, Liberal-institutionalist, Nationalist-conservative, Social-democratic, Libertarian, and Green/Ecological — each examine the same layer content through their distinct lens. Running all 6 in parallel keeps cascade latency manageable: since each critic uses a cheaper/faster model (default Sonnet), the total critic phase takes roughly as long as one critic run, not 6× as long.

The synthesis runner operates after all critics complete. It receives all 6 critic outputs and integrates them into a unified assessment. Crucially, the synthesis does not seek false consensus — it explicitly identifies unresolved tensions between perspectives. A Realist and a Liberal-institutionalist may fundamentally disagree on an approach to NATO cooperation; the synthesis names that tension rather than papering over it.

This step also introduces the **score mixin** — SQLite storage for critic results and synthesis results, linked to a cascade run and layer. The architecture spec defines a score mixin for idea evaluation (6-axis scores), but the critic assessment storage shares the same table pattern: structured results linked to a parent entity (cascade + layer for critics, idea for evaluations). This step builds the critic/synthesis result storage; step 018 extends or parallels it for idea scores.

The critic runner and synthesis runner are also used outside the cascade. The idea evaluation pipeline (step 018) runs the same 6 critics against individual ideas, followed by the same synthesis. The functions built here are generic enough to serve both use cases — they accept "content to critique" without assuming it's always a layer. The layer-specific context gathering (reading layer items, narrative summary) happens in the orchestrator before calling these functions.

## Reuse Opportunities

- The **`asyncio.gather(*tasks, return_exceptions=True)` pattern** from cc-runner is directly applicable for running all 6 critic agents in parallel. The return_exceptions flag ensures that if one critic fails, the gather still completes and the other 5 results are available. The critic runner inspects each result and handles exceptions individually.
- The **agent session creation pattern** from step 014 is used 7 times in this step: once per critic (6) and once for synthesis. The pattern is: resolve model for role → create session config → create session with emitter → build prompt → run → record result. This repeated pattern could be extracted into a helper that creates, runs, and records an agent session in one call.
- The **prompt construction helper** from step 014 is used directly: load critic prompt by archetype name, substitute the layer content as a template variable.
- The **agent run recording** from step 015 is used for all 7 agent invocations. Each critic and the synthesis agent get an agent run record for auditability.
- The **store mixin pattern** from steps 004 and 015 is extended for critic and synthesis result storage.

## Deliverables

### Critic Result Mixin

A store mixin at `src/policy_factory/store/critic_result.py` added to the composed `PolicyStore`:

**Critic results table:**
- Stores individual critic assessments: unique ID, cascade ID (nullable — for idea evaluation, this is null), layer slug (nullable — for idea evaluation, this is null), idea ID (nullable — for layer critiques in a cascade, this is null), critic archetype name (the slug: realist, liberal-institutionalist, nationalist-conservative, social-democratic, libertarian, green-ecological), the full assessment text (the raw output from the critic agent after meditation filtering), a structured assessment as JSON (parsed from the critic's output if it follows the expected structure: agreement/disagreement per item, reasoning, alternatives, and an overall score), the agent run ID (linking to the agent run record), and created timestamp.
- Indexed on cascade_id + layer_slug for efficient cascade-scoped queries, and on idea_id for idea-scoped queries.

**Synthesis results table:**
- Stores synthesis assessments: unique ID, cascade ID (nullable), layer slug (nullable), idea ID (nullable), the full synthesis text (raw output after meditation filtering), a structured synthesis as JSON (parsed from the synthesis output: consensus points, tension points, recommendations), the agent run ID, and created timestamp.
- Indexed similarly to critic results.

**Critic result methods:**
- `store_critic_result(cascade_id, layer_slug, idea_id, archetype, assessment_text, structured_assessment, agent_run_id)` — creates a critic result record. Returns the record ID.
- `get_critic_results(cascade_id, layer_slug)` — returns all 6 critic results for a specific cascade run and layer. Used by the synthesis runner and by the layer detail page (step 010).
- `get_critic_results_for_idea(idea_id)` — returns all critic results for an idea evaluation. Used by the idea detail view (step 021).
- `get_critic_result_by_archetype(cascade_id, layer_slug, archetype)` — returns a single critic's result. Used when the UI displays a specific critic's assessment.

**Synthesis result methods:**
- `store_synthesis_result(cascade_id, layer_slug, idea_id, synthesis_text, structured_synthesis, agent_run_id)` — creates a synthesis result record. Returns the record ID.
- `get_synthesis_result(cascade_id, layer_slug)` — returns the synthesis result for a specific cascade run and layer.
- `get_synthesis_result_for_idea(idea_id)` — returns the synthesis result for an idea evaluation.

**Latest results helpers:**
- `get_latest_critic_results(layer_slug)` — returns the most recent set of critic results for a given layer (from the latest completed cascade that processed that layer). This is what the layer detail page displays by default — the user sees the most recent critic assessment, not historical ones.
- `get_latest_synthesis_result(layer_slug)` — returns the most recent synthesis result for a given layer.

### Critic Runner

An async function at `src/policy_factory/cascade/critic_runner.py` that runs all 6 critics against a body of content:

**Function signature contract (matching the callable expected by step 015's orchestrator):**
- Accepts: layer slug, cascade ID (nullable for idea evaluation), content to critique (the full layer content: items + narrative summary, or an idea text), the store instance, the EventEmitter, and a callable for creating agent sessions.
- Returns: a result object containing the individual critic results (6 entries, one per archetype, each with success/failure status and the assessment text), a count of successful critics, and a count of failed critics.

**Execution flow:**

1. For each of the 6 critic archetypes, prepare a coroutine that:
   a. Emits a `CriticStarted` event with the cascade ID, layer slug, and archetype name.
   b. Creates an agent run record (started) via the store.
   c. Builds the critic prompt using the prompt construction helper: loads the archetype-specific critic template from `prompts/critics/{archetype-slug}.md`, substitutes the content to critique as a template variable. The meditation preamble is prepended as with all agents.
   d. Creates an agent session with the critic model (resolved via the model resolver for role "critic"), working directory set to `data/`, and the EventEmitter for streaming. The agent label is set to the archetype's agent label (e.g., "Realist critic").
   e. Runs the agent session with the assembled prompt.
   f. On success: records the agent run completion, parses the assessment from the agent output (attempting to extract structured data — agreement/disagreement, reasoning, alternatives, score — but falling back to storing the raw text if parsing fails), stores the critic result in the database.
   g. On failure: records the agent run as failed, stores a failed critic result (null assessment).
   h. Emits a `CriticCompleted` event.
   i. Returns the result (success/failure + assessment).

2. Launch all 6 coroutines concurrently using `asyncio.gather` with `return_exceptions=True`.

3. After all 6 complete (or fail), inspect the results:
   - Collect successful assessments.
   - Log failed critics with their error details.
   - The overall critic run succeeds if at least one critic succeeded (partial results are acceptable — the cascade continues). The critic run fails only if all 6 critics failed.

4. Return the collected results to the caller (the orchestrator).

**Parsing critic output:**

The critic prompt instructs the AI to produce structured output (agreement/disagreement per item, reasoning, alternatives, score). The critic runner attempts to parse this structure from the agent's text output. The parsing strategy:
- The prompt template instructs the AI to format its assessment in a consistent way (e.g., with clear section headers for each item, explicit agreement/disagreement labels, and a numerical score at the end).
- The parser attempts to extract these sections from the text output. If the output follows the expected format, a structured assessment object is created and stored in the database alongside the raw text.
- If parsing fails (the AI produced free-form text instead of the expected structure), the raw text is stored as the assessment text, and the structured assessment field is null. This is not an error — the critic run succeeds, but the structured data is unavailable. The UI falls back to displaying the raw text.
- The parser should be lenient: some sections may be present while others are missing. Extract what can be extracted.

### Synthesis Runner

An async function at `src/policy_factory/cascade/synthesis_runner.py` that integrates all critic assessments:

**Function signature contract (matching the callable expected by step 015's orchestrator):**
- Accepts: layer slug, cascade ID (nullable), the critic results (as returned by the critic runner — a list of individual assessments with archetype labels), the original content that was critiqued, the store instance, the EventEmitter, and a callable for creating agent sessions.
- Returns: a result object containing the synthesis text, an optional structured synthesis (consensus points, tension points, recommendations), success/failure status.

**Execution flow:**

1. Emit `SynthesisStarted` event.
2. Create an agent run record (started) via the store.
3. Build the synthesis prompt: load the synthesis template from `prompts/synthesis/synthesis.md`, substitute template variables including the layer slug, the original layer content, and all successful critic outputs. Each critic output is labeled with its archetype name in the prompt so the synthesis agent can attribute perspectives correctly. Failed critics are noted as absent (e.g., "The Realist critic's assessment is not available due to an error").
4. Create an agent session with the synthesis model (resolved for role "synthesis"), working directory set to `data/`, and the EventEmitter for streaming. The agent label is "Synthesis".
5. Run the agent session.
6. On success: record the agent run completion, parse the synthesis output (attempting to extract consensus points, tension points, and recommendations), store the synthesis result in the database.
7. On failure: record the agent run as failed, return a failure result.
8. Emit `SynthesisCompleted` event.
9. Return the result.

**The synthesis runs even with partial critic results.** If only 4 out of 6 critics succeeded, the synthesis still runs with those 4 assessments. The synthesis prompt notes which perspectives are missing so the synthesis can acknowledge the gaps. The synthesis only does not run if zero critics succeeded (no input to synthesize — in that case, the orchestrator's error handling takes over).

### Layer Content Gathering Helper

A utility function (in the cascade module or extending the data layer utilities) that gathers the full content of a layer as a single text blob suitable for passing to critic and synthesis agent prompts:

- Reads the layer's narrative summary (README.md).
- Reads all items in the layer with their frontmatter and markdown body.
- Assembles them into a formatted text string: the narrative summary first, followed by each item with its title and content.
- This formatted text is what gets substituted into the critic and synthesis prompt templates as the `{layer_content}` variable.
- Used by the cascade orchestrator before calling the critic runner, so the orchestrator gathers content once and passes it to both the critic runner and synthesis runner.
