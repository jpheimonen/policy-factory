# 002: Design system and theming

## Context

Policy Factory is shown to politicians and policy professionals — the UI must look polished and professional, not like a developer tool. This step establishes the complete visual foundation: a dual-theme design token system (dark and light), global styles, layer identity colors, and a starter set of atomic components that every subsequent UI step depends on.

cc-runner's design system is dark-only and developer-focused. Policy Factory needs a dual-theme system with system-preference detection, a more refined Notion/Linear aesthetic, Inter font instead of system fonts, and five distinct layer identity colors so users can visually identify which policy layer they're looking at. The theme token structure from cc-runner is a good starting point but needs restructuring to support two theme objects with the same shape.

This step also establishes the theme persistence and switching mechanism (localStorage + zustand store) so the preference survives page reloads. Step 001 created the bare minimum App.tsx — this step wraps it in ThemeProvider, adds GlobalStyles, and creates the foundational atoms.

## Deliverables

### Theme Token System

Two complete theme objects (dark and light) that share the same TypeScript type. Both must have identical keys with different values. The theme type should cover:

**Colors:**
- `bg` — primary, secondary, tertiary, elevated backgrounds. Dark theme uses dark grays/near-blacks; light theme uses whites/light grays.
- `text` — primary, secondary, muted. Dark theme uses light text; light theme uses dark text.
- `border` — default and subtle. Borders should be subtle in both themes (the Notion/Linear aesthetic avoids heavy borders).
- `accent` — blue, green, red, yellow, orange, purple, cyan. These should be visible and pleasant in both themes. The dark theme can use brighter/more saturated accents; the light theme should use slightly muted/darker accents for readability on white backgrounds.
- `status` — active, success, warning, error, pending. Each with bg, border, and text sub-tokens. Use semi-transparent versions of the accent colors so they work on both theme backgrounds.
- `layers` — five named layer colors, one per policy layer: values, situational-awareness, strategic-objectives, tactical-objectives, policies. Each with a primary color, a faded/background variant, and a text variant. These colors should be visually distinct from each other so users can instantly identify which layer they're looking at. Choose colors that work in both dark and light themes.

**Typography:**
- `fonts.sans` — Inter as the primary font, with a system font fallback stack. Inter must be loaded (via Google Fonts CDN link in index.html or bundled).
- `fonts.mono` — A monospace font for any code/technical display (JetBrains Mono or similar with fallbacks).
- `fontSizes` — A scale from xs through xl, sized appropriately for a professional dashboard (not a developer tool — slightly larger than cc-runner's scale which is quite compact).

**Spacing, Radii, Shadows:**
- `spacing` — A scale from xs through xxl. Generous spacing consistent with the Notion/Linear aesthetic.
- `radii` — A scale from sm through xl. Slightly rounded corners consistent with modern SaaS apps.
- `shadows` — sm, md, lg. The dark theme uses darker shadows; the light theme uses softer, more visible shadows. Overall, prefer subtle borders over heavy shadows (Notion style).

**Animation and Layout:**
- `transitions` — fast, normal, slow. Used for hover states, theme transitions, interactive elements.
- `zIndices` — dropdown, sticky, modal, tooltip. Standard stacking context values.

### TypeScript Theme Augmentation

A `styled.d.ts` file that augments styled-components' `DefaultTheme` with the shared theme type. This enables type-safe theme access in all styled components — theme property typos become compile-time errors.

### Global Styles

A `GlobalStyles` component (using `createGlobalStyle`) that:
- Applies a CSS reset (box-sizing border-box, zero margins/padding)
- Sets the body font to Inter from the theme tokens
- Sets background and text colors from the theme
- Styles scrollbars using theme tokens (works in both dark and light modes)
- Sets html, body, and #root to full height
- Sets a base line-height for readability
- Applies smooth font rendering (antialiasing)

### Theme Store

A zustand store for theme preference management:
- Tracks the user's preference: "system", "dark", or "light"
- On initial load, reads the preference from localStorage (default: "system")
- When set to "system", detects the OS-level preference using `prefers-color-scheme` media query and returns the appropriate theme
- When explicitly set to "dark" or "light", uses that theme regardless of system preference
- Persists the preference to localStorage on change
- Exposes: the current preference, the resolved theme object (dark or light), and a setter function
- Listens for changes to the system `prefers-color-scheme` media query so the theme updates live if the user changes their OS setting while the app is open

### App Shell Integration

Update `App.tsx` (created in step 001) to:
- Import the theme store and get the resolved theme object
- Wrap the app in `ThemeProvider` with the resolved theme
- Include `GlobalStyles` inside the provider
- Wrap in the root `AppContainer` styled component

### Inter Font Loading

Include the Inter font in `index.html` via a Google Fonts CDN link (or bundle it). The font should include regular (400), medium (500), and semibold (600) weights — enough for body text, labels, and headings. Variable font preferred for smaller download size.

### Atom Components

A set of foundational styled components in `ui/src/components/atoms/`. These are the building blocks used by all page-level components in subsequent steps. Each atom should use transient props (`$variant`, `$size`, etc.) for styling variations, reference theme tokens for all visual properties, and include appropriate interactive states (hover, focus, disabled).

**Button:**
- Variants: primary (accent fill), secondary (subtle fill), danger (red fill), ghost (transparent, text-only), outline (border only)
- Sizes: sm, md, lg
- States: hover highlight, focus ring (accessibility), disabled (reduced opacity, no pointer events), loading (spinner animation replacing content)
- Full-width option

**Card:**
- Default card with subtle border, themed background
- Elevated variant with slight shadow or gradient
- Interactive variant with hover effect (for clickable cards like layer cards on the stack overview)
- Padding options

**Badge:**
- Variants: info (blue), success (green), warning (yellow), error (red), neutral (muted)
- Small, rounded, inline display
- Used for status indicators, layer labels, counts

**Input:**
- Sizes: sm, md
- States: default, focused (accent border), error (red border), disabled
- Full-width by default
- Placeholder styling from theme tokens

**Textarea:**
- Same base styling as Input (shared style fragment)
- Resizable vertically
- Used for idea submission, item editing

**Select:**
- Same base styling as Input
- Custom dropdown arrow styling
- Sized consistently with Input

**Toggle:**
- On/off switch component
- Used for theme switching and other boolean settings
- Animated transition between states

**IconButton:**
- Button variant for icon-only actions
- Square aspect ratio
- Ghost and outline variants
- Used for dismiss buttons, action icons

**Text:**
- Styled text component with variant and size props
- Variants: heading, body, label, caption, muted
- Makes it easy to apply consistent typography without inline styles

### Barrel Export

An `index.ts` barrel file in the atoms directory that re-exports all atom components for clean imports (e.g., `import { Button, Card, Badge } from '@/components/atoms'`).

### Storybook Stories (Optional but Recommended)

If the project has storybook configured (step 001 included the dev dependency), create `.stories.tsx` files for each atom component showcasing all variants and states. This serves as both documentation and a visual testing aid.
