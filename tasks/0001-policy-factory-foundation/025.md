# 025: End-to-end integration tests

## Context

All 24 prior steps have built the complete Policy Factory application — from the project scaffold through authentication, the five-layer data model, cascade orchestration, the critic system, idea evaluation, the tiered heartbeat, every frontend page, and the Docker deployment pipeline. This final step adds the integration and end-to-end test suite that verifies everything works together.

The testing.md spec defines three categories of tests: unit tests, integration tests, and browser/E2E tests. Unit tests are written alongside their respective features in steps 001–024 as part of normal development. This step focuses exclusively on the cross-cutting tests that verify multi-component interactions — the integration tests and browser/E2E tests that cannot be meaningfully written until all the pieces exist.

Integration tests exercise the full backend stack: FastAPI app with real SQLite databases, real file system operations in temporary directories, real git operations, and real WebSocket connections — but with agent execution mocked. Every test that involves Claude Code agent sessions (cascades, critics, idea evaluation, heartbeat tiers, input classification, seeding) must mock the agent framework's session wrapper so tests do not make real API calls. The mocked agents return deterministic, predictable output that allows the tests to verify orchestration logic, state transitions, event emission, data persistence, and API responses.

Browser/E2E tests use Playwright to drive a real browser against a running application instance (backend + frontend). These tests verify the full user experience: page navigation, form submissions, real-time WebSocket updates, and visual correctness. Like integration tests, agent execution is mocked at the backend level — the browser tests verify UI behaviour, not AI output quality.

### Test Infrastructure

**Backend test directory** at `tests/` containing all Python integration tests and shared fixtures.

**conftest.py** — A shared pytest configuration file providing reusable fixtures:

- A fixture that creates a fresh SQLite database in a temporary directory for each test (or test class). The database has all tables initialised via `init_db()`.
- A fixture that creates a temporary `data/` directory with the five-layer subdirectory structure and an initialised git repo, mimicking first-run initialisation. Pre-seeded values files are included.
- A fixture that creates a fully configured FastAPI `TestClient` with initialised dependencies (store, ws_manager, event_emitter, broadcast handler) using the temp database and temp data directory. All agent-related dependencies are wired to mock implementations.
- A fixture that registers an admin user and returns a valid JWT for authenticated API requests.
- A fixture that registers a regular (non-admin) user and returns a valid JWT.
- A helper function or fixture that creates an authenticated test client (a TestClient with the JWT already set in the default headers).
- A mock agent session factory that replaces the real agent session wrapper. The mock accepts a role name (e.g., "generator-values", "critic-realist", "heartbeat-skim") and returns deterministic output appropriate for that role. For generation agents, the mock writes predictable markdown files to the data directory. For critics, the mock returns structured assessment text. For synthesis, the mock returns a balanced assessment. For heartbeat tiers, the mock returns escalation/no-escalation decisions as configured by the test.

### Backend Integration Tests

Each integration test section below corresponds directly to the "Integration Tests" section of testing.md. Tests use the shared fixtures from conftest.py.

**Auth flow end-to-end** — A test module verifying the complete authentication lifecycle:

- Register the first user and verify they receive admin role. Log in with their credentials and access a protected endpoint with the returned JWT.
- As admin, create a second user via the users endpoint. Log in as that user and verify they are non-admin.
- As a non-admin user, attempt admin-only operations (create user, delete user) and verify 403 responses.
- Call all protected endpoints without a JWT and verify 401 responses.

**Data layer round-trips** — A test module verifying markdown file operations through the API:

- Create a layer item via POST, read it back via GET, and verify frontmatter and body match exactly.
- Update a layer item via PUT, read it back, and verify the updated content is returned.
- Delete a layer item via DELETE, verify it returns 204, then verify the item no longer appears in the listing.
- Create an item and verify it appears in the layer listing with correct summary metadata from frontmatter.
- Create an item and verify that a git commit was created in the data repo with a descriptive message.
- Create items in two different layers that reference each other via frontmatter fields. Verify the cross-layer reference endpoint returns bidirectional references.

**First-run initialisation** — A test module verifying the application's bootstrap behaviour:

- Start the app with no data directory. Verify: the data directory is created, a git repo is initialised inside it, all five layer subdirectories exist, pre-seeded values files exist in the values directory with valid frontmatter.
- Start the app with an existing populated data directory. Verify it is not overwritten or re-initialised.

**Cascade orchestrator end-to-end (mocked agents)** — A test module verifying cascade orchestration with mock agents that write predictable files:

- Trigger a cascade from the values layer via the API. Verify it processes all 5 layers in sequence (values → SA → strategic → tactical → policies), running generation + critics + synthesis at each layer.
- Trigger a cascade from the policies layer. Verify it only processes the policies layer.
- Trigger two cascades simultaneously. Verify the second one queues behind the first and starts automatically when the first completes.
- Configure the mock agent to fail on a specific layer. Verify the cascade pauses with the correct error state, layer, and step recorded.
- Resume the paused cascade via the API. Verify it retries from the failed step and continues.
- Cancel a paused cascade via the API. Verify the lock is released and the next queued cascade starts.
- Connect a WebSocket client (authenticated with JWT), trigger a cascade, and verify the client receives cascade lifecycle events (started, layer generation started/completed, critic started/completed, synthesis started/completed, cascade completed) in the correct order.

**Critic system end-to-end (mocked agents)** — A test module verifying the critic pipeline:

- Run the critic system against a layer. Verify all 6 critic assessments are stored in the database with the correct archetype names and linked to the cascade run and layer.
- Verify the synthesis agent receives all 6 critic outputs. Verify the synthesis result is stored in the database linked to the cascade run and layer.
- Verify that critics are launched concurrently (not sequentially) by checking that the elapsed time is closer to the duration of one critic than the sum of all six.

**Idea pipeline end-to-end (mocked agents)** — A test module verifying the idea evaluation flow:

- Submit an idea via the API. Trigger its evaluation. Verify that 6-axis scores, all 6 critic results, and the synthesis assessment are stored in the database linked to the idea.
- Verify that idea evaluation does not acquire the cascade lock. Start a cascade, then submit and evaluate an idea concurrently. Both should proceed without blocking each other.
- Trigger AI idea generation via the API. Verify that new ideas are created in the database with source "AI".
- Run two idea evaluations concurrently. Verify both complete successfully with correct results stored independently.

**Heartbeat tier escalation (mocked agents)** — A test module verifying the heartbeat's tiered escalation:

- Configure the mock Tier 1 agent to return "nothing noteworthy". Trigger a heartbeat. Verify Tier 2 is not triggered and the run is logged with highest tier = 1.
- Configure Tier 1 to flag items and Tier 2 to return no actionable items. Trigger a heartbeat. Verify Tier 3 is not triggered and the run is logged with highest tier = 2.
- Configure Tiers 1–2 to escalate and Tier 3 to update SA files. Trigger a heartbeat. Verify Tier 4 triggers a cascade from the SA layer and the run is logged with highest tier = 4.

**Input classification end-to-end (mocked agents)** — A test module verifying the input → classification → cascade flow:

- Submit free-text input via the cascade trigger endpoint. Verify the classifier mock determines a target layer. Verify a cascade is triggered from that target layer.

**WebSocket event flow** — A test module verifying WebSocket behaviour:

- Connect a WebSocket client with a valid JWT. Trigger a cascade. Verify the client receives cascade lifecycle events (started, progress, completed) with correct event types and data.
- Connect a client, disconnect it mid-cascade (after receiving some events), reconnect, and replay missed events via the REST replay endpoint. Verify the replayed events are correct and that the client's combined event stream (pre-disconnect + replayed) has no duplicates (verified by event IDs).
- Attempt to connect a WebSocket client with an invalid JWT. Verify the connection is rejected.

**Event persistence and activity feed** — A test module verifying event storage and retrieval:

- Trigger a cascade. Fetch the activity endpoint. Verify cascade events (started, layer processing, completed) appear in the response.
- Submit an idea and trigger evaluation. Verify idea events (submitted, evaluation started, evaluation completed) appear in the activity feed.
- Verify activity endpoint filtering by event type: request only cascade events and verify no heartbeat or idea events are returned.
- Verify activity endpoint filtering by layer: request events for a specific layer slug and verify only events for that layer are returned.

**Frontend-backend integration** — A test module verifying the API contract between frontend and backend. These are API-level tests (using TestClient, not a browser) that verify the response shapes the frontend expects:

- The login endpoint returns a response shape that the frontend auth store can consume (JWT token in the expected field).
- The layers listing endpoint returns the response shape that the frontend layer store expects (array of layers with item counts, timestamps, narrative previews).
- The cascade status endpoint returns the response shape the frontend cascade store expects (status, current layer, current step, queue depth).
- The ideas listing endpoint returns the response shape the frontend idea store expects (array of ideas with status, scores when available).
