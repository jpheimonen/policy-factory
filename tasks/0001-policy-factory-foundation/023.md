# 023: Frontend admin panel and input panel

## Context

Two frontend features remain before the production build step: the admin panel for user management and the input panel for submitting free-text input to the system.

**Admin panel** — Policy Factory is a multi-user system with role-based access. The backend user management endpoints exist since step 005: `GET /api/users/` lists all users, `POST /api/users/` creates a new user, and `DELETE /api/users/:id` deletes a user. All require admin-level JWT auth. The auth store (step 006) already exposes `isAdmin` for role checking, and the navigation (step 009) already conditionally shows the admin link only to admins. But there is no admin page to consume these endpoints — the placeholder at `/admin` from step 009 needs replacing with a real user management interface.

The admin panel is also the natural place for system status information: heartbeat scheduler status (step 019 provides `GET /api/heartbeat/status`), seed status (step 017 provides `GET /api/seed/status`), and general system health. This gives administrators a single dashboard for managing both users and system configuration.

**Input panel** — The architecture spec describes a "prominent, always-accessible" way to submit free-text input to the system. This is distinct from the idea submission form (step 021, which submits ideas for evaluation) — the input panel submits information that the system classifies to a layer and then cascades. For example: "The EU just passed the AI Act" gets classified to the Situational Awareness layer and triggers a cascade from there. The backend for this exists: `POST /api/cascade/trigger` (step 017) accepts free-text input, runs the input classifier, and triggers a cascade.

The input panel is always accessible from any page — not just the home page. This means it is a persistent UI element integrated into the app shell (navigation area or a floating element), not a page-specific component.

## Deliverables

### Admin Page

Replace the placeholder `AdminPage` at `ui/src/pages/AdminPage.tsx` (route `/admin`) with the full admin panel. The page is only accessible to admin users.

**Access guard:**
- On mount, check the auth store's `isAdmin` value. If the user is not an admin, redirect to the home page (`/`). This is a safety net — the navigation already hides the admin link for non-admins, but a direct URL visit should also be handled.

**Page layout:**
- Page title using the `admin.title` translation key.
- Multiple sections stacked vertically: User Management, System Status.

### User Management Section

The primary section of the admin panel:

**User list:**
- Fetches the user list from `GET /api/users/` on mount.
- Displays each user as a row or card showing: email, role (with a badge — "Admin" in a distinct colour, "User" in a neutral colour), and creation date as a human-readable date.
- The current user (the logged-in admin) is visually indicated (e.g., "(you)" label next to their email, or a subtle highlight).

**Create user form:**
- A section or inline form above or below the user list for creating new accounts.
- Fields: email (required, validated to contain @), password (required, minimum length indication).
- A "Create user" button. Disabled when fields are empty or validation fails. Shows loading state during the API call.
- On success: clears the form, shows a brief success message (auto-dismissing after 3 seconds), and refreshes the user list to show the new user.
- On failure: displays the error message (e.g., "Email already registered" for 409 responses).
- New users are always created with the "user" role (the API enforces this).

**Delete user action:**
- Each user row (except the current admin's own row) has a delete button.
- The current admin's row does not show a delete button (admin cannot delete themselves — the API also enforces this, but the UI should not even offer the action).
- Clicking delete opens a confirmation dialog (ConfirmModal pattern from cc-runner) with the `danger` variant. The dialog shows the user's email and warns that this action is permanent.
- On confirmation: calls `DELETE /api/users/:id`. Shows loading state on the confirm button. On success, the user is removed from the list. On failure, shows an error message.
- The dialog is dismissed on cancel (Escape key or Cancel button).

### System Status Section

A read-only section displaying system configuration and health:

**Heartbeat status:**
- Fetches heartbeat system status from `GET /api/heartbeat/status` on mount.
- Displays: whether the scheduler is active, the configured interval (e.g., "Every 4 hours"), the next scheduled run time, whether a heartbeat is currently executing, and a summary of the most recent heartbeat run (when it ran, highest tier reached, outcome).
- A "Trigger heartbeat" button that calls `POST /api/heartbeat/trigger` to start a manual heartbeat run. Shows loading state while the request is in flight. Returns 409 if a heartbeat is already running — the button shows a disabled state in this case with a tooltip explaining that a heartbeat is in progress.

**Seed status:**
- Fetches seed status from `GET /api/seed/status` on mount.
- Displays: whether initial seeding has been performed. If not, shows a prompt or note indicating that the system needs seeding (with a link to the seed action or instructions).

**Data refreshing:**
- The system status section fetches data once on mount. A manual refresh button allows re-fetching all status information.

### Confirmation Dialog Component

### Input Panel

A persistent, always-accessible UI element for submitting free-text input to the system. This element is visible on every authenticated page, not just the home page.

**Placement options (implementer chooses the best approach):**
- Option A: A floating action button (FAB) in a fixed position (e.g., bottom-right corner) that opens a popover or small modal with the text input. This is unobtrusive and follows common SaaS patterns.
- Option B: A persistent section in the navigation sidebar (if sidebar navigation was chosen) with a compact text input.
- Option C: A collapsible drawer that slides in from the bottom or side when activated.

Whichever approach is chosen, the input panel must:
- Be accessible from every authenticated page without navigation.
- Not obscure critical page content when in its default (closed/compact) state.
- Expand to reveal a text input area when activated.

**Input panel contents (when expanded/active):**
- A text area for entering free-text input. Placeholder text using an i18n key suggesting what to enter (e.g., "Share new information, policy developments, or observations...").
- A submit button. Disabled when the text area is empty. Shows loading state while the classification + trigger request is in flight.
- The text area should be multi-line but compact (2–4 rows by default, expandable).

**Submission flow:**
1. User types input and clicks submit.
2. The panel calls `POST /api/cascade/trigger` with the input text.
3. The response includes: the classification result (which layer was targeted and why) and whether the cascade started or was queued.
4. On success: the panel shows a brief result summary — the classified layer name and what happened (cascade started or queued). The text area is cleared. After a short delay (or immediately), the panel collapses back to its compact state.
5. If the user wants to see the cascade progress, they navigate to the cascade viewer. The panel may offer a "View cascade" link after a successful submission.
6. On failure: the panel shows an error message. The text is preserved so the user can retry.

**Classification result display:**
- After successful submission, briefly display the classification result: "Classified as {layer name}: {explanation}". This helps users understand how the system interpreted their input.
- The display is temporary (auto-dismisses after a few seconds) or dismissed by the user.

**Interaction with the cascade store:**
- The input panel does not directly interact with the cascade store. The cascade trigger returns immediately, and the cascade's progress is communicated via WebSocket events that the cascade store handles independently. The navigation's cascade status indicator (step 009) and the cascade viewer (step 020) show progress.

### Input Panel Integration in App Shell

The input panel trigger element is added to the app shell layout so it appears on all authenticated pages:

- If implemented as a FAB: add a fixed-position button to the protected route wrapper or the navigation component. The button opens the input panel overlay/popover.
- If implemented as a sidebar element: add the input area to the navigation component.
- The input panel is only rendered when the user is authenticated (inside the protected route wrapper).
- The input panel's state (open/closed, input text) can be managed locally within the component — it does not need a zustand store since it has no cross-page state requirements.
