# 015: Cascade orchestrator

## Context

The cascade orchestrator is the central coordination engine of Policy Factory. When a layer update is triggered ‚Äî whether by user input, a manual layer refresh, the heartbeat system, or initial seeding ‚Äî the orchestrator drives the entire multi-layer update sequence: acquire a lock, run the generation agent for the starting layer, auto-commit the file changes, run all 6 critics in parallel, run the synthesis agent, record feedback memos, advance to the next layer above, and repeat until the topmost layer is processed.

Prior steps have established all the infrastructure the orchestrator needs:
- The agent framework (step 014) provides session wrappers for running Claude Code agents with per-role model selection, streaming output via EventEmitter, retry logic, and meditation filtering.
- The event system (step 012) provides typed events and the EventEmitter for broadcasting cascade progress to all WebSocket clients.
- The data layer (step 007) provides utilities for reading/writing markdown files and git commit operations.
- The SQLite store (step 004) provides the mixin composition pattern for adding domain-specific tables.

This step builds three things:

1. **Cascade mixin** ‚Äî SQLite storage for cascade run records, the single-writer lock, and the cascade queue. This is a new store mixin added to the composed `PolicyStore`.

2. **Agent run mixin** ‚Äî SQLite storage for individual agent invocations (who ran, when, how long, success/failure, model used, cost). Every agent session run is logged here for the activity feed and diagnostics.

3. **Cascade orchestrator** ‚Äî The async coordination function that drives the layer-by-layer update sequence with lock management, agent execution, git commits, progress event emission, error handling with pause/resume/cancel, and queue processing.

The critic runner and synthesis agent are separate concerns (step 016). This step's orchestrator calls into them as black-box functions ‚Äî the orchestrator doesn't know how critics work internally, it just calls "run critics for this layer" and "run synthesis for this layer" and handles their results. Step 016 will implement those functions. For this step, the orchestrator defines the interface it expects from the critic runner and synthesis runner (async functions that accept layer content and return results), and the acceptance criteria verify the orchestrator's own behavior using mock implementations of those functions.

Similarly, the feedback memo mixin is a separate concern (step 017). The orchestrator in this step records that feedback memos were produced (the generation agent's output may include them), but the mixin for storing and retrieving memos is built in step 017. The orchestrator defines the interface it needs ‚Äî "store these feedback memos" and "get pending feedback memos for this layer" ‚Äî and this step uses stub implementations that store memos in memory or no-op until step 017 provides the real mixin.

The cascade orchestrator is a pure async function, not a class ‚Äî following cc-runner's pipeline orchestrator pattern. It receives all its dependencies as arguments (store, emitter, agent runner functions, data utilities) rather than holding state as instance attributes. The cascade's mutable state (which layer, which step, error info) lives in the SQLite cascade record, not in-memory.

## Reuse Opportunities

- The **pipeline orchestrator structure** from cc-runner (async function iterating through steps, emitting events at each transition, checking for pause between steps, supporting resume from a position) maps directly to the cascade orchestrator. The key structural adaptation: cc-runner iterates through a flat list of step IDs; Policy Factory iterates through a list of layer slugs, and within each layer executes a fixed sequence of sub-steps (generation ‚Üí critics ‚Üí synthesis).
- The **SessionController state machine** from cc-runner is adapted for cascade control. The cascade controller tracks state (RUNNING/PAUSED/COMPLETED/FAILED/CANCELLED) with the same strict transitions and `asyncio.Event` flags for cooperative signaling. The additions are: CANCELLED as a terminal state, and tracking of the current position (layer slug + sub-step) for resume.
- The **module-level registry pattern** from cc-runner's deps.py (dict mapping run ID ‚Üí controller) is directly reusable for tracking active cascade controllers.
- The **store mixin pattern** from prior steps and cc-runner is used for both the cascade mixin and the agent run mixin.
- The **event emission pattern** from cc-runner (emit events at the orchestration level, not inside sub-components) is followed: the orchestrator emits lifecycle and progress events, while the session wrapper emits streaming text events.

## Deliverables

### Cascade Store Mixin

A store mixin at `src/policy_factory/store/cascade.py` added to the composed `PolicyStore`:

**Cascade runs table:**
- Tracks each cascade execution: unique ID, trigger source (user-input, layer-refresh, heartbeat, seed ‚Äî a string enum), starting layer slug, current layer slug (updated as the cascade progresses), current step within the current layer (generation, critics, synthesis ‚Äî a string enum), status (running, paused, completed, failed, cancelled), error message (populated on failure/pause), the layer slug where the error occurred (for resume targeting), created timestamp, and completed/failed timestamp.

**Cascade queue table:**
- Tracks queued cascade requests waiting for the lock: unique ID, trigger source, starting layer slug, queued timestamp, and an optional context field for additional data (e.g., the user input text that triggered the cascade). Queue entries are ordered by queued timestamp (FIFO).

**Lock management:**
- The single-writer cascade lock is implemented as a query against the cascade runs table: a lock is held if there exists a cascade run with status "running" or "paused". There is no separate lock table ‚Äî the cascade run record itself represents the lock.
- `acquire_lock(cascade_id)` ‚Äî checks whether any cascade is currently running or paused. If not, creates a new cascade run record with status "running" and returns success. If a lock is held, returns failure (the caller should queue the cascade instead).
- `release_lock(cascade_id)` ‚Äî updates the specified cascade run's status to its terminal state (completed, failed, or cancelled). The lock is implicitly released because no cascade is in "running" or "paused" state.
- `is_lock_held()` ‚Äî returns whether the lock is currently held and, if so, the cascade ID holding it.
- `get_active_cascade()` ‚Äî returns the cascade run record for the currently running or paused cascade, or None.

**Cascade CRUD:**
- `create_cascade(trigger_source, starting_layer, context)` ‚Äî creates a cascade run record. Does not acquire the lock (that's a separate call).
- `update_cascade_progress(cascade_id, current_layer, current_step)` ‚Äî updates the current position in the cascade. Called by the orchestrator at each state transition.
- `update_cascade_status(cascade_id, status, error_message, error_layer)` ‚Äî updates the cascade status. Used for transitions to paused, completed, failed, or cancelled.
- `get_cascade(cascade_id)` ‚Äî returns the full cascade run record by ID.
- `list_cascades(limit, offset)` ‚Äî returns recent cascade runs in reverse chronological order. Used for the activity feed.

**Queue management:**
- `enqueue_cascade(trigger_source, starting_layer, context)` ‚Äî adds a cascade request to the queue. Returns the queue entry ID and the queue position.
- `dequeue_cascade()` ‚Äî removes and returns the oldest entry from the queue (FIFO). Returns None if the queue is empty.
- `get_queue()` ‚Äî returns all queued cascade requests in order. Used for displaying queue status in the UI.
- `cancel_queued_cascade(queue_id)` ‚Äî removes a specific entry from the queue. Returns success/failure.
- `get_queue_depth()` ‚Äî returns the number of entries in the queue.

**Schema additions:**
- The cascade runs and cascade queue table definitions are added to the schema initialization.

### Cascade Controller

A module at `src/policy_factory/cascade/controller.py` adapted from cc-runner's `SessionController`:

**State machine:**
- States: RUNNING, PAUSED, COMPLETED, FAILED, CANCELLED.
- Valid transitions: RUNNING ‚Üí PAUSED (user pause), RUNNING ‚Üí COMPLETED (cascade finishes), RUNNING ‚Üí FAILED (unrecoverable error), PAUSED ‚Üí RUNNING (user resume), PAUSED ‚Üí CANCELLED (user cancel). No other transitions are valid.
- Invalid transition attempts are silently rejected (return False).

**Controller attributes:**
- Cascade ID, current state, the EventEmitter reference, the current position (layer slug + step), error information.
- An `asyncio.Event` flag for cooperative pause detection: the orchestrator checks this flag between steps and between sub-steps within a layer.

**Methods:**
- `pause()` ‚Äî async method transitioning RUNNING ‚Üí PAUSED. Sets the pause event flag. Emits a `CascadePaused` event.
- `resume()` ‚Äî async method transitioning PAUSED ‚Üí RUNNING. Clears the pause flag. Emits a `CascadeResumed` event.
- `cancel()` ‚Äî async method transitioning PAUSED ‚Üí CANCELLED. Emits a `CascadeCancelled` event.
- `complete()` ‚Äî marks the cascade as completed. Emits `CascadeCompleted`.
- `fail(error_message, error_layer, error_step)` ‚Äî marks the cascade as failed. Records the error position. Emits `CascadeFailed`.
- `is_pause_requested()` ‚Äî non-blocking check of the pause event flag. The orchestrator calls this between steps.
- Properties for reading the current state, cascade ID, current position, and error info.

### Cascade Orchestrator

The core orchestration function at `src/policy_factory/cascade/orchestrator.py`:

**Entry point ‚Äî trigger function:**

An async function that is the public API for starting a cascade. It accepts: the trigger source, the starting layer slug, optional context data (e.g., user input text), the store instance, the EventEmitter, and references to the critic runner and synthesis runner functions (these are passed as callables so the orchestrator doesn't directly import step 016's code ‚Äî keeping step dependencies clean).

The trigger function:
1. Checks whether the cascade lock is held. If held, enqueues the cascade request, emits a `CascadeQueued` event with the queue position, and returns the queue entry ID (not a cascade run ID ‚Äî the cascade hasn't started yet).
2. If the lock is free, creates a cascade run record, acquires the lock, creates a cascade controller, registers it in the deps module, emits `CascadeLockAcquired` and `CascadeStarted` events, and launches the orchestration loop as a background asyncio task.
3. Returns the cascade ID immediately ‚Äî the actual cascade runs asynchronously.

**Orchestration loop:**

An async function that drives the cascade through all layers from the starting layer to the topmost layer (policies). This function is not called directly ‚Äî it is launched as a background task by the trigger function.

For each layer from the starting layer upward through the hierarchy:
1. **Pause check** ‚Äî before starting each layer, check the controller's pause flag. If pause is requested, update the cascade status to "paused" in the store (recording the current position), and return. The cascade remains in the paused state with the lock held until a user resumes or cancels.
2. **Generation step:**
   a. Update cascade progress in the store (current layer, step = generation).
   b. Emit `LayerGenerationStarted` event.
   c. Gather context for the generation agent: content from layers below the current layer, pending feedback memos targeting this layer, any user input context if this is the first layer and the trigger was user input.
   d. Build the generation prompt using the prompt construction helper (step 014): meditation preamble + generator template for this layer + dynamic context variables.
   e. Create an agent session (step 014) with the generator model, working directory set to `data/`, and the EventEmitter for streaming.
   f. Record the agent run in the agent runs table (started).
   g. Run the session with the prompt. The session handles streaming, retry, and meditation filtering internally.
   h. Record the agent run completion (success/failure, cost, full output).
   i. If the agent failed (after internal retries), transition to the error handling path (described below).
   j. On success: auto-commit the data git repo with a descriptive message (e.g., "Generate values layer [cascade {id}]").
   k. Emit `LayerGenerationCompleted` event.
3. **Pause check** ‚Äî between generation and critics.
4. **Critics step:**
   a. Update cascade progress (step = critics).
   b. Call the critic runner function (provided as a callable ‚Äî implemented in step 016). Pass the current layer slug, the cascade ID, the store, the emitter, and the agent session factory. The critic runner handles launching all 6 critics in parallel, recording their runs, and collecting results.
   c. If the critic runner reports a failure (all critics failed, or a critical error), transition to error handling. If individual critics fail but others succeed, the cascade continues (partial results are acceptable).
   d. Store critic results (step 016's responsibility ‚Äî the orchestrator passes through whatever the critic runner returns).
5. **Pause check** ‚Äî between critics and synthesis.
6. **Synthesis step:**
   a. Update cascade progress (step = synthesis).
   b. Call the synthesis runner function (provided as a callable ‚Äî implemented in step 016). Pass the layer slug, the critic results, the cascade ID, the store, the emitter, and the agent session factory.
   c. If the synthesis runner fails, transition to error handling.
   d. Store synthesis results (step 016's responsibility).
7. After all three sub-steps for a layer complete, advance to the next layer above in the hierarchy and repeat.

**After the topmost layer is processed:**
1. Release the cascade lock by updating the cascade status to "completed".
2. Unregister the cascade controller from the deps registry.
3. Emit `CascadeLockReleased` and `CascadeCompleted` events.
4. Check the cascade queue. If non-empty, dequeue the next cascade request and start it (by calling the trigger function again, which will now successfully acquire the lock).

**Error handling:**

When an agent fails after the session wrapper's internal retries (3 retries within the session ‚Äî step 014), the orchestrator does not retry the agent again. Instead:
1. Updates the cascade status to "paused" with the error message and the position (layer slug + step) where the failure occurred.
2. Emits a `CascadePaused` event with error details.
3. The cascade lock remains held ‚Äî this prevents other cascades from running on potentially inconsistent data (the generation agent may have partially modified files).
4. The cascade waits for user intervention: either resume (retry the failed step) or cancel (release the lock and discard remaining steps).

**Resume behaviour:**

When a user resumes a paused cascade (via the API, which calls `controller.resume()`):
1. The orchestrator re-enters the orchestration loop from the recorded position (the layer and step where it paused).
2. It retries the failed step ‚Äî for example, if the synthesis step on strategic-objectives failed, the resume re-runs synthesis on strategic-objectives.
3. The orchestrator loop continues normally from that point onward.

The resume mechanism uses the cascade controller's state machine: when paused, the orchestrator function is waiting (not looping). Resuming clears the pause flag and the orchestrator resumes from its position. Implementation-wise, the orchestrator function could use an `asyncio.Event` to wait while paused, waking up when resume or cancel is triggered.

**Cancel behaviour:**

When a user cancels a paused cascade:
1. The cascade status is updated to "cancelled".
2. The cascade lock is released.
3. The cascade controller is unregistered.
4. `CascadeCancelled` and `CascadeLockReleased` events are emitted.
5. The cascade queue is checked ‚Äî if non-empty, the next queued cascade starts.

**Queue processing:**

After any cascade completes, fails terminally, or is cancelled, the orchestrator checks the queue. If there are pending requests, it dequeues the oldest one and triggers a new cascade with that request's parameters. This creates a chain: cascades process one at a time, in FIFO order.
