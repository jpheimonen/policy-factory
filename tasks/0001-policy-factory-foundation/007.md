# 007: Data layer — markdown and git

## Context

Policy Factory's layer content is stored as markdown files with YAML frontmatter in a `data/` directory that is its own separate git repository, independent from the application code. This is the core data model — the five-layer policy stack (values → situational awareness → strategic objectives → tactical objectives → policies) lives entirely in these files. The SQLite database (step 004) stores operational state; the markdown files are the source of truth for policy content.

This step builds the complete data layer: utilities for reading/writing markdown files with frontmatter, layer directory management, cross-layer reference resolution, git operations (init, commit, history), and the first-run initialization that creates the data directory with pre-seeded Finnish values.

Agents (built in step 014) will read and write these files directly using Claude Code's native file tools. The REST API (step 008) serves these files to the frontend. The cascade orchestrator (step 015) auto-commits changes after each agent run. All of these depend on the utilities built in this step.

Policy Factory's git operations are simpler (init, add, commit, log — no branching, no PRs, no remotes) but should follow the same subprocess approach.

**From Policy Factory (prior steps):**

## Reuse Opportunities

- The **frontmatter parsing pattern** from cc-runner's `frontmatter.py` is reusable — same `---` delimiter splitting, same `yaml.safe_load`. But return the raw dict instead of a typed dataclass, since Policy Factory's items have variable frontmatter schemas across layers.
- The **git subprocess pattern** from cc-runner's `git_helpers.py` is directly reusable — `subprocess.run` with capture, timeout, error handling. Policy Factory needs a simpler subset: init, add, commit, log. No branching, no remotes, no PRs.
- **`pyyaml`** is already in cc-runner's dependencies and should be added to Policy Factory's `pyproject.toml`.

## Deliverables

### Markdown File Utilities

A module at `src/policy_factory/data/markdown.py` providing functions for working with markdown files that have YAML frontmatter:

**Reading a file:**
- Accepts a file path, returns a tuple of (frontmatter dict, body string).
- Parses the YAML frontmatter block (delimited by `---` at the start and end) into a Python dict.
- The body is everything after the closing `---` delimiter.
- If the file has no YAML frontmatter (no opening `---`), returns an empty dict for frontmatter and the entire content as the body.
- If the YAML is malformed, returns an empty dict for frontmatter and the full content as body (graceful degradation, not an error).
- If the file doesn't exist, raises an appropriate error.

**Writing a file:**
- Accepts a file path, frontmatter dict, and body string.
- Serializes the frontmatter as a YAML block between `---` delimiters, followed by the body.
- If the frontmatter dict is empty, writes just the body without YAML delimiters.
- Creates parent directories if they don't exist.

**Round-trip integrity:**
- Writing then reading a file must produce identical frontmatter and body. YAML serialization must be deterministic (consistent key ordering, consistent formatting).

### Layer Directory Utilities

A module at `src/policy_factory/data/layers.py` providing functions for working with layer directories:

**Layer definitions:**
- The five layers in hierarchical order (bottom to top): values, situational-awareness, strategic-objectives, tactical-objectives, policies.
- Each layer has a slug (used in directory names and URLs), a display name, and its position in the hierarchy.
- A function to validate a layer slug and return the corresponding layer info. Returns None or raises for invalid slugs.
- A function to get the filesystem path for a layer directory given the data root path and layer slug.

**Listing items in a layer:**
- Accepts the data root path and a layer slug.
- Scans the layer directory for `.md` files, excluding `README.md`.
- For each file, reads the YAML frontmatter and extracts summary metadata (title, status, last modified date, last modified by).
- Returns a list of items with their filename and summary metadata, sorted by title or filename.

**Reading a single item:**
- Accepts the data root path, layer slug, and filename.
- Returns the full frontmatter dict and markdown body for the specified item.
- Returns an appropriate error if the file doesn't exist or the slug is invalid.

**Writing an item:**
- Accepts the data root path, layer slug, filename, frontmatter dict, and body.
- Writes the file to the correct layer directory.
- Automatically sets or updates the `last_modified` field in the frontmatter to the current timestamp.

**Deleting an item:**
- Accepts the data root path, layer slug, and filename.
- Removes the file from the layer directory.
- Idempotent — deleting a non-existent file does not error.

**Narrative summary (README.md):**
- A function to read the README.md in a layer directory, returning its content as a string.
- A function to write/update the README.md in a layer directory.
- Returns empty string if README.md doesn't exist yet.

### Cross-Layer Reference Resolution

A function (or set of functions) in the layer utilities that resolves cross-layer references:

**How references work:**
- Each item's frontmatter can contain a `references` field — a list of filenames from other layers that this item relates to. For example, a strategic objective might reference value items with `references: ["national-security.md", "economic-prosperity.md"]`.
- References are expressed as filenames (not full paths). The layer they belong to can be inferred by scanning all layer directories for the filename, or the reference format could include the layer slug (e.g., `values/national-security.md`). The implementer should choose the clearest approach.

**Resolution:**
- Given an item (layer slug + filename), return two lists:
  1. **Forward references**: items that this item explicitly references in its frontmatter.
  2. **Backward references**: items in other layers whose frontmatter references this item.
- Forward references are a direct lookup from the item's frontmatter.
- Backward references require scanning all items across all layers to find those that reference the given item. This is acceptable for the current scale (tens to low hundreds of files total).
- Each reference in the result includes: the layer slug, filename, and item title (from frontmatter).
- Returns empty lists when no references exist in either direction.

### Git Operations

A module at `src/policy_factory/data/git.py` providing git operations for the data directory:

**Auto-initialization:**
- A function that checks if the data directory exists and contains a git repo. If not, creates the directory, runs `git init`, creates the five layer subdirectories, writes initial README.md files (empty or with a placeholder title) in each layer directory, writes the pre-seeded values files, and creates an initial commit.
- If the data directory and git repo already exist, does nothing (idempotent).
- This function is called during FastAPI lifespan startup.

**Committing changes:**
- A function that stages all changes in the data directory (`git add .`) and commits with a provided descriptive message.
- The commit author should be identifiable as the system (e.g., "Policy Factory" or configurable via environment variable).
- If there are no changes to commit, the function succeeds silently (no error for empty commits).
- Used by the cascade orchestrator (step 015) after each agent run.

**Retrieving history:**
- A function that returns recent git commits affecting a specific layer directory.
- Each commit entry includes: commit hash, timestamp (ISO 8601), commit message, and the author name.
- Accepts a limit parameter (default: 20 most recent commits).
- Uses `git log` with a path filter to show only commits that touched files in the specified layer directory.
- Returns an empty list if no commits exist for that directory.

**Subprocess execution:**
- Git commands should be executed via `subprocess.run` with capture, timeout, and error handling — following the cc-runner pattern.
- Operations can be synchronous (called from async code via `run_in_executor`) since git operations on a local repo are fast.

### Data Directory Configuration

- The data directory path defaults to `data/` relative to the current working directory.
- The path is configurable via an environment variable (`POLICY_FACTORY_DATA_DIR`).
- Add the data directory path to the dependency injection module so it's accessible to routers and other components.
- The `.gitignore` file (from step 001) already includes `data/` so the data repo is excluded from the application repo.

### Pre-Seeded Finnish Values

On first-run initialization, the values layer is populated with approximately 10 pre-seeded markdown files representing foundational Finnish national values and interests. Each file has YAML frontmatter (title, status: "active", created date, created by: "system") and a markdown body with a paragraph or two describing the value.

The pre-seeded values should include (at minimum):
- National security and defence
- Economic prosperity and competitiveness
- EU membership and European solidarity
- Arctic sovereignty and interests
- Democratic institutions and rule of law
- Social welfare and equality
- Technological competitiveness and innovation
- Cultural identity and heritage
- Environmental sustainability and climate action
- Education and research excellence

Each value file should have a descriptive filename (e.g., `national-security.md`, `economic-prosperity.md`) and a concise but substantive body describing why this value matters for Finland. The content should be factual and non-partisan — these are foundational values that transcend party lines.

An initial commit is created after writing all seed files with a descriptive commit message.
