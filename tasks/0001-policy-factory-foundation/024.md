# 024: Production build and Docker

## Context

All backend and frontend features are now complete — the application has authentication, the five-layer data model, cascade orchestration, the critic system, idea evaluation, the tiered heartbeat, and every frontend page. This step packages everything for production deployment.

In development, the Vite dev server and FastAPI backend run as separate processes with Vite proxying API and WebSocket requests. In production, the frontend is compiled into static assets that the FastAPI server serves directly — a single process handles both the API and the UI. The compiled frontend is baked into the Python package so the entire application can be distributed as a single installable artifact.

Step 001 established the project scaffold including the `make dev` and `make build` targets, the SPA fallback route in the FastAPI app factory, the hatchling build configuration with `artifacts = ["src/policy_factory/static/dist/"]`, and the Vite config outputting to `../src/policy_factory/static/dist/`. This step extends the Makefile with `make run`, `make docker-build`, and `make docker-run` targets, creates the Dockerfile, and ensures the full production serving pipeline works end-to-end.

Policy Factory has a unique deployment requirement compared to cc-runner: it needs the `claude` CLI tool available at runtime because `claude-agent-sdk` shells out to it for running Claude Code sessions. The Docker image must include both the Python application and the claude CLI. Additionally, the `data/` directory (which is its own git repo) and the SQLite database file must persist across container restarts via volume mounts.

## Deliverables

### Version Stamping

Before the frontend build, generate a version.json file in the frontend's public directory containing the current git commit hash (short form) and the build timestamp. The Vite build copies public directory files into the output, so version.json ends up in the dist directory. This allows the UI to display build information (useful for debugging which version is deployed).

### Makefile Extensions

Add the following targets to the existing Makefile:

**`make run`** — Production server target. Runs `make build` first to ensure the frontend is compiled, then starts the server via the CLI entry point (`policy-factory server`). The server binds to the default host and port (127.0.0.1:8765) unless overridden via environment variables or CLI arguments passed through. This target is for running locally in production mode (not in Docker).

**`make docker-build`** — Builds the Docker image. Tags the image with the project name (e.g., `policy-factory`). Passes through any build arguments needed (e.g., for configuring the base image or claude CLI version).

**`make docker-run`** — Runs the Docker container with appropriate configuration:
- Maps the container port to a host port (default 8765)
- Mounts a host directory for the `data/` volume (the separate git repo where layer markdown files live) — defaults to `./data` on the host
- Mounts a host directory for the SQLite database file — defaults to a local directory on the host
- Passes through the `ANTHROPIC_API_KEY` environment variable (required for claude-agent-sdk agents)
- Passes through any additional environment variables from a `.env` file if one exists
- Runs in detached mode by default, with an option to run in foreground for debugging

**`make docker-stop`** — Stops the running Docker container.

### Dockerfile

A Dockerfile at the project root that produces a self-contained image capable of running the complete Policy Factory application. The image must include:

**Python runtime** — A recent Python version (3.12 or later) as the base. The official Python slim image is appropriate.

**Node.js/bun** — Required only during the build phase to compile the frontend. If using a multi-stage build, bun is only needed in the builder stage and does not appear in the final runtime image.

**The `claude` CLI** — The claude-agent-sdk package requires the `claude` CLI tool to be available on the system PATH at runtime. The Dockerfile must install this tool. The installation method should follow Anthropic's recommended approach (npm global install or direct binary download). This is a runtime dependency, not just a build dependency.

**The policy_factory package** — Installed via `uv pip install .` (or equivalent) with the compiled frontend assets already baked in. The package includes everything: FastAPI server, all backend logic, agent framework, prompt templates, and static frontend assets.

**Git** — Required at runtime because the data layer auto-initializes a git repo in the `data/` directory and performs auto-commits after agent modifications. Git must be available on the system PATH.

**Build approach** — A multi-stage build is preferred to keep the final image small. The first stage installs bun, compiles the frontend, and builds the Python wheel. The second stage installs only runtime dependencies: Python, git, the claude CLI, and the built wheel. This avoids shipping bun, node_modules, and build tooling in the production image.

**Volume mounts** — The Dockerfile should document (via VOLUME instructions or comments) two expected volume mounts:
1. The `data/` directory — mounted at the application's expected data path. This is where layer markdown files, the data git repo, and pre-seeded values live. Must persist across container restarts.
2. The SQLite database directory — mounted at the path configured by the database environment variable. Contains the SQLite database file with users, ideas, cascade state, heartbeat logs, and agent run history.

**Environment variables** — The Dockerfile should document (via ENV instructions with defaults or comments) the key environment variables:
- `ANTHROPIC_API_KEY` — Required for claude-agent-sdk to function. No default.
- `POLICY_FACTORY_DB_PATH` — Path to the SQLite database file. Defaults to `~/.policy-factory/store.db`.
- `POLICY_FACTORY_DATA_DIR` — Path to the data directory. Defaults to `./data` relative to the working directory.
- `POLICY_FACTORY_HOST` — Host to bind uvicorn to. Defaults to `0.0.0.0` in Docker (all interfaces).
- `POLICY_FACTORY_PORT` — Port to bind uvicorn to. Defaults to `8765`.
- Heartbeat configuration: `POLICY_FACTORY_HEARTBEAT_INTERVAL` (default 4 hours), `POLICY_FACTORY_HEARTBEAT_ENABLED` (default true).
- Model configuration variables for per-agent model selection (documented in step 014).

**Entry point** — The container's entry point runs the CLI command `policy-factory server` with appropriate host and port arguments. The host defaults to `0.0.0.0` (all interfaces) so the container's port can be mapped to the host.

**Working directory** — Set to a directory within the container where the `data/` volume mount will be accessible. The working directory matters because agents operate on files relative to `data/`.

**Health check** — The Dockerfile includes a HEALTHCHECK instruction that curls the `/api/health/check` endpoint to let Docker and orchestrators know whether the application is responsive.

### .dockerignore

A `.dockerignore` file at the project root to keep the Docker build context small. Excludes: `node_modules/`, `ui/node_modules/`, `.git/`, `data/`, `__pycache__/`, `*.pyc`, `.env` files (secrets should be passed via environment variables, not baked into the image), `src/policy_factory/static/dist/` (rebuilt inside the Docker build), `.venv/`, IDE configuration files, and any SQLite database files.

### Production Serving Verification

No changes to the FastAPI app factory are needed — step 001 already established the SPA fallback and static file serving. However, this step verifies the full production pipeline works end-to-end:

- The `make build` output in `src/policy_factory/static/dist/` contains all compiled assets (index.html, JS/CSS bundles in assets/, favicon, version.json)
- The FastAPI server serves the compiled index.html at the root path
- The FastAPI server serves JS/CSS bundles from the `/assets` path
- Client-side routing works: navigating directly to `/cascade`, `/ideas`, `/admin`, or any other frontend route returns index.html (which React Router then handles)
- API routes (`/api/*`) are not intercepted by the SPA fallback
- WebSocket connections (`/ws`) are not intercepted by the SPA fallback
