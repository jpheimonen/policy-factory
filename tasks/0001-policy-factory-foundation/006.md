# 006: Frontend auth flow

## Context

The backend auth system is complete (steps 004–005). Now the frontend needs to support authentication: login and registration pages, a zustand auth store that persists the JWT to localStorage, an authenticated HTTP client that includes the token in all API requests, a protected route wrapper that redirects unauthenticated users to login, and the full app shell provider hierarchy.

This step is the first step that creates real, user-facing pages in the frontend. The login and registration pages are the first things users see. They must look polished — clean forms with the Inter font, validation feedback, error messages — using the design system atoms from step 002 and translation keys from step 003.

The registration page has a special first-user flow: when no users exist, the registration page is shown and anyone can register (becoming admin). After the first user exists, the registration page is no longer publicly accessible — the admin creates accounts from the admin panel (step 023). The frontend needs to detect whether users exist (via a lightweight API check) to decide which page to show on first visit.

## Deliverables

### Auth Store

A zustand store at `ui/src/stores/authStore.ts` managing authentication state:

**State:**
- The JWT access token (string or null)
- The current user's info: id, email, role (or null if not logged in)
- A loading flag for async operations (login, register, refresh)
- An error state for displaying auth errors in the UI

**Actions:**
- **Login**: accepts email and password, calls `POST /api/auth/login`, on success stores the JWT and user info in the store and persists the JWT to localStorage. On failure, sets the error state with the error message.
- **Register**: accepts email and password, calls `POST /api/auth/register` (without auth header for first-user flow). On success, stores the JWT and user info (first user gets JWT back immediately). On failure, sets the error state.
- **Logout**: clears the JWT and user info from the store and removes the JWT from localStorage.
- **Refresh**: calls `POST /api/auth/refresh` with the current token, updates the stored JWT with the new one. Called proactively when the token approaches expiry.
- **Initialize**: on app load, checks localStorage for a stored JWT. If found, decodes it to extract user info (without server validation — the server will reject expired tokens on first API call). Sets the store state accordingly.
- **Clear error**: resets the error state (used when the user starts typing after a failed login).

**Persistence:**
- The JWT is persisted to localStorage under a consistent key.
- On page reload, the stored JWT is restored automatically.
- On logout, the JWT is removed from localStorage.

**Computed values:**
- `isAuthenticated` — true if a JWT is present in the store.
- `isAdmin` — true if the current user has the "admin" role.

### Authenticated HTTP Client

A utility module (or wrapper function) that provides fetch/request capabilities with automatic auth header injection:

- All API requests from the frontend use this client instead of raw `fetch`.
- If a JWT exists in the auth store, the client automatically adds `Authorization: Bearer <token>` to every request.
- If any API response returns 401, the client attempts a token refresh. If the refresh succeeds, the original request is retried with the new token. If the refresh also fails (e.g., token fully expired), the client triggers logout and redirects to the login page.
- The client handles JSON serialization/deserialization for request bodies and response parsing.
- Error responses are parsed and returned in a consistent format for the UI to display.

### Registration Check

A mechanism to determine whether users exist in the system, so the frontend knows whether to show the registration page or the login page on first visit:

- The simplest approach is a lightweight API endpoint that returns whether any users exist. This could be a new endpoint (e.g., `GET /api/auth/status` returning `{ hasUsers: true/false }`) or the health endpoint could be extended with this info. The implementer should choose the cleanest approach.
- Alternatively, the frontend can attempt to call `POST /api/auth/register` and check if it requires auth — but a dedicated check is cleaner.
- This check runs once on app load if the user is not already logged in (has no stored JWT).

### Login Page

A page component at `ui/src/pages/LoginPage.tsx` (route: `/login`):

- A centered card on the page with the app name/logo at the top.
- Email and password input fields using the Input atom component.
- A login button using the Button atom (primary variant).
- Client-side validation: both fields required, email must contain @, password must not be empty. Validation errors shown inline below the relevant field.
- On submit, calls the auth store's login action. Shows a loading state on the button while the request is in flight.
- On success, redirects to the home page (`/`).
- On failure (wrong credentials), displays an error message using the Text atom (error styling). The error clears when the user starts typing again.
- All visible text uses i18n translation keys.
- The page should look professional — centered vertically and horizontally on the page, generous whitespace, subtle card border, the Notion/Linear aesthetic.

### Registration Page

A page component at `ui/src/pages/RegisterPage.tsx` (route: `/register`):

- Only accessible when no users exist in the system. If users already exist and an unauthenticated visitor navigates to `/register`, they are redirected to `/login`.
- Same visual design as the login page — centered card with the app name.
- Email and password fields with a "Create Account" button.
- Same client-side validation as the login page.
- On submit, calls the auth store's register action. On success, the user is logged in (JWT stored) and redirected to the home page.
- On failure, displays the error message.
- A subtitle or note indicating that this is the first account and will have admin privileges.
- All visible text uses i18n translation keys.

### Protected Route Wrapper

A component that wraps protected routes (everything except login and register):

- Checks the auth store for a valid token. If no token is present, redirects to `/login`.
- If a token is present, renders the child route.
- This is used in App.tsx to wrap all routes except `/login` and `/register`.
- When the user logs out from any page, they are redirected to `/login`.

### App Shell Update

Update `App.tsx` to include the complete provider and routing structure:

**Provider hierarchy** (outermost to innermost):
1. ThemeProvider (from step 002) — provides the resolved theme
2. i18n Provider (from step 003) — provides translations
3. GlobalStyles (from step 002) — applies global CSS
4. BrowserRouter — enables client-side routing

**Route structure:**
- `/login` — LoginPage (public)
- `/register` — RegisterPage (public, with redirect-if-users-exist guard)
- `/` — placeholder home page (protected) — the actual stack overview comes in step 009
- All other routes defined in the architecture (`/layers/:slug`, `/ideas`, `/cascade`, `/activity`, `/history/:slug`, `/admin`) as placeholder protected routes that render a "Coming soon" or the page name — enough to verify the routing works. Real pages come in subsequent steps.
- `*` catch-all — redirects to `/`

**Auth initialization:**
- On app mount, the auth store initializes (restores JWT from localStorage).
- If no JWT is stored, the registration check runs to determine if the app should show login or registration.
- The app renders a brief loading state while the auth check completes (preventing flash of login page for already-authenticated users).

### Navigation Component

A basic navigation component (header or sidebar) that:
- Shows the app name.
- Shows the current user's email.
- Includes a theme toggle (using the theme store from step 002).
- Includes a logout button.
- Shows navigation links to the main pages (home, ideas, cascade, activity). The admin link is only visible to admin users.
- Is rendered inside the protected route wrapper (not visible on login/register pages).
- All labels use i18n translation keys.

This is a basic version — it will be refined in later steps as pages are built. The important thing is that the nav structure exists and the auth-related controls (user display, logout, admin-only visibility) work correctly.
