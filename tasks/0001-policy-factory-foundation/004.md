# 004: SQLite store foundation

## Context

Policy Factory needs persistent storage for users, ideas, scores, cascade state, feedback memos, heartbeat logs, and agent run history. The layer content itself lives in markdown files in the data git repo — but all operational state lives in SQLite. This step establishes the database foundation: schema initialization, WAL mode, the mixin composition pattern, and the auth mixin (the first domain-specific mixin, needed for step 005's auth backend).

This step intentionally creates only the base store infrastructure and the auth mixin. The remaining mixins (idea, score, cascade, feedback memo, heartbeat, agent run) are created in their respective feature steps (015–019) where they are needed. This keeps each step focused and independently testable.

The cc-runner store pattern is battle-tested and directly applicable: raw sqlite3 with `row_factory = sqlite3.Row`, schema defined as SQL strings with `CREATE TABLE IF NOT EXISTS`, migrations via `ALTER TABLE` wrapped in try/except, and mixin classes that declare `conn: sqlite3.Connection` as a type hint and access the shared connection. The composed store class inherits from the base and all mixins.

## Related Code

Existing files/patterns to reference or extend (all in cc-runner at `/home/jp/projects/modernpath/cc-runner`):

The `init_db(db_path)` function connects to SQLite with `check_same_thread=False`, sets `row_factory = sqlite3.Row`, executes the schema, enables WAL mode, and runs migrations (each migration is a `SELECT column FROM table LIMIT 1` wrapped in try/except — if it fails, the column doesn't exist and the ALTER TABLE runs). Also has `get_default_db_path()` returning `~/.mp-sw-factory/runs.db` with directory auto-creation. Policy Factory should follow the same pattern with `~/.policy-factory/store.db`.

Has dataclass models (`Run`, `StoredEvent`) and methods that do raw SQL via `self.conn.execute()` with parameter binding, manual `self.conn.commit()` calls, and `_row_to_X()` methods that convert `sqlite3.Row` to dataclasses. Timestamps stored as ISO 8601 strings, converted to `datetime` on read.
- `src/cc_runner/store/__init__.py` — The composed store class using Python multiple inheritance: `RunStore(BaseRunStore, SparStoreMixin, SparArtifactStoreMixin, ApiKeyStoreMixin)`. The `__init__` just calls `super().__init__(db_path)`. This is the only class instantiated by consumers — individual mixins are never used directly.
- `src/cc_runner/store/spar.py` — `SparStoreMixin` class demonstrating the mixin pattern: declares `conn: sqlite3.Connection` as a type hint (the connection is provided by `BaseRunStore` at runtime), all methods use `self.conn.execute()`. Methods return plain dicts (not dataclasses) by converting `sqlite3.Row` inline. Uses `uuid.uuid4()` for IDs and `datetime.now(timezone.utc).isoformat()` for timestamps.
- `src/cc_runner/store/api_keys.py` — `ApiKeyStoreMixin` — a simpler mixin showing the minimal pattern: type hint for `conn`, CRUD methods, idempotent operations (deleting non-existent key doesn't error), upsert via `INSERT OR REPLACE`.
- `src/cc_runner/main.py` — Shows how the store is initialized in `server_command()`: `store = RunStore(get_default_db_path())`, then passed to `create_app(store=store)`.
- `src/cc_runner/server/deps.py` — Shows how the store singleton is made available to routers via `get_store()` dependency.

## Reuse Opportunities

- The **`init_db()` function pattern** is directly reusable: connect → set row_factory → executescript schema → enable WAL → run migrations → return connection. Policy Factory has different tables but the same initialization approach.
- The **`get_default_db_path()` pattern** is reusable: check for `~/.policy-factory/` directory, create if absent, return path to `store.db`. No legacy fallback needed (new project).
- The **`BaseRunStore.__init__` pattern** is reusable: accept `db_path`, call `init_db()`, store `self.conn`.
- The **mixin class pattern** is directly reusable: declare `conn: sqlite3.Connection` type hint, write methods that do SQL via `self.conn.execute()` with manual commits.
- The **composed store class** pattern is directly reusable: `PolicyStore(BaseStore, AuthMixin, ...)` with a single `__init__` that calls `super().__init__(db_path)`.
- The **timestamp handling** pattern (store as ISO 8601 strings, parse with `datetime.fromisoformat()` on read) should be followed for consistency.
- The **`_row_to_X()` conversion pattern** from base.py should be followed — convert `sqlite3.Row` to dataclasses for type safety on read operations.
- The **ID generation** pattern (`str(uuid.uuid4())`) should be used for all primary keys.

## Deliverables

### Schema Module

A schema module at `src/policy_factory/store/schema.py` containing:

- A schema string defining the `users` table. The users table tracks: unique ID (UUID string, primary key), email (unique, not null), hashed password (not null), role (text: "admin" or "user", not null, default "user"), and created_at (ISO 8601 timestamp, not null).
- Appropriate indexes on the users table (at minimum, a unique index on email for fast lookup during login).
- An `init_db(db_path)` function that: opens a sqlite3 connection with `check_same_thread=False`, sets `row_factory = sqlite3.Row`, executes the schema, enables WAL journal mode, runs any migrations, and returns the connection.
- A `get_default_db_path()` function that returns `~/.policy-factory/store.db`, creating the `~/.policy-factory/` directory if it doesn't exist. The database path should also be overridable via an environment variable (e.g., `POLICY_FACTORY_DB_PATH`) so Docker deployments can point it at a mounted volume.
- The schema string should only include the users table for now. Other tables (ideas, scores, cascade_runs, feedback_memos, heartbeat_runs, agent_runs) will be added by their respective feature steps, each extending the schema string or adding their own table creation to `init_db`.

### Base Store

A base store class at `src/policy_factory/store/base.py`:

- `BaseStore` class that takes `db_path: Path` in `__init__`, calls `init_db(db_path)`, and stores the connection as `self.conn`.
- No domain-specific methods in the base — it just provides the connection and initialization. The base class exists to be the root of the mixin composition hierarchy.

### Auth Mixin

An auth store mixin at `src/policy_factory/store/auth.py`:

- `AuthStoreMixin` class with `conn: sqlite3.Connection` type hint.
- **Create user**: accepts email, already-hashed password, and role. Generates a UUID for the ID. Stores the record with a created_at timestamp. Returns the user ID. Note: the mixin stores the pre-hashed password — the hashing itself happens in the auth service layer (step 005), not in the store mixin. The store never sees plaintext passwords.
- **Get user by email**: looks up a user by email (case-insensitive comparison), returns a user dict/dataclass with all fields (including hashed password, needed for login verification), or None if not found.
- **Get user by ID**: looks up a user by UUID, returns user data or None.
- **List users**: returns all users ordered by created_at, with all fields except hashed password (the password hash should be excluded from list results to avoid accidental exposure).
- **Delete user**: removes a user by ID. Idempotent — deleting a non-existent user does not error.
- **Count users**: returns the total number of users. Used by the registration logic to determine if this is the first user (who gets admin role automatically).
- **Check email exists**: a lightweight check for whether an email is already registered. Used to reject duplicate registrations before attempting an insert.

### Composed Store

A composed store class at `src/policy_factory/store/__init__.py`:

- `PolicyStore(BaseStore, AuthStoreMixin)` — initially composed of just the base and auth mixin. Additional mixins will be added in later steps as new feature domains are built.
- Exports: `PolicyStore`, `init_db`, `get_default_db_path`, and any user-related dataclass models.
- This is the only store class that consumers import and instantiate.
