# 008: Layer REST API and history

## Context

Step 007 built the data layer utilities — reading/writing markdown files, listing items, cross-layer references, and git operations. This step exposes all of that functionality through REST API endpoints so the frontend can read and display layer data, and users can edit items through the UI.

This is the first "content" API — everything before this was infrastructure (auth, store, data utilities). After this step, the frontend can fetch and display the five-layer policy stack, individual items, narrative summaries, cross-layer references, and version history. Steps 009–011 build the frontend pages that consume these endpoints.

The layers router provides full CRUD for layer items plus read-only access to layer metadata, narrative summaries, and cross-layer references. The history router provides git commit history per layer. All endpoints require JWT authentication (using the `get_current_user` dependency from step 005) — except the health endpoint which remains unauthenticated.

Write operations (create, update, delete) through the API also trigger a git commit in the data repo, so that manual edits through the UI are versioned alongside agent-generated changes.

## Reuse Opportunities

- The **layer utilities from step 007** do all the heavy lifting — the routers are thin HTTP wrappers that validate inputs, call the utility functions, and format responses. No duplicated file I/O logic in the routers.
- The **auth dependency** (`get_current_user`) from step 005 is applied uniformly to all endpoints — no additional auth logic needed.
- The **router pattern** from cc-runner (APIRouter, prefix, tags, Depends, HTTPException, Pydantic request models) is directly reusable.
- The **path traversal prevention** pattern from cc-runner's tasks router should be applied to layer slug and filename path parameters — slugs are validated against the known layer list, and filenames are sanitized to prevent directory traversal.

## Deliverables

### Layers Router

A router at `src/policy_factory/server/routers/layers.py` with prefix `/api/layers` and tag `layers`. All endpoints require JWT authentication via the `get_current_user` dependency.

**GET `/api/layers/`** — List all layers with metadata.
- Returns all 5 layers in hierarchical order (bottom to top: values → policies).
- For each layer: slug, display name, item count (number of `.md` files excluding README.md), last updated timestamp (from the most recently modified item's frontmatter, or from the most recent git commit affecting that directory), narrative summary preview (first ~200 characters of README.md), and pending feedback memo count (hardcoded to 0 for now — the feedback memo mixin comes in step 017).
- This is a read-heavy endpoint called every time the home page loads — it should be efficient. Reading frontmatter from every file on every request is acceptable at the expected scale but the implementer may add lightweight caching if desired.

**GET `/api/layers/:slug/items`** — List all items in a layer.
- Validates the layer slug against the known list. Returns 404 for invalid slugs.
- Returns all items in the layer with summary metadata from frontmatter: filename, title, status, last modified date, last modified by.
- Sorted by title alphabetically.

**GET `/api/layers/:slug/items/:filename`** — Get a single item.
- Validates the slug and filename. Prevents path traversal by checking that the filename doesn't contain path separators or `..`.
- Returns the full frontmatter (as a JSON object) and the markdown body (as a string).
- Returns 404 if the file doesn't exist.

**PUT `/api/layers/:slug/items/:filename`** — Update an existing item.
- Accepts a request body with the updated frontmatter (JSON object) and body (string).
- Validates the slug and filename.
- Returns 404 if the file doesn't exist.
- Writes the updated content using the layer utilities.
- The `last_modified` and `last_modified_by` frontmatter fields are set automatically: timestamp is the current time, and modified-by is the authenticated user's email.
- After writing, triggers a git commit with a descriptive message (e.g., "Update {filename} in {layer name}").
- Returns the updated item (frontmatter + body).

**POST `/api/layers/:slug/items`** — Create a new item.
- Accepts a request body with the filename, frontmatter (JSON object), and body (string).
- Validates the slug. Validates the filename: must end in `.md`, must not be `README.md`, must not contain path separators or `..`, must not already exist in the layer directory.
- Returns 409 (conflict) if the filename already exists.
- Sets `created_at`, `last_modified`, and `last_modified_by` frontmatter fields automatically.
- After writing, triggers a git commit with a descriptive message.
- Returns the created item with status 201.

**DELETE `/api/layers/:slug/items/:filename`** — Delete an item.
- Validates the slug and filename.
- Returns 404 if the file doesn't exist.
- Deletes the file using the layer utilities.
- After deleting, triggers a git commit with a descriptive message (e.g., "Delete {filename} from {layer name}").
- Returns 204 (no content).

**GET `/api/layers/:slug/summary`** — Get the narrative summary.
- Validates the slug.
- Returns the README.md content for the layer as a string.
- Returns an empty string if README.md doesn't exist yet (not a 404 — the file may not exist until the first agent generates it).

**GET `/api/layers/:slug/items/:filename/references`** — Get cross-layer references.
- Validates the slug and filename.
- Returns 404 if the item doesn't exist.
- Calls the cross-layer reference resolution utility from step 007.
- Returns two lists: forward references (items this item references) and backward references (items that reference this item). Each reference entry includes: layer slug, filename, and item title.

### History Router

A router at `src/policy_factory/server/routers/history.py` with prefix `/api/history` and tag `history`. All endpoints require JWT authentication.

**GET `/api/history/:slug`** — Get recent git history for a layer.
- Validates the layer slug. Returns 404 for invalid slugs.
- Calls the git history retrieval utility from step 007, filtered to the specified layer directory.
- Accepts an optional `limit` query parameter (default: 20, max: 100).
- Returns a list of commit entries, each with: commit hash (short), timestamp (ISO 8601), commit message, and author name.
- Returns an empty list if no commits exist for that layer.

### Input Validation and Security

All endpoints must validate path parameters to prevent path traversal attacks:

- **Layer slugs** are validated against the known list of 5 layer slugs. Any slug not in the list returns 404.
- **Filenames** are validated: must not contain `/`, `\`, or `..`. Must end in `.md`. Must not be `README.md` (for item endpoints — the summary endpoint handles README.md separately).
- Request body validation uses Pydantic models for type safety on the frontmatter and body fields.

### App Factory and Router Registration

Update `create_app()` in `app.py` to include the layers router and history router.

Update the router barrel export in `__init__.py` to include `layers_router` and `history_router`.

### Git Commit on Write Operations

Write operations (create, update, delete) through the API must commit changes to the data git repo. The commit message should be descriptive:
- Create: "Create {filename} in {layer display name}"
- Update: "Update {filename} in {layer display name}"
- Delete: "Delete {filename} from {layer display name}"

The commit author should include the authenticated user's email so the activity is attributable (e.g., the git author name could be "User ({email})" or just the email).

If the git commit fails (e.g., data directory not initialized), the write operation itself should still succeed — the file change is persisted, but the commit failure is logged as a warning. The user should not see an error for a git commit failure since the actual write succeeded.
