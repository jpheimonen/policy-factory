# 022: Frontend activity feed and version history

## Context

Policy Factory generates a rich stream of events as it operates: cascades start and progress through layers, critics produce assessments, ideas are submitted and evaluated, heartbeats scan the news, users log in and edit items. All of these events are captured in the backend event system (step 012), persisted to SQLite, and broadcast via WebSocket to all connected clients. The activity store (step 013) accumulates these events in real-time. But there is no page to display them — the events are flowing through the system but invisible to users.

This step builds two related pages:

1. **Activity feed** (route `/activity`) — a chronological stream of all system events with filtering by event category and layer. New events appear in real-time without page refresh. The page also loads historical events from the REST endpoint on mount, so users see past activity even if they weren't connected when it happened.

2. **Version history** (route `/history/:layerSlug`) — a per-layer view showing recent git commits affecting that layer's data directory. Each entry shows the commit date, message (which describes what changed), and what triggered the change (agent, user, heartbeat). This is a simpler, git-focused view complementing the event-based activity feed.

Both pages are pure frontend efforts — all backend infrastructure is already in place:
- The activity REST endpoint (`GET /api/activity/`, step 012) returns persisted events with filtering and pagination.
- The history REST endpoint (`GET /api/history/:layerSlug`, step 008) returns git commits for a layer directory.
- The activity store (step 013) receives all events in real-time via WebSocket and holds a capped in-memory list.
- The central event dispatcher (step 013) pushes every event to the activity store.

The activity feed is the system's audit log — politicians and policy professionals can see exactly what happened, when, and why. The version history is the content audit trail — showing how each layer's files have evolved over time.

Both page routes (`/activity` and `/history/:layerSlug`) were registered as placeholders in step 009. This step replaces them with real implementations.

## Deliverables

### Activity Feed Page

Replace the placeholder `ActivityPage` at `ui/src/pages/ActivityPage.tsx` (route `/activity`) with the full activity feed.

**Page header:**
- Page title using the `activity.title` translation key.
- Filter controls area (described below).

**Filter controls:**
- A category filter: a segmented button group or dropdown with options: All, Cascade, Heartbeat, Ideas, System. The default is "All". Selecting a category updates the activity store's filter and re-renders the filtered event list.
- A layer filter: a dropdown listing all 5 layers by display name plus an "All layers" option. The default is "All layers". Selecting a layer filters events to only those involving that specific layer. Events without a layer context (system events, heartbeat events at Tier 1/2) are shown when "All layers" is selected but hidden when a specific layer is chosen.
- The filters apply to both the in-memory real-time events and the REST-fetched historical events. The activity store's `filteredEvents` derived value handles this.

**Event list — real-time section:**
- Displays events from the activity store in reverse chronological order (newest at the top).
- New events arriving via WebSocket appear at the top of the list with a subtle animation (fade-in or slide-in) to draw attention without being disruptive.
- The list uses the activity store's `filteredEvents` so it respects the current filter selections.
- Each event is rendered as an event item (described below).

**Historical events — load more:**
- On page mount, the page calls `fetchHistory()` on the activity store to load recent historical events from `GET /api/activity/`. This populates the store with events from before the user connected.
- The store merges historical events with real-time events, deduplicating by event ID.
- At the bottom of the event list, a "Load more" button fetches older events using the offset parameter. Each load appends the next page of historical events.
- If there are no more historical events to load, the button is hidden.

**Loading and error states:**
- While historical events are loading on mount, show a loading indicator below the real-time events (or skeleton placeholders if no events exist at all).
- If the history fetch fails, show an error message with a retry button.

**Empty state:**
- When no events exist (no real-time events and no historical events), show an empty state explaining that events will appear as the system operates (cascades run, ideas are submitted, heartbeats fire, etc.).

### Event Item Component

A reusable component at `ui/src/components/molecules/EventItem.tsx` (or similar location) that renders a single event in the activity feed:

**Layout per event:**
- An icon on the left indicating the event type.
- The event timestamp formatted as relative time (e.g., "3 minutes ago") with the full ISO timestamp available on hover as a tooltip.
- A category badge (cascade, heartbeat, idea, system) with category-specific colouring.
- A description text describing what happened, generated from the event data.
- Contextual details: layer name with identity colour badge (for events involving a specific layer), agent name or critic archetype (for events involving agents), cascade or idea ID references.

**Event type rendering — each event type produces a specific icon, description, and context:**

Cascade lifecycle events:
- `cascade_started` — Play icon. Description: "Cascade started from {layer name}" with the trigger source (user input, refresh, heartbeat, seed).
- `cascade_completed` — Check icon. Description: "Cascade completed successfully".
- `cascade_failed` — Error icon. Description: "Cascade failed at {layer name} ({step})". Shows the error message.
- `cascade_paused` — Pause icon. Description: "Cascade paused at {layer name} ({step})". Shows the error.
- `cascade_resumed` — Play icon. Description: "Cascade resumed".
- `cascade_cancelled` — X icon. Description: "Cascade cancelled".
- `cascade_queued` — Queue/clock icon. Description: "Cascade queued (position {n})".

Layer processing events:
- `layer_generation_started` — Loader icon. Description: "Generating {layer name}".
- `layer_generation_completed` — Check icon. Description: "{layer name} generated".
- `critic_started` — Comment/speech icon. Description: "{critic name} analyzing {layer name}".
- `critic_completed` — Check icon. Description: "{critic name} completed for {layer name}".
- `synthesis_started` — Merge icon. Description: "Synthesizing assessments for {layer name}".
- `synthesis_completed` — Check icon. Description: "Synthesis completed for {layer name}".

Agent streaming events:
- `agent_text_chunk` — These are high-frequency events that should NOT appear as individual items in the activity feed. They are filtered out. (The cascade store handles them for the live cascade viewer.)

Heartbeat events:
- `heartbeat_started` — Pulse/heart icon. Description: "Heartbeat started ({trigger type})".
- `heartbeat_tier_completed` — Tier-specific icon or numbered indicator. Description: "Heartbeat Tier {n}: {outcome}". If escalated: append "(escalating to Tier {n+1})".
- `heartbeat_completed` — Check icon. Description: "Heartbeat completed — reached Tier {n}".

Idea events:
- `idea_submitted` — Lightbulb icon. Description: "New idea submitted" with the source (human/AI).
- `idea_evaluation_started` — Loader icon. Description: "Idea evaluation started".
- `idea_evaluation_completed` — Check icon. Description: "Idea evaluation completed".
- `idea_generation_started` — Sparkle/magic icon. Description: "AI idea generation started".
- `idea_generation_completed` — Check icon. Description: "{count} ideas generated".

System events:
- `user_login` — User icon. Description: "{email} logged in".
- `user_created` — User-plus icon. Description: "User {email} created ({role})".
- `cascade_lock_acquired` — Lock icon. Description: "Cascade lock acquired".
- `cascade_lock_released` — Unlock icon. Description: "Cascade lock released".

**Styling:**
- Each event type has a subtle left-border accent colour based on its category: cascade events use the cascade accent colour, heartbeat events use a heartbeat-specific colour, idea events use the idea accent, system events use a neutral colour.
- New events arriving in real-time have a brief fade-in animation.
- Events are visually compact — each event is a single row (or at most two lines for events with error messages). The activity feed is a dense information stream, not a spacious card layout.
- The component is reusable and can be used outside the activity feed (e.g., in a sidebar summary or notification area in future steps).

### Version History Page

Replace the placeholder `HistoryPage` at `ui/src/pages/HistoryPage.tsx` (route `/history/:layerSlug`) with the version history view:

**Page header:**
- Page title showing the layer name: "{layer display name} — Version History" using the `history.title` translation key with the layer name substituted.
- A back link navigating to the layer detail page (`/layers/:layerSlug`).
- The header uses the layer's identity colour as an accent, consistent with the layer detail page.

**Commit list:**
- Fetches git commit history from `GET /api/history/:layerSlug` on mount. The layer slug is extracted from the route parameters.
- Validates the layer slug against known layers. If invalid, shows an error state.
- Displays commits in reverse chronological order (newest first) as a vertical list.
- Each commit entry shows:
  - Commit timestamp formatted as an absolute date and time (e.g., "27 Feb 2026, 14:32") with relative time as secondary text (e.g., "2 hours ago").
  - Commit message — the descriptive message explaining what changed (e.g., "Generate values layer [cascade abc123]", "Heartbeat SA update — EU AI Act developments", "Update sovereignty-principles.md in Values"). The message is the primary content of each entry.
  - Short commit hash displayed as a subtle monospace label (e.g., "a1b2c3d").
  - Author name — indicates what triggered the change. Agent-triggered changes show the agent name/type. User-triggered changes show the user email. Heartbeat changes show "Heartbeat". This is derived from the git commit author field, which is set by Policy Factory during commits (steps 007, 008, 015, 017, 019).
- If no commits exist for the layer, show an empty state message (e.g., "No version history yet — this layer has not been modified").

**Pagination:**
- Initial fetch loads the default number of commits (e.g., 20).
- A "Load more" button at the bottom fetches additional commits. The implementer can use offset-based pagination or cursor-based (using the last commit hash).
- If there are no more commits, the button is hidden.

**Navigation context:**
- The version history page is linked from the layer detail page (step 010 — a "View history" link in the layer detail header or metadata area). This step does not modify the layer detail page — the link was established when the route was defined.
- Each layer has its own version history URL: `/history/values`, `/history/situational-awareness`, etc.

**Loading and error states:**
- Loading indicator while commits are being fetched.
- Error state with retry if the fetch fails.
- 404 handling for invalid layer slugs.
