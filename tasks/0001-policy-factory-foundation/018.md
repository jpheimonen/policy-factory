# 018: Idea pipeline

## Context

The idea pipeline gives Policy Factory its creative dimension. While the cascade system (steps 015–017) maintains and refines the existing policy stack, the idea pipeline explores new possibilities — new policies, new tactical approaches, new strategic objectives that the current framework may not have considered.

Ideas flow in from three sources:
1. **Human submission** — a user types an idea into the UI (e.g., "Finland should create a sovereign AI compute fund").
2. **AI generation on demand** — a user clicks "generate ideas", optionally scoped to a specific strategic or tactical objective.
3. **AI generation via heartbeat** — after a cascade triggered by the heartbeat system (step 019, Tier 4), the system autonomously generates ideas that respond to newly discovered developments.

Every idea goes through the same evaluation pipeline:
1. Read the full layer stack (all 5 layers) as context.
2. Score the idea on 6 axes: strategic fit, feasibility, cost, risk, public acceptance, international impact.
3. Run all 6 ideological critics against the idea in parallel.
4. Run the synthesis agent to produce an overall assessment.
5. Store everything in SQLite: the idea record, the 6-axis scores, the critic assessments, and the synthesis.

Critically, idea evaluation operates independently from the cascade. Evaluations are read-only against layer data — they don't modify any files, they don't need the cascade lock, and multiple evaluations can run concurrently. This means a user can submit 3 ideas and have them all evaluating simultaneously while a cascade is running.

This step builds the idea mixin (SQLite storage for ideas), the score mixin (6-axis scores linked to ideas), the evaluation orchestrator, the generation agent, and the ideas REST router. The critic runner and synthesis runner from step 016 are reused directly — they were designed to be generic enough to critique both layers and ideas.

## Deliverables

### Idea Mixin

A store mixin at `src/policy_factory/store/idea.py` added to the composed `PolicyStore`:

**Ideas table:**
- Each idea record tracks: unique ID, text (the idea content), source (human or AI), optional target objective (a reference to a specific strategic or tactical objective item that this idea relates to — expressed as a layer slug + filename, or null if the idea is general), status (pending, evaluating, evaluated, archived), submitted by (user email for human submissions, "system" for AI-generated), submission timestamp, evaluation started timestamp (nullable), evaluation completed timestamp (nullable).

**Idea methods:**
- `create_idea(text, source, target_objective, submitted_by)` — creates an idea with status "pending". Returns the idea ID.
- `get_idea(idea_id)` — returns the full idea record by ID, or None.
- `list_ideas(status, sort_by, sort_order, limit, offset)` — returns ideas with optional filtering and sorting. Filtering by status (pending, evaluating, evaluated, archived). Sorting options: by submission timestamp (default, newest first), by overall score (highest first — requires joining with the scores table), by status. Pagination via limit/offset.
- `update_idea_status(idea_id, status)` — transitions the idea's status. Sets the evaluation_started timestamp when moving to "evaluating" and the evaluation_completed timestamp when moving to "evaluated". Returns success/failure. Does not enforce strict transitions (the caller is responsible for valid transitions) but logs a warning if the transition is unusual (e.g., evaluated → pending).
- `count_ideas(status)` — returns the count of ideas matching an optional status filter. Used for summary statistics in the UI.
- `archive_idea(idea_id)` — sets status to "archived". Archived ideas are excluded from default listings.

### Score Mixin

A store mixin at `src/policy_factory/store/score.py` added to the composed `PolicyStore`:

**Idea scores table:**
- Stores the 6-axis evaluation scores for an idea: unique ID, idea ID (foreign key), strategic fit score (numeric), feasibility score (numeric), cost score (numeric), risk score (numeric), public acceptance score (numeric), international impact score (numeric), overall score (a computed or stored aggregate — the implementer can choose between computing it on read as an average or storing it alongside the axes), the agent run ID (linking to the evaluation agent run that produced these scores), and created timestamp.
- Indexed on idea_id for fast lookup.

**Score methods:**
- `store_scores(idea_id, strategic_fit, feasibility, cost, risk, public_acceptance, international_impact, agent_run_id)` — stores the 6-axis scores for an idea. Computes and stores the overall score (average of the 6 axes). Returns the score record ID.
- `get_scores(idea_id)` — returns the scores for an idea, including all 6 axes and the overall score. Returns None if no scores exist yet (idea hasn't been evaluated).
- `get_top_scored_ideas(limit)` — returns idea IDs ordered by overall score descending. Used for the "top ideas" display.

### Evaluation Orchestrator

An async function at `src/policy_factory/ideas/evaluator.py` that drives the evaluation pipeline for a single idea:

**Evaluation flow:**

1. Update the idea status to "evaluating" via the store.
2. Emit an `IdeaEvaluationStarted` event.
3. **Gather context** — read summaries of all 5 layers to provide the full policy stack context. This uses the stack summary helper (described below) which produces a formatted text of all layer summaries.
4. **Scoring step:**
   a. Build the evaluation prompt using the prompt construction helper: load `prompts/ideas/evaluate.md`, substitute the idea text and the stack summary.
   b. Create an agent session with the idea-evaluator model.
   c. Record an agent run.
   d. Run the evaluation agent.
   e. Parse the 6-axis scores from the agent's output. The prompt instructs the AI to return scores in a structured format. The parser extracts numeric values for each axis. If parsing fails (the AI didn't follow the expected format), default scores are recorded (e.g., all zeros) with a note that parsing failed, and the raw evaluation text is preserved.
   f. Store the scores via the score mixin.
   g. Record agent run completion.
5. **Critics step:**
   a. Prepare the content to critique: the idea text combined with the evaluation context (stack summary and scores).
   b. Call the critic runner (step 016) with `cascade_id=None`, the idea ID, and the content. The critic runner handles launching all 6 critics in parallel, recording agent runs, and storing results.
6. **Synthesis step:**
   a. Call the synthesis runner (step 016) with the critic results, `cascade_id=None`, and the idea ID. The synthesis runner handles the synthesis agent, recording, and storage.
7. Update the idea status to "evaluated".
8. Emit an `IdeaEvaluationCompleted` event.
9. Return the evaluation result: scores, critic result summary, synthesis summary.

**Error handling:**
- If the scoring step fails, the idea status remains "evaluating" and the error is logged. The evaluation does not proceed to critics/synthesis since there are no scores to contextualise.
- If the critic step fails (all critics failed), the evaluation still proceeds to synthesis — but the synthesis may produce a limited assessment. The idea is still marked as "evaluated" since partial results are useful.
- If any step fails with a non-recoverable error, the idea status is set back to "pending" so it can be re-evaluated later.

**Concurrency:**
- Multiple evaluations can run simultaneously. Each evaluation operates on its own idea record and its own agent sessions. There are no shared mutable state concerns — each evaluation reads layer data (read-only) and writes to its own idea/score/critic records.

### Generation Agent

An async function at `src/policy_factory/ideas/generator.py` that generates new policy ideas:

**Generation flow:**

1. Emit an `IdeaGenerationStarted` event.
2. Gather the full stack summary (all 5 layers).
3. Build the generation prompt using the prompt construction helper: load `prompts/ideas/generate.md`, substitute the stack summary and optional scoping context (if the generation was scoped to a specific objective).
4. Create an agent session with the idea-generator model (default Sonnet).
5. Record an agent run.
6. Run the generation agent. The prompt instructs the AI to produce multiple ideas (typically 3–5), each as a distinct, clearly delineated suggestion.
7. Parse the agent's output to extract individual ideas. Each idea is a separate text block that becomes its own idea record.
8. For each extracted idea:
   a. Create an idea record with source "AI", the scoping context as the target objective (if applicable), and "system" as the submitter.
   b. Emit an `IdeaSubmitted` event for each idea.
9. Record agent run completion.
10. Emit an `IdeaGenerationCompleted` event with the count of generated ideas.
11. Optionally trigger evaluation for each generated idea (by launching evaluation as background tasks). Whether auto-evaluation happens can be configurable — the default is to auto-evaluate so users see fully scored ideas immediately.
12. Return the list of generated idea IDs.

**Scoping:**
- When scoped to a specific objective (e.g., "generate ideas for strategic objective: strengthen semiconductor supply chain"), the prompt includes that objective's content prominently as context, and the generated ideas are tagged with the objective as their target.
- When unscoped, the agent considers the entire policy stack and generates ideas across the framework.

### Stack Summary Helper

A utility function that gathers brief summaries of all 5 layers for use in idea evaluation and generation prompts:

- For each of the 5 layers (in hierarchical order, values to policies):
  - Reads the layer's narrative summary (README.md).
  - Optionally reads item titles from the layer (not full content — just titles to keep the summary concise).
  - Assembles a brief per-layer summary.
- Combines all 5 per-layer summaries into a single formatted text block.
- This is distinct from the layer content gathering helper in step 016, which gathers the full content of a single layer. The stack summary is broader (all layers) but shallower (summaries, not full content).

### Ideas Router

A router at `src/policy_factory/server/routers/ideas.py` with prefix `/api/ideas` and tag `ideas`. All endpoints require JWT authentication.

**POST `/api/ideas/`** — Submit a human idea.
- Accepts a request body with: the idea text (string, required), and an optional target objective (layer slug + item filename).
- Creates an idea record with source "human" and the authenticated user's email as the submitter.
- Emits an `IdeaSubmitted` event.
- Launches evaluation as a background asyncio task (the user doesn't wait for evaluation).
- Returns the idea ID and status "pending" immediately.

**POST `/api/ideas/generate`** — Trigger AI idea generation.
- Accepts an optional request body with: scoping context (an objective's layer slug + item filename, or null for unscoped generation).
- Launches the generation agent as a background asyncio task.
- Returns immediately with a confirmation that generation has been started.
- The generated ideas appear in the idea list as they are created (the frontend idea store receives `IdeaSubmitted` events via WebSocket).

**GET `/api/ideas/`** — List ideas.
- Accepts optional query parameters: `status` (filter by pending/evaluating/evaluated/archived), `sort` (submission_date, score — default: submission_date), `order` (asc/desc — default: desc), `limit` (default 50, max 200), `offset` (for pagination).
- Returns ideas with summary information: ID, text (truncated to a preview length), source, target objective, status, submission timestamp, overall score (if evaluated), submitted by.
- Archived ideas are excluded by default unless `status=archived` is explicitly passed.

**GET `/api/ideas/:idea_id`** — Get full idea detail.
- Returns the complete idea record including: full text, source, target objective, status, timestamps, all 6-axis scores (if evaluated), all 6 critic assessments (from the critic result mixin's `get_critic_results_for_idea`), the synthesis result (from `get_synthesis_result_for_idea`), and the submitted_by field.
- Returns 404 for an unknown idea ID.

**PUT `/api/ideas/:idea_id/archive`** — Archive an idea.
- Sets the idea status to "archived".
- Returns the updated idea.
- Returns 404 for an unknown idea ID.

**POST `/api/ideas/:idea_id/evaluate`** — Re-trigger evaluation for a specific idea.
- Useful for re-evaluating an idea after the policy stack has changed.
- Only allowed when the idea is in "pending" or "evaluated" status (not currently being evaluated).
- Returns 409 if the idea is currently being evaluated.
- Resets the idea status to "pending", then launches evaluation as a background task.
- Returns immediately.
