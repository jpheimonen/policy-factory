# 009: Frontend stack overview and navigation

## Context

The auth flow is complete (step 006) ã users can log in, register, and navigate to placeholder protected routes. The backend layer API is live (step 008), serving layer metadata, item listings, narrative summaries, and feedback memo counts. Now the frontend needs its first real page: the stack overview home page.

The stack overview is the application's centrepiece. It visualises the five-layer policy model as a vertical stack ã Values at the bottom, Policies at the top ã with each layer rendered as a card in its identity colour. Clicking a layer navigates to its detail view. The page also shows a cascade status indicator in the header area (reflecting whether a cascade is currently running, paused, or idle) so users always have situational awareness of system activity.

This step also finalises the app routing structure by replacing the placeholder routes from step 006 with properly named page-level placeholders that import their styles module and render a title (real content comes in later steps). The navigation component from step 006 is refined into a proper sidebar or top-nav with all page links, active-route highlighting, and the cascade status indicator.

This step introduces the **layer store** ã a zustand store that fetches layer listing data from the API (`GET /api/layers/`) and exposes it to the stack overview page. It follows the same zustand patterns established by the auth store (step 006) and the theme store (step 002).

## Deliverables

### Layer Store

A zustand store at `ui/src/stores/layerStore.ts` managing layer listing data:

**State:**
- The list of layers returned by `GET /api/layers/` ã each layer containing: slug, display name, item count, last updated timestamp, narrative summary preview text, and pending feedback memo count.
- A loading flag (true during the initial fetch).
- An error state for API failures.

**Actions:**
- **Fetch layers**: calls `GET /api/layers/` via the authenticated HTTP client, populates the store with the response. Called on mount of the stack overview page.
- **Refresh**: re-fetches layer data without showing a loading spinner (keeps stale data visible during the fetch, replaces on success). Used when returning to the stack overview from another page or on visibility change.
- **Initialize**: one-time setup that triggers the first fetch. Follows the cc-runner `projectInfoStore` pattern with an `init()` function and an `initialized` guard to prevent duplicate calls.

**Derived/computed:**
- The store provides layers in their canonical order (Values at bottom through Policies at top) regardless of the order the API returns them. The ordering is determined by the layer definitions established in step 007 (the five layers have a fixed hierarchy).

### Stack Overview Page

A page component at `ui/src/pages/StackOverviewPage.tsx` (route: `/`) serving as the authenticated home page:

**Layout:**
- The page renders within the app shell (navigation visible at all times).
- A page title at the top using the `stackOverview.title` translation key.
- The five layers rendered as a vertical stack of cards ã ordered from bottom (Values) to top (Policies) ã so that the visual hierarchy mirrors the conceptual model. The bottommost layer (Values) appears at the bottom of the stack on screen.
- Each layer card is visually distinct via its identity colour.
- The page handles loading state (skeleton placeholders while the API call is in flight) and error state (a message if the API call fails with a retry option).

**Layer card content (per card):**
- The layer's display name, prominently styled with its identity colour accent.
- Item count ã how many markdown items exist in that layer directory.
- Last updated timestamp ã a human-readable relative time (e.g., "2 hours ago", "yesterday"). Use a lightweight relative-time formatting approach.
- Narrative summary preview ã the first ~2 lines of the layer's README.md content, truncated with ellipsis. Gives a sense of the layer's current state without navigating in.
- Pending feedback memo count ã if there are any pending memos targeting this layer, show a badge with the count. If zero, omit the badge (no "0 memos" clutter).

**Interactions:**
- Clicking a layer card navigates to `/layers/:layerSlug` (the layer detail view ã placeholder page until step 010).
- The page fetches data from the layer store on mount. If the user navigates away and returns, it refreshes (but keeps stale data visible during the refresh).

**Responsive behaviour:**
- The vertical card stack works well at all viewport widths. Cards stretch to fill the available width (with a max-width to prevent overly wide cards on large screens).
- At narrower viewports (below 768px), the layout adjusts gracefully ã cards may become full-width with reduced padding.

### Navigation Component Refinement

The basic navigation component from step 006 is upgraded:

**Page links:**
- Home (stack overview) ã route `/`
- Ideas ã route `/ideas`
- Cascade ã route `/cascade`
- Activity ã route `/activity`
- Admin ã route `/admin` (visible only to admin users, per step 006)

Each link highlights when its route is active (using the current URL path). The active state uses the theme's accent colour or a subtle background change to indicate the current page.

**Cascade status indicator:**
- A small, always-visible status element in the navigation or header area showing the current cascade state: idle, running (with which layer is being processed, if available), or paused (indicating attention needed).
- For now (before the WebSocket integration in step 013), this indicator fetches its initial state from `GET /api/cascade/status` on mount. It shows static status until the WebSocket integration makes it real-time.
- Visual states: idle shows nothing or a subtle "idle" indicator; running shows an animated indicator (e.g., a pulsing dot or spinner) with context text; paused shows a warning-coloured indicator.
- The cascade store (which will be fully wired to WebSocket events in step 013) is introduced here with its basic shape ã it holds the cascade status fetched from the REST API. Step 013 upgrades it to receive real-time WebSocket events.

**User controls (carried forward from step 006):**
- Current user's email display.
- Theme toggle (dark/light).
- Logout button.

**Layout decision:**
- The navigation can be a left sidebar or a top header bar ã the implementer should choose whichever produces a cleaner result given the page layout. The architecture spec does not prescribe sidebar vs. header. For reference, cc-runner uses a per-page header pattern (NavHeader) rather than a persistent sidebar. Policy Factory may benefit from a persistent sidebar since users navigate between a fixed set of pages frequently, but a top nav is also acceptable.
- Whichever layout is chosen, the navigation must be visible on all protected pages and must not overlap or push the main page content in an awkward way.

### Cascade Store (Initial Shell)

A zustand store at `ui/src/stores/cascadeStore.ts` with the initial shape:

**State:**
- Current cascade status: idle, running, paused, or failed.
- If running: the current layer being processed and the current step (generation, critics, synthesis).
- Queue depth (how many cascades are waiting).
- Error info if paused/failed.

**Actions:**
- **Fetch status**: calls `GET /api/cascade/status` via the authenticated HTTP client, populates the store. Called on mount of the navigation component so the cascade indicator is always current.
- **Refresh**: re-fetch without loading state.

This is a minimal shell ã step 013 (Frontend WebSocket integration) will extend it with WebSocket event handlers that update the store in real-time. For now it is REST-only and provides enough for the cascade status indicator in the navigation.
