# 003: i18n infrastructure

## Context

Every user-visible string in Policy Factory must be externalized from day one — zero hardcoded strings in components. English is the only language for now, but the architecture must support adding locales by adding translation files without touching component code. This is a hard requirement because the tool serves Finland's cross-party tech group, and Finnish translations are a natural future addition.

cc-runner has zero i18n infrastructure — all strings are hardcoded throughout components. This means there is no existing pattern to follow and no i18n library in the dependency tree. Policy Factory needs to establish the full i18n pattern from scratch in this step, before any real UI components are built in subsequent steps (steps 006–011, 020–023). Getting this right now prevents a painful retrofit later.

The i18n approach should be lightweight. Policy Factory's translation needs are straightforward — static UI labels, status messages, button text, page titles, error messages — with no complex pluralization, date formatting, or ICU message format requirements. A simple key-lookup system with a flat namespace is sufficient. The implementer should choose between a lightweight library (react-i18next, or similar) and a minimal custom hook — either approach is acceptable as long as it satisfies the acceptance criteria.

## Related Code

There is no i18n code in cc-runner to reference. The following cc-runner files show the *problem* — hardcoded strings throughout components that Policy Factory must avoid:

- `cc-runner/ui/src/pages/ModeSelector.styles.ts` — Page-level styled components with hardcoded text in the UI (titles, descriptions, button labels). Policy Factory components must reference translation keys instead.
- `cc-runner/ui/src/components/organisms/HeaderBase.tsx` — Header component with inline text. Illustrates the pattern to avoid.
- `cc-runner/ui/src/components/molecules/ConfirmModal.tsx` — Modal with hardcoded "Cancel", "Confirm" strings. Policy Factory equivalents must use translation keys.

From Policy Factory itself (created in prior steps):

- `ui/src/App.tsx` (step 001) — The app root that needs to be wrapped with the i18n provider. The architecture specifies the provider order: ThemeProvider → i18n Provider → AuthProvider → GlobalStyles → BrowserRouter → Routes.
- `ui/src/styles/theme.ts` (step 002) — The theme store pattern (zustand + localStorage persistence) can inform how a locale store might work if one is needed in the future.

## Reuse Opportunities

- **No direct reuse** — cc-runner has no i18n infrastructure. This is entirely new functionality.
- The **zustand store pattern** established in step 002 (theme store) could be followed if a locale store is needed in the future to track the user's language preference. For now, English is the only language, so a locale store is not required in this step — but the architecture should not preclude adding one.
- The **flat key namespace** pattern (`page.section.element`) is standard across all i18n solutions and does not depend on the specific library chosen.

## Deliverables

### Translation File

A single English translation file containing all the keys that will be used throughout the application. The file should use a flat dot-notation namespace organized by page or feature area. The namespace structure should cover these areas (with initial keys — more keys will be added in subsequent steps as pages are built):

- `common` — shared strings used across multiple pages: button labels (save, cancel, delete, confirm, close, back, refresh, submit), status labels (loading, error, success), and generic UI text (no items, empty state)
- `nav` — navigation labels: all page names (stack overview, ideas, cascade, activity, admin, version history), app name
- `auth` — authentication flow: login page title, email label, password label, login button, register page title, registration strings, error messages (invalid credentials, session expired)
- `stackOverview` — home page: page title, layer names (values, situational awareness, strategic objectives, tactical objectives, policies), item count labels, last updated label, feedback memo count, input panel placeholder, submit button
- `layers` — layer detail view: narrative summary heading, items heading, refresh button, feedback memo section (accept, dismiss), critic assessment heading
- `items` — item detail/editing: edit button, save button, cancel edit, frontmatter field labels, cross-references heading, attribution labels (last modified by, timestamp)
- `ideas` — idea inbox: submit form placeholder, submit button, generate button, status labels (pending, evaluating, evaluated, archived), score axis names (strategic fit, feasibility, cost, risk, public acceptance, international impact), radar chart labels
- `cascade` — cascade viewer: status labels (idle, running, paused, completed, failed, cancelled, queued), step labels (generation, critics, synthesis), resume button, cancel button, queue heading, error display
- `heartbeat` — heartbeat: trigger button, tier names (news skim, triage analysis, SA update, full cascade), log heading
- `activity` — activity feed: page title, filter labels, event type names, layer filter
- `admin` — admin panel: page title, user list heading, create user form labels, delete confirmation
- `history` — version history: page title, date column, change column, trigger column
- `critics` — critic perspective names: realist, liberal-institutionalist, nationalist-conservative, social-democratic, libertarian, green-ecological. Plus the synthesis label.
- `errors` — error messages: network error, unauthorized, forbidden, not found, server error, cascade lock held, generic fallback

This file serves as the canonical list of all UI strings. Components never contain raw display text — they always reference a key from this file.

### i18n Provider and Hook

A mechanism for components to access translated strings. This must include:

- A **provider component** that wraps the app and makes translations available to all descendants. This goes between ThemeProvider and AuthProvider in the App.tsx provider stack.
- A **hook or function** that components use to look up a translation by key. It should accept a translation key string and return the translated string. The lookup must work with the dot-notation namespace (e.g., asking for `cascade.status.running` returns the value from the nested structure).
- **Missing key handling**: when a component references a key that doesn't exist in the translation file, the system should return the key itself as a fallback (so the missing key is visible in the UI as a debugging aid) and optionally log a warning in development mode.
- **Interpolation support**: the system must support inserting dynamic values into translation strings (e.g., a string like "Last updated {time}" where `{time}` is substituted at render time). This is needed for timestamps, counts, and user names in various UI strings.

### App Shell Update

Update `App.tsx` to include the i18n provider in the correct position in the provider hierarchy: ThemeProvider → i18n Provider → (future AuthProvider) → GlobalStyles → BrowserRouter → Routes.

### Atom Component Update

Update any atom components from step 002 that contain placeholder text (e.g., if Button has a default label, or if any test fixtures use hardcoded strings) to use translation keys instead, establishing the pattern for all subsequent component development.

### Testing Utilities

A test helper or mock for the i18n system so that component tests can run without needing the full i18n provider. Tests should be able to either:
- Wrap components in a lightweight test provider that returns keys as-is (so assertions match on keys), or
- Wrap components in a provider loaded with the real English translations (so assertions match on English text)

Both approaches are valid. The important thing is that the testing pattern is established here so subsequent steps follow it consistently.

## Acceptance Criteria

- [x] A single English translation file exists containing all namespaced keys organized by page/feature area
- [x] An i18n provider wraps the app in the correct position in the provider hierarchy (between ThemeProvider and the future AuthProvider position)
- [x] Components can access translations via a hook or function that accepts a dot-notation key and returns the English string
- [x] Looking up a non-existent translation key returns the key itself as a fallback (visible debugging aid)
- [x] Translation strings support dynamic value interpolation (e.g., inserting a count or name into a template string)
- [x] Adding a second locale (e.g., Finnish) requires only adding a new translation file and a locale selector — no component code changes
- [x] The translation file includes keys for all major UI areas: common actions, navigation, auth, stack overview, layers, items, ideas, cascade, heartbeat, activity, admin, history, critics, and errors
- [x] The atom components from step 002 do not contain any hardcoded display strings (any visible text uses translation keys)
- [x] A testing utility or mock exists so that component tests can render components that use translations without errors
- [x] The i18n solution adds minimal bundle size overhead (prefer lightweight solutions — the translation needs are simple key lookups with interpolation, not complex ICU message formatting)
