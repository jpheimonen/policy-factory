# 017: Input classifier and cascade trigger API

## Context

Steps 015 and 016 built the cascade orchestrator and the critic/synthesis system, but there is no way to trigger them yet. The cascade trigger function (step 015) exists as internal code — no REST endpoint calls it, no user action invokes it. This step builds the API surface that connects user actions to the cascade machinery, plus two additional components that close gaps in the pipeline: the input classifier agent and the feedback memo system.

This step delivers four things:

1. **Cascade router** — REST endpoints for triggering cascades (from user input, from a specific layer refresh, from a full refresh), checking cascade status, pausing/resuming/cancelling cascades, and viewing the queue. This is the primary API surface for cascade operations, consumed by the frontend (steps 020, 023).

2. **Input classifier agent** — When a user submits free-text input (e.g., "Finland should prioritise semiconductor manufacturing independence"), the system needs to determine which layer it most affects before triggering a cascade. The classifier is a lightweight AI agent that receives the input text and brief descriptions of each layer, then returns the target layer with an explanation. After classification, the cascade orchestrator is triggered with the user input as additional context for the target layer's generation agent.

3. **Seed router and seed agent** — The initial population of the Situational Awareness layer. On a fresh system, the values layer is pre-seeded (step 007) but all other layers are empty. The seed endpoint triggers a special agent that uses web search to research Finland's current tech policy landscape and populates the Situational Awareness layer. After seeding, an upward cascade is triggered from SA through the remaining layers.

4. **Feedback memo mixin** — The bidirectional feedback system that connects layers. During generation, an agent for layer N may discover tensions or feasibility issues with items in layers below N. These are recorded as feedback memos — transient, actionable notes targeting specific layers. The next time the target layer is regenerated, the pending memos are included in its generation agent's context. The cascade orchestrator (step 015) already references feedback memos in its context gathering, but the actual storage and retrieval mechanism was deferred to this step.

The cascade router follows the same pattern as cc-runner's runs router: POST endpoints that trigger background operations and return immediately, with status queryable via GET endpoints and control via POST stop/resume/cancel endpoints.

## Reuse Opportunities

- The **runs router pattern** from cc-runner (POST trigger → background task → return ID, GET status, POST stop/resume) maps directly to the cascade router. The structural similarity is very high — the main difference is that Policy Factory uses a cascade lock + queue instead of cc-runner's single-active-run guard.
- The **cascade trigger function** from step 015 handles all the complex logic (lock, queue, background task). The router endpoint is a thin HTTP wrapper that validates input, calls the trigger function, and returns the result.
- The **cascade controller** from step 015 handles all pause/resume/cancel state transitions. The router endpoints just look up the controller and call its methods.
- The **agent session + prompt construction** pattern from step 014 is used for both the classifier and seed agents. Same pattern as every other agent: resolve model → build prompt → create session → run → handle result.
- The **store mixin pattern** from steps 004 and 015 is extended for the feedback memo mixin.
- The **layer definitions** from step 007 provide the layer descriptions needed by the classifier prompt.

## Deliverables

### Cascade Router

A router at `src/policy_factory/server/routers/cascade.py` with prefix `/api/cascade` and tag `cascade`. All endpoints require JWT authentication.

**POST `/api/cascade/trigger`** — Trigger a cascade from user input.
- Accepts a request body with: the user's free-text input (string).
- First runs the input classifier agent (described below) to determine which layer the input affects.
- Then calls the cascade trigger function (step 015) with trigger source "user-input", the classified target layer as the starting layer, and the user's input text as context.
- Returns immediately with: the classification result (target layer and explanation), and either a cascade ID (if the cascade started) or a queue entry ID (if the cascade was queued because the lock was held).
- The classification happens synchronously within the request (it's a fast, lightweight agent call). The cascade itself runs asynchronously in the background.

**POST `/api/cascade/refresh`** — Trigger a layer refresh.
- Accepts a request body with: the layer slug to refresh.
- Validates the layer slug.
- Calls the cascade trigger function with trigger source "layer-refresh" and the specified layer as the starting layer.
- Returns the cascade ID or queue entry ID.
- No classifier is needed — the user explicitly chose which layer to refresh.

**POST `/api/cascade/full`** — Trigger a full cascade from values.
- No request body needed (or optional configuration).
- Calls the cascade trigger function with trigger source "layer-refresh" and "values" as the starting layer.
- Returns the cascade ID or queue entry ID.
- This regenerates the entire policy stack from the ground up.

**GET `/api/cascade/status`** — Get current cascade status.
- Returns: whether a cascade is currently running or paused (with its cascade ID, current layer, current step, trigger source, started timestamp), the queue depth and queue entries (each with queue ID, trigger source, starting layer, queued timestamp), and the ID of the most recently completed cascade.
- If no cascade is running and the queue is empty, returns an idle status.
- This endpoint is polled by the cascade status indicator in the navigation (step 009) and the live cascade viewer (step 020).

**GET `/api/cascade/:cascade_id`** — Get detailed cascade run information.
- Returns the full cascade run record: ID, trigger source, starting layer, current layer, current step, status, error message (if paused/failed), timestamps, and the list of agent runs associated with this cascade (from the agent run mixin — each with agent type, model, target layer, success/failure, timing).
- Returns 404 for an unknown cascade ID.

**GET `/api/cascade/history`** — List recent cascade runs.
- Returns cascade runs in reverse chronological order.
- Accepts optional query parameters: `limit` (default 20, max 100), `offset` (for pagination), `status` (filter by status: completed, failed, cancelled).
- Each entry includes: ID, trigger source, starting layer, status, started/completed timestamps.

**POST `/api/cascade/:cascade_id/pause`** — Pause a running cascade.
- Looks up the cascade controller in the deps registry.
- Returns 404 if the cascade is not found (not active).
- Calls `controller.pause()`.
- Returns 409 if the cascade is not in a pausable state (already paused, completed, etc.).
- Returns success with the updated cascade status.

**POST `/api/cascade/:cascade_id/resume`** — Resume a paused cascade.
- Looks up the cascade controller in the deps registry.
- Returns 404 if the cascade is not found.
- Calls `controller.resume()`.
- Returns 409 if the cascade is not paused.
- Returns success with the updated cascade status.

**POST `/api/cascade/:cascade_id/cancel`** — Cancel a paused cascade.
- Looks up the cascade controller in the deps registry.
- Returns 404 if not found.
- Calls `controller.cancel()`.
- Returns 409 if not in a cancellable state.
- Returns success.

**DELETE `/api/cascade/queue/:queue_id`** — Cancel a queued cascade.
- Removes a specific entry from the cascade queue.
- Returns 404 if the queue entry doesn't exist.
- Returns 204 on success.

### Input Classifier

An async function at `src/policy_factory/cascade/classifier.py` that classifies user input to determine the target layer:

**Classification flow:**
1. Gather brief descriptions of each of the 5 layers: for each layer, include the layer's display name, a one-sentence description of what it contains, and a brief summary of its current state (number of items, or a snippet of the narrative summary). These descriptions are built from the layer definitions (step 007) and current layer data.
2. Build the classifier prompt using the prompt construction helper: load the classifier template from `prompts/classifier/classifier.md`, substitute the user's input text and the layer descriptions.
3. Create an agent session with the classifier model (resolved for role "classifier" — default Sonnet), working directory set to `data/`, and the EventEmitter for streaming.
4. Record an agent run in the agent runs table.
5. Run the agent session.
6. Parse the classification result from the agent output. The classifier prompt instructs the AI to return a structured response: the target layer slug, an optional list of secondary affected layers, a confidence level, and an explanation.
7. Validate the returned layer slug against the known layer slugs. If the AI returns an invalid slug, fall back to "situational-awareness" as a safe default (since most external inputs relate to the current state of the world).
8. Record the agent run completion.
9. Return the classification result: target layer slug, explanation, and confidence.

**Classification is synchronous within the API request.** Unlike the cascade itself (which runs in the background), classification is a quick operation — a single lightweight agent call. The API endpoint waits for classification to complete before returning the result to the client and triggering the cascade.

### Seed Router

A router at `src/policy_factory/server/routers/seed.py` with prefix `/api/seed` and tag `seed`. All endpoints require JWT authentication.

**POST `/api/seed/`** — Trigger initial Situational Awareness seeding.
- Checks whether the SA layer already has content (items beyond what was pre-seeded). If it does, returns a 409 indicating seeding has already been performed. Seeding is a one-time operation for initial system setup — after seeding, the heartbeat and user input drive SA updates.
- Reads the current values layer content (pre-seeded in step 007) as context for the seed agent.
- Creates an agent session with the seed model (resolved for role "seed" — default Opus, since this is a one-time operation that needs high-quality output with web search), working directory set to `data/`, and the EventEmitter for streaming.
- Records an agent run.
- Runs the seed agent with the seed prompt (from `prompts/seed/seed.md`), which instructs the AI to use web search to research Finland's current tech policy landscape and create SA layer content.
- After the seed agent completes: auto-commits the data git repo with a descriptive message.
- Triggers an upward cascade from the SA layer (by calling the cascade trigger function with trigger source "seed" and starting layer "situational-awareness"). This populates strategic objectives, tactical objectives, and policies based on the newly created SA content.
- Returns the result: success/failure, the cascade ID for the triggered cascade.
- The seeding itself runs synchronously (since it's a one-time setup operation and the user is waiting for it), but the subsequent cascade runs asynchronously.

**GET `/api/seed/status`** — Check seeding status.
- Returns whether seeding has been performed (whether the SA layer has non-pre-seeded content).
- Used by the frontend to determine whether to show a "Seed the system" prompt on initial setup.

### Feedback Memo Mixin

A store mixin at `src/policy_factory/store/feedback_memo.py` added to the composed `PolicyStore`:

**Feedback memos table:**
- Each memo tracks: unique ID, source layer slug (the layer whose generation produced the memo), target layer slug (the layer the memo is directed at), cascade ID (the cascade during which the memo was generated), content (the text of the feedback — what the generation agent discovered that the target layer should address), referenced items (an optional JSON list of item filenames in the target layer that the feedback relates to), status (pending, accepted, dismissed — memos start as pending and are either accepted or dismissed by the next generation of the target layer, or manually by a user), created timestamp, and resolved timestamp (when the status changed from pending).

**Feedback memo methods:**
- `create_feedback_memo(source_layer, target_layer, cascade_id, content, referenced_items)` — creates a pending feedback memo. Returns the memo ID.
- `get_pending_memos(target_layer)` — returns all pending memos targeting a specific layer. Used by the cascade orchestrator (step 015) when gathering context for a layer's generation agent. Sorted by creation date (oldest first).
- `get_pending_memo_count(target_layer)` — returns the count of pending memos for a layer. Used by the stack overview page (step 009) to display memo counts per layer.
- `update_memo_status(memo_id, status)` — updates a memo's status to accepted or dismissed. Sets the resolved timestamp. Used by the generation agent (which processes memos during generation) and by the frontend (manual accept/dismiss on the layer detail page, step 010).
- `batch_update_memo_status(memo_ids, status)` — updates multiple memos at once. Used after a generation agent processes all pending memos for a layer.
- `list_memos(target_layer, source_layer, status, cascade_id, limit, offset)` — returns memos with optional filtering. Used by the layer detail page to display incoming feedback.
- `get_memo(memo_id)` — returns a single memo by ID.

**Schema additions:**
- The feedback_memos table definition is added to the schema initialization with indexes on target_layer + status (for the common "get pending memos for layer" query) and on cascade_id.

### Feedback Memo Router

A router at `src/policy_factory/server/routers/feedback.py` with prefix `/api/feedback` and tag `feedback`. All endpoints require JWT authentication.

**GET `/api/feedback/:layer_slug`** — List feedback memos for a layer.
- Validates the layer slug.
- Accepts optional query parameters: `status` (filter by pending/accepted/dismissed, default: all), `limit` (default 50), `offset`.
- Returns memos in reverse chronological order, each with: ID, source layer, content, referenced items, status, creation timestamp.

**PUT `/api/feedback/:memo_id`** — Update a memo's status.
- Accepts a request body with: the new status (accepted or dismissed).
- Returns 404 if the memo doesn't exist.
- Updates the memo status and sets the resolved timestamp.
- Returns the updated memo.
