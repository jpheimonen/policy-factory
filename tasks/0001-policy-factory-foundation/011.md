# 011: Frontend item detail and editing

## Context

The layer detail view (step 010) displays all items in a layer as cards. Clicking an item card navigates to `/layers/:layerSlug/:itemSlug` — currently a placeholder page. This step replaces that placeholder with the full item detail and editing page.

This is the deepest drill-down in the layer hierarchy: stack overview → layer detail → item detail. The item detail page renders the full content of a single markdown file — all frontmatter fields as labelled fields and the markdown body as formatted HTML. It also displays cross-layer references as clickable links and shows attribution metadata (who or what last modified the item, when).

The page supports inline editing: a toggle switches between view mode and edit mode. In edit mode, frontmatter fields become editable form inputs and the markdown body becomes a textarea. Saving calls `PUT /api/layers/:slug/items/:filename` which writes the file, auto-sets the `last_modified` and `last_modified_by` fields, and triggers a git commit. The architecture spec also notes that saving an item in a lower layer could potentially trigger a cascade — for now the save action writes the item and optionally offers a "trigger cascade" action after saving (since the cascade trigger endpoint arrives in step 017, this is handled gracefully).

This page also fetches and displays cross-layer references via `GET /api/layers/:slug/items/:filename/references`, showing both forward references (items this item points to) and backward references (items that point to this item) as clickable navigation links.

**Backend endpoints (step 008):**

- `GET /api/layers/:slug/items/:filename` — Returns the full item: frontmatter (JSON object) and body (markdown string).

- `GET /api/layers/:slug/items/:filename/references` — Returns forward and backward cross-layer references, each with layer slug, filename, and item title.

## Deliverables

### FormField Molecule

A reusable form field wrapper component at `ui/src/components/molecules/FormField.tsx`:

- Wraps a form input (Input, Textarea, Select, or any child) with a label above it and optional error text below.
- Accepts: `label` (string, displayed above the input), `error` (string, displayed below in error styling), `helpText` (string, displayed below in muted styling), `required` (boolean, shows a required indicator next to the label), and `htmlFor` (string, for accessibility linking the label to the input).
- Uses i18n-compatible labels — the parent passes pre-translated label strings.
- Styled consistently with the design system using theme tokens.
- Follows the cc-runner `FormField.tsx` pattern.

### ConfirmModal Molecule

A reusable confirmation dialog at `ui/src/components/molecules/ConfirmModal.tsx`:

- A modal overlay with a title, message, and two buttons: confirm and cancel.
- Accepts: `isOpen`, `onConfirm`, `onCancel`, `title`, `message`, and an optional `variant` (default, warning, danger) that affects the confirm button's colour.
- Keyboard support: Escape closes/cancels, Enter confirms.
- The confirm button for the "danger" variant uses destructive styling (red/error colour) to indicate irreversible actions.
- Used on this page for: discarding unsaved changes when leaving edit mode, and confirming item deletion. Will be reused across many pages.
- All button labels and the title use i18n translation keys.

### Unsaved Changes Warning Hook

A custom hook (e.g., `ui/src/hooks/useBeforeUnload.ts`) that warns the user when they have unsaved changes:

- Accepts a boolean condition (e.g., `isDirty`).
- When the condition is true, registers the browser's `beforeunload` event to show the native "Leave site?" dialog if the user tries to close the tab or navigate away.
- Cleans up the event listener when the condition becomes false or the component unmounts.
- Follows the cc-runner `useBeforeUnload.ts` pattern.

### Item Detail State

The item detail page needs state for the currently viewed item. This can be managed in one of two ways — the implementer should choose the approach that integrates cleanest with the existing layer store:

**Option A — Extend the layer store:** Add a `currentItem` slice to the existing `layerStore.ts` with the item's frontmatter, body, and references. This keeps all layer-related data in one store.

**Option B — Separate custom hook:** Create a `useItem` hook (similar to cc-runner's `useTask` hook) that fetches a single item by slug + filename and returns `{ item, references, loading, error, refetch }`. This keeps the layer store focused on layer-level data.

Either way, the state management must support:

- Fetching the item data: calls `GET /api/layers/:slug/items/:filename` to get frontmatter + body.
- Fetching cross-layer references: calls `GET /api/layers/:slug/items/:filename/references` to get forward and backward references.
- Both fetches happen in parallel on mount and when the route params change.
- Loading and error states.
- A refresh action for re-fetching after a save.

### Item Detail Page — View Mode

Replace the placeholder `ItemDetailPage` from step 009 with a full implementation at `ui/src/pages/ItemDetailPage.tsx`:

**Route:** `/layers/:layerSlug/:itemSlug`

The `itemSlug` route parameter is the filename (without path) of the markdown file. The page uses both `layerSlug` and `itemSlug` to fetch the item from the API.

**Page header:**
- The item's title (from frontmatter), styled prominently.
- The layer name with its identity colour, displayed as a subtitle or breadcrumb (e.g., "Values > National Security") so the user knows which layer this item belongs to.
- A back link to the layer detail view (`/layers/:layerSlug`).
- An "Edit" button in the header's right area that toggles into edit mode.
- A "Delete" button (styled subtly or placed in a secondary/danger area) that deletes the item after confirmation.

**Frontmatter fields section:**
- All frontmatter fields rendered as labelled key-value pairs in view mode.
- **Title** — displayed prominently (also used in the header).
- **Status** — rendered as a Badge atom with appropriate variant colouring.
- **Created at** — human-readable date.
- **Last modified** — human-readable relative time (e.g., "3 hours ago").
- **Last modified by** — the user email or agent name.
- **Cross-layer references** — listed as a frontmatter field but also rendered as clickable links in the references section (see below).
- Any additional frontmatter fields the item may have are rendered as generic labelled values. The page should handle arbitrary frontmatter keys gracefully — not just the known fields.

**Markdown body section:**
- The markdown body rendered as formatted HTML using the Markdown component from step 010.
- If the body is empty, show a subtle message ("No content yet") rather than a blank area.
- The section has a visual separator or heading to distinguish it from the frontmatter fields.

**Cross-layer references section:**
- Displayed as a distinct section after or alongside the body.
- Split into two groups: forward references (items this item references) and backward references (items that reference this item).
- Each reference is rendered as a clickable link showing: the target layer name (with identity colour), the item title, and the filename. Clicking navigates to that item's detail view (`/layers/:targetLayerSlug/:targetFilename`).
- If there are no references in either direction, the respective group is hidden.
- If there are no references at all, the section shows a subtle empty state message or is hidden entirely.
- The section headings use i18n keys (e.g., `items.referencesForward`, `items.referencesBackward`).

**Attribution:**
- Displayed at the bottom of the page or integrated into the frontmatter section.
- Shows who or what last modified the item (user email for manual edits, agent name for AI edits) and when (relative time + absolute timestamp on hover or as tooltip).

**Loading and error handling:**
- While the item data is being fetched, the page shows LoadingState.
- If the fetch fails (item not found, network error), the page shows ErrorState with a retry button.
- If the layer slug or item slug is invalid, the page shows an appropriate error or redirects to the layer detail view.

### Item Detail Page — Edit Mode

When the user clicks the "Edit" button, the page transitions to edit mode:

**Edit mode entry:**
- The current frontmatter and body values are captured into a local editing buffer (not modifying the store's data until save).
- The "Edit" button in the header changes to "Save" and "Cancel" buttons.
- The `useBeforeUnload` hook activates with `isDirty` as the condition.

**Frontmatter field editing:**
- Each frontmatter field becomes an editable form input wrapped in a FormField molecule.
- **Title** — an Input atom (text).
- **Status** — a Select atom with predefined options (the implementer defines reasonable status values, e.g., "active", "draft", "archived", "deprecated").
- **Cross-layer references** — displayed as a list of current references. Editing references is complex (it requires knowing what items exist in other layers); for the initial implementation, references are displayed as read-only text even in edit mode. The user can manually type reference filenames if needed, but the primary mechanism for creating references is through the AI agents during cascade generation.
- **Created at** and **last modified** — read-only, not editable (these are set automatically by the backend).
- **Last modified by** — read-only (set automatically).
- Any unknown/additional frontmatter fields are displayed as editable text inputs.

**Markdown body editing:**
- The Markdown component is replaced with a Textarea atom (using the largest size variant) for editing the raw markdown.
- The textarea should be tall enough to show a reasonable amount of content (minimum height with resize enabled, or auto-expanding based on content).
- There is no need for a live preview alongside the editor in this step — the user edits raw markdown and sees the rendered result after saving (or toggling back to view mode). A split-pane preview would be a future enhancement.

**Dirty detection:**
- The page tracks whether any field has been modified compared to the original values captured on edit mode entry.
- If the user tries to cancel or navigate away with unsaved changes, the ConfirmModal is shown asking whether to discard changes.
- The dirty state is computed by comparing the current editing buffer against the original buffer (not the store state).

**Save workflow:**
- Clicking "Save" sends `PUT /api/layers/:slug/items/:filename` with the updated frontmatter and body via the authenticated HTTP client.
- The save button shows a loading state while the request is in flight.
- On success: the page transitions back to view mode with the updated data. The editing buffer is cleared. A brief success indicator is shown (e.g., a temporary "Saved" text near the save button, visible for ~2 seconds). The item data in the store/hook is refreshed to reflect the server's response (which includes the auto-set `last_modified` and `last_modified_by` fields).
- On failure: the page stays in edit mode. An error message is displayed near the save button or at the top of the page. The user's edits are preserved so they can retry.
- The `useBeforeUnload` hook deactivates after a successful save (no unsaved changes).

**Cancel workflow:**
- Clicking "Cancel" checks for unsaved changes.
- If dirty: shows the ConfirmModal asking whether to discard changes. On confirm, reverts to view mode with the original data. On cancel, stays in edit mode.
- If not dirty: transitions directly back to view mode.

### Item Deletion

The delete action is available from the page header (in view mode, not edit mode):

- Clicking "Delete" opens the ConfirmModal with a danger variant, asking the user to confirm deletion.
- On confirm: sends `DELETE /api/layers/:slug/items/:filename` via the authenticated HTTP client. Shows a loading state on the confirm button.
- On success: navigates back to the layer detail view (`/layers/:layerSlug`). The layer detail store/view refreshes to reflect the removed item.
- On failure: closes the modal and shows an error message.
