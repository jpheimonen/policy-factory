# 021: Frontend idea inbox

## Context

The idea pipeline backend is complete: the idea mixin (step 018) stores ideas with status tracking, the score mixin stores 6-axis evaluation scores, the evaluation orchestrator scores and critiques ideas, the generation agent produces new ideas, and the ideas router (step 018) exposes endpoints for submission, generation, listing, detail retrieval, archiving, and re-evaluation. The frontend idea store shell (step 013) already receives WebSocket events for real-time status updates — `IdeaSubmitted`, `IdeaEvaluationStarted`, `IdeaEvaluationCompleted`, `IdeaGenerationStarted`, `IdeaGenerationCompleted` — but has no REST-based data fetching, no UI beyond event handlers, and no page to display ideas.

This step builds the idea inbox page — the interface where users submit ideas, trigger AI idea generation, browse the idea list with filtering and sorting, view 6-axis radar charts for evaluated ideas, and drill into full critic assessments and synthesis results. The page also supports archiving and re-evaluation triggers.

The idea inbox is a data-rich page combining three interaction patterns:
1. **Input** — a submission form for human ideas, and a generate button for AI ideas.
2. **List browsing** — a filterable, sortable list of ideas with status badges and score previews.
3. **Detail drill-down** — expanding an idea reveals full scores (radar chart), all 6 critic assessments, and the synthesis result.

The idea store shell from step 013 is extended into a full-featured store with REST-based CRUD, filtering/sorting state, and optimistic updates on submission. The route `/ideas` was registered as a placeholder in step 009; this step replaces it with the real page.

This step introduces recharts `RadarChart` for the 6-axis score visualisation — the first use of recharts in Policy Factory. The cc-runner codebase uses recharts for bar charts in its stats views, establishing the library pattern (ResponsiveContainer wrapping, theme integration, styled-components co-location).

## Deliverables

### Idea Store Extension

Extend the idea store at `ui/src/stores/ideaStore.ts` from its step 013 shell into a full-featured store:

**Additional state:**
- The filtered and sorted idea list as returned by the API. Each idea in the list includes: ID, text (preview-length), source (human/AI), target objective, status, submission timestamp, overall score (if evaluated), submitted by.
- The currently expanded idea's full detail: all fields from the list plus full text, all 6-axis scores, all 6 critic assessments (each with archetype slug, display name, assessment text, structured assessment if available), and the synthesis result.
- Current filter state: selected status filter (all, pending, evaluating, evaluated — archived is a separate view), selected sort field (submission_date or score), sort order (asc/desc).
- Loading flags: list loading, detail loading, submitting, generating.
- Error states for each operation.
- A flag indicating whether AI idea generation is currently in progress (set by the `IdeaGenerationStarted` event handler, cleared by `IdeaGenerationCompleted`).

**REST-based actions:**

- **Fetch ideas** — calls `GET /api/ideas/` with the current filter and sort parameters. Populates the list. Called on page mount and when filters change.
- **Fetch idea detail** — calls `GET /api/ideas/:idea_id`. Populates the expanded idea detail. Called when a user expands an idea.
- **Submit idea** — calls `POST /api/ideas/` with the idea text and optional target objective. Performs an optimistic update: immediately adds the idea to the list with status "pending" and source "human" before the server responds. On server success, updates the optimistically added idea with the real ID. On server failure, removes the optimistic entry and shows an error.
- **Trigger generation** — calls `POST /api/ideas/generate` with optional scoping context. Sets the generating flag. The response is immediate; actual ideas appear in the list via `IdeaSubmitted` WebSocket events as they are created by the backend.
- **Archive idea** — calls `PUT /api/ideas/:idea_id/archive`. Removes the idea from the active list (unless the user is viewing archived ideas).
- **Re-evaluate idea** — calls `POST /api/ideas/:idea_id/evaluate`. The idea's status changes to "pending" and then to "evaluating" via WebSocket events.
- **Set filter** — updates the status filter. Triggers a re-fetch of the idea list with the new filter.
- **Set sort** — updates the sort field and order. Triggers a re-fetch.

**WebSocket event handler enhancements (extending the existing handlers from step 013):**
- `handleIdeaSubmitted` — if the idea was added optimistically (human submission), reconcile by updating the temporary entry with the real data. If it's an AI-generated idea (arriving from generation), add it to the list.
- `handleIdeaEvaluationStarted` — update the idea's status in the list to "evaluating".
- `handleIdeaEvaluationCompleted` — update the idea's status to "evaluated". If this idea is currently expanded, refresh its detail to show the new scores and critic results.
- `handleIdeaGenerationStarted` — set the generating flag to true.
- `handleIdeaGenerationCompleted` — clear the generating flag. Trigger a list refresh to ensure all generated ideas are visible.

### Idea Inbox Page

Replace the placeholder `IdeasPage` at `ui/src/pages/IdeasPage.tsx` (route `/ideas`) with the full idea inbox. The page has a header area, a submission/generation area, filter/sort controls, and the idea list.

**Page header:**
- Page title using the `ideas.title` translation key.
- Summary statistics: total ideas count, count by status (pending, evaluating, evaluated). Fetched from the idea list or computed from the store.

**Submission area:**
- A text area for typing a new idea. Placeholder text using an i18n key suggesting what to submit (e.g., "Describe a policy idea, strategic initiative, or tactical proposal...").
- An optional scoping selector: a dropdown or link that allows the user to scope the idea to a specific strategic or tactical objective. The objectives are fetched from the layer store (strategic objectives and tactical objectives layers). If no scoping is selected, the idea is general.
- A submit button. Disabled when the text area is empty. Shows loading state during submission. On successful submission, clears the text area and the new idea appears in the list (optimistically).

**Generation area:**
- A "Generate ideas" button that triggers AI idea generation.
- An optional scoping control: scope generation to a specific objective or leave unscoped for broad generation.
- While generation is in progress (the generating flag is true), the button shows a loading state with a label indicating generation is running. As ideas are generated, they appear in the list in real-time via WebSocket events.
- When generation completes (flag clears), the button returns to its normal state. A brief notification or status message indicates how many ideas were generated.

**Filter and sort controls:**
- A status filter control (dropdown or button group) with options: All, Pending, Evaluating, Evaluated. An additional "Archived" option shows archived ideas. The default is "All" (which excludes archived).
- A sort control with options: Newest first (submission date desc — default), Oldest first (submission date asc), Highest score (overall score desc), Lowest score (overall score asc). Score-based sorting is only meaningful for evaluated ideas.
- Changing either filter or sort triggers a re-fetch of the idea list.

**Idea list:**
- Displays ideas as a vertical list of cards.
- Each idea card in the collapsed (summary) state shows:
  - The idea text, truncated to a preview length (e.g., 2–3 lines) with ellipsis for overflow.
  - A status badge (pending, evaluating, evaluated, archived) with colour coding: pending = info/blue, evaluating = warning/yellow, evaluated = success/green, archived = muted/grey.
  - A source badge: "Human" or "AI" with distinct styling.
  - The submission timestamp as relative time (e.g., "3 hours ago").
  - The submitter (user email or "System" for AI-generated).
  - If evaluated: the overall score displayed as a numerical value or small visual indicator (e.g., a filled bar or score badge).
  - The target objective reference, if any, displayed as a subtle label linking to the referenced objective.
- Clicking an idea card expands it to reveal the full detail (described below).
- The list handles pagination: if there are more ideas than the initial page size, a "Load more" button or infinite scroll fetches additional ideas using the offset parameter.

**Empty state:**
- When no ideas exist, show an empty state message with guidance: explain that ideas can be submitted manually or generated by AI, and offer the submission form and generate button prominently.

**Loading and error states:**
- While the idea list is loading (initial fetch), show skeleton placeholders.
- If the fetch fails, show an error state with a retry button.

### Expanded Idea Detail

When a user clicks an idea card, it expands inline (within the list) to reveal the full evaluation detail. Only one idea can be expanded at a time — expanding a new idea collapses the previously expanded one.

**Full idea text:**
- The complete, untruncated idea text.

**Score radar chart (only for evaluated ideas):**
- A 6-axis radar chart (recharts `RadarChart`) displaying the idea's scores:
  - Strategic fit
  - Feasibility
  - Cost
  - Risk
  - Public acceptance
  - International impact
- Each axis is labeled using i18n translation keys.
- The chart uses a single data series (the idea's scores) rendered as a filled polygon with a semi-transparent fill colour from the theme.
- The chart is wrapped in a `ResponsiveContainer` for responsive sizing. Target size: approximately 250–300px square, centred within the detail area.
- The axis scale is consistent (e.g., 0–10 or 0–100, matching whatever scale the backend uses).
- Below or beside the chart, display the overall score prominently.
- If scores are not yet available (idea is pending or evaluating), the chart section shows a placeholder message (e.g., "Scores available after evaluation").

**Score axis detail (optional companion to the chart):**
- A list or table showing each of the 6 axis scores as numerical values alongside the chart. This provides exact numbers for users who prefer text over visual.

**Critic assessments (only for evaluated ideas):**
- A section displaying all 6 critic perspectives. Each critic's assessment is rendered in a collapsible sub-section (following the CollapsibleSection pattern from cc-runner):
  - Header: the critic's display name (e.g., "Realist / Security hawk"), collapsed by default.
  - Content when expanded: the critic's assessment text rendered with the Markdown component. If the structured assessment is available (agreement/disagreement, reasoning, alternatives), it is rendered with clear visual formatting. If only raw text is available, it is rendered as-is.
- The 6 critics are displayed in their canonical order (Realist, Liberal-institutionalist, Nationalist-conservative, Social-democratic, Libertarian, Green/Ecological).
- If critics have not run yet, this section shows a placeholder message.

**Synthesis result (only for evaluated ideas):**
- A section below the critic assessments showing the synthesis.
- The synthesis text is rendered with the Markdown component.
- If the structured synthesis is available (consensus points, tension points, recommendations), these are rendered with visual grouping — tension points may be highlighted distinctly since they represent unresolved disagreements.
- If the synthesis has not run yet, a placeholder message is shown.

**Action buttons within the expanded detail:**
- **Archive** — archives the idea. Styled as a muted/secondary action. Only shown for ideas that are not already archived.
- **Re-evaluate** — triggers re-evaluation. Shown for ideas in "pending" or "evaluated" status. Useful after the policy stack has changed. Returns the idea to the evaluation pipeline. The button is disabled (with a tooltip) if the idea is currently being evaluated.
- **Collapse** — a way to close the expanded detail (clicking the header area again, or an explicit close button).

**Behaviour:**
- Accepts the 6 score values and renders a recharts `RadarChart` with `PolarGrid`, `PolarAngleAxis` (for axis labels), `PolarRadiusAxis` (for scale), and a single `Radar` data series.

- The chart renders gracefully when one or more scores are zero (the polygon simply has a vertex at the centre for that axis).

**This component is reusable** — it can be used on the idea inbox page and potentially on any future view that needs to display 6-axis scores.
