# 012: Event system and WebSocket

## Context

Until now, all frontend data has come from REST API polling — the layer store fetches on page mount, the cascade store fetches status on navigation component mount. This is sufficient for the current pages but insufficient for the real-time features coming in later steps: the live cascade viewer (step 020) streams agent reasoning text in real-time, the activity feed (step 022) shows new events as they happen, and the idea inbox (step 021) updates evaluation progress live.

This step builds the backend event infrastructure that powers all of those real-time features: a typed event system with dataclass events, an async EventEmitter for pub/sub decoupling, a WebSocket connection manager with JWT authentication on handshake, and a broadcast handler that bridges the EventEmitter to all connected WebSocket clients.

The event system also persists events to SQLite (via an agent run mixin — step 015 adds the cascade-specific mixin, but the general event storage infrastructure is established here). This enables the REST-based event replay endpoint that the frontend uses to recover missed events after a WebSocket reconnection.

This is pure backend infrastructure — no frontend changes in this step. The frontend WebSocket integration comes in step 013. The backend components built here are:
1. Typed event dataclasses covering all event categories defined in the architecture spec
2. An EventEmitter with async-safe pub/sub
3. A WebSocket ConnectionManager with JWT authentication
4. A broadcast handler that subscribes to the EventEmitter and pushes events to all connected clients
5. Event persistence in SQLite for replay
6. A REST endpoint for replaying events (used by the frontend on WebSocket reconnection)
7. The WebSocket endpoint registered in the FastAPI app

## Reuse Opportunities

- The **EventEmitter class** from cc-runner can be brought into Policy Factory nearly verbatim. The async-safe emit pattern (checking for coroutines and awaiting them) is well-proven and needs no modification.
- The **ConnectionManager** from cc-runner can be used directly with one key addition: JWT authentication on the `connect()` method. cc-runner's ConnectionManager accepts all connections; Policy Factory needs to validate the JWT from the query parameter before accepting.
- The **WebRenderer bridge pattern** from cc-runner (subscribe to EventEmitter → persist to SQLite → broadcast via WebSocket) is directly applicable. Policy Factory's version follows the same architecture with the addition of event IDs in the broadcast payload.
- The **BaseEvent dataclass pattern** (with `to_dict()`, typed `event_type` field, ISO datetime serialisation) is used for all Policy Factory events. The specific event types differ (cascade events, heartbeat events, etc. instead of run/phase/tool events), but the structural pattern is identical.
- The **event storage schema** from cc-runner (events table with run_id, event_type, data JSON, timestamp) is adapted. Policy Factory uses a similar schema but replaces `run_id` with a more flexible identifier since events may not all belong to a "run" (e.g., system events, user actions).
- The **deps.py singleton pattern** from prior steps is extended with new getters for the WebSocket manager and event emitter.

## Deliverables

### Typed Event Dataclasses

A module at `src/policy_factory/events.py` defining all typed event dataclasses:

**Base event:**
- A `BaseEvent` dataclass with fields common to all events: a unique event ID (generated on creation, used for deduplication on the client), a timestamp (datetime, serialised to ISO 8601), and an `event_type` string (set by each subclass, not passed by the caller).
- A `to_dict()` method that returns a JSON-serialisable dictionary of all fields. Datetime fields are converted to ISO 8601 strings.

**Cascade lifecycle events:**
- `CascadeStarted` — cascade ID, trigger source (user input, layer refresh, heartbeat, seed), starting layer slug.
- `CascadeCompleted` — cascade ID, success flag.
- `CascadeFailed` — cascade ID, error message, failed layer slug, failed step (generation/critics/synthesis).
- `CascadePaused` — cascade ID, error message, paused layer slug, paused step.
- `CascadeResumed` — cascade ID.
- `CascadeCancelled` — cascade ID.
- `CascadeQueued` — cascade ID, queue position.

**Layer processing events:**
- `LayerGenerationStarted` — cascade ID, layer slug.
- `LayerGenerationCompleted` — cascade ID, layer slug.
- `CriticStarted` — cascade ID, layer slug, critic archetype name (e.g., "realist", "liberal-institutionalist").
- `CriticCompleted` — cascade ID, layer slug, critic archetype name.
- `SynthesisStarted` — cascade ID, layer slug.
- `SynthesisCompleted` — cascade ID, layer slug.

**Agent streaming events:**
- `AgentTextChunk` — cascade ID (or other context ID), agent label (human-readable, e.g., "Realist critic"), text chunk (the actual reasoning text). These are the high-frequency events that power the live cascade viewer's text streaming.

**Heartbeat events:**
- `HeartbeatStarted` — heartbeat run ID.
- `HeartbeatTierCompleted` — heartbeat run ID, tier number (1–4), outcome description, escalated (boolean).
- `HeartbeatCompleted` — heartbeat run ID, highest tier reached.

**Idea events:**
- `IdeaSubmitted` — idea ID, source (human/AI).
- `IdeaEvaluationStarted` — idea ID.
- `IdeaEvaluationCompleted` — idea ID.
- `IdeaGenerationStarted` — (no specific ID, this is a batch operation).
- `IdeaGenerationCompleted` — count of ideas generated.

**System events:**
- `UserLogin` — user email.
- `UserCreated` — user email, role.
- `CascadeLockAcquired` — cascade ID.
- `CascadeLockReleased` — cascade ID.

All event classes follow the cc-runner pattern: dataclass with typed fields, `event_type` set via `field(default="event_name", init=False)`, and a `to_dict()` method that extends the base dict with the subclass-specific fields.

A union type or literal type listing all valid event type strings should be defined for type safety.

### EventEmitter

An `EventEmitter` class in the same events module (or a separate `emitter.py`):

- Maintains a list of subscribed handlers.
- `subscribe(handler)` — adds a handler function. Handlers can be sync or async callables that accept a single `BaseEvent` argument.
- `unsubscribe(handler)` — removes a handler.
- `emit(event)` — async method that calls all subscribed handlers, awaiting any that return coroutines. Follows the cc-runner pattern of checking `asyncio.iscoroutine()` on the handler result.
- If a handler raises an exception, it should be logged but not propagate to the emitter or other handlers — one broken handler must not stop events from reaching other handlers.

### WebSocket Connection Manager

A module at `src/policy_factory/server/ws.py` adapted from cc-runner's ConnectionManager, extended with JWT authentication:

**Connection lifecycle:**
- `connect(websocket, token)` — validates the JWT token (using the same JWT validation logic from the auth service, step 005). If valid, accepts the WebSocket connection and adds it to the active connections list. If invalid (expired, malformed, missing), rejects the connection by closing the WebSocket with an appropriate close code and reason.
- `disconnect(websocket)` — removes the connection from the active list.

**Broadcasting:**
- `broadcast(event_dict)` — serialises the event dictionary as JSON and sends it to all active connections. If sending to a connection fails (client disconnected), the connection is cleaned up automatically. Follows the cc-runner pattern of collecting disconnected clients during iteration and removing them afterward.
- `send_to(websocket, event_dict)` — sends an event to a specific connection (used for connection-specific messages, e.g., a welcome message on connect).

**No channel/filtering:** All authenticated connections receive all events. There is no per-user or per-cascade filtering at the WebSocket level — the frontend filters events it cares about in its event dispatch logic (step 013).

### Broadcast Handler

A handler (analogous to cc-runner's WebRenderer) that bridges the EventEmitter to persistence and WebSocket broadcasting:

- Created during app startup (in the lifespan function).
- Subscribes to the EventEmitter.
- On each event:
  1. Persists the event to SQLite (using the event store mixin), receiving a database-generated event ID.
  2. Broadcasts the event to all WebSocket clients (via the ConnectionManager), including the database-generated event ID in the payload so the frontend can use it for deduplication and replay tracking.
- The handler is the single subscriber that handles persistence and broadcasting. Other subscribers (if any) can be added for other purposes (e.g., logging).

### Event Storage (SQLite)

An event mixin (or extension to the base store) for event persistence:

**Schema — events table:**
- Auto-incrementing integer ID (primary key, used for deduplication and replay cursor).
- Event type string.
- Event data (JSON string containing the full serialised event).
- Timestamp.
- Optional context fields for efficient querying: a layer slug field (populated from events that involve a specific layer, null for events without layer context) and an event category field (cascade, heartbeat, idea, system — derived from the event type, used for filtering).

**Methods:**
- `add_event(event_type, data, timestamp, layer_slug, category)` — inserts an event row and returns the auto-generated ID.
- `get_events(since_id, event_type, layer_slug, category, limit)` — retrieves events with optional filters. The `since_id` parameter returns only events with ID greater than the given value (used for incremental replay after WebSocket reconnection). Default limit of 100, max 500.
- `get_recent_events(limit, offset, event_type, layer_slug, category)` — retrieves recent events in reverse chronological order with optional filters (used for the activity feed page).

Add the events table creation to the schema initialisation in `init_db()`.

### WebSocket Endpoint

Register a WebSocket endpoint in the FastAPI app:

**Endpoint:** `ws://host/ws?token=<JWT>`

**Connection flow:**
1. The client connects to `/ws` with the JWT as a query parameter named `token`.
2. The server extracts the token from the query string.
3. The server validates the JWT using the same validation logic as `get_current_user` (step 005).
4. If valid: the connection is accepted and added to the ConnectionManager. Optionally, a welcome message is sent to the client confirming the connection.
5. If invalid: the connection is closed immediately with an appropriate WebSocket close code (e.g., 4001 for auth failure).

**Receive loop:**
- After connection, the endpoint enters a receive loop listening for inbound client messages. For now, no inbound message types are needed — the WebSocket is primarily for server-to-client broadcasting. Future steps may add inbound messages (e.g., cascade control commands). Unknown messages are silently ignored.
- The loop handles disconnection exceptions by cleaning up the connection via the ConnectionManager.

**Registration:**
- The WebSocket endpoint is defined directly in the app factory (inside `create_app()`), following the cc-runner pattern of defining the `/ws` route inline rather than in a router module. This is because FastAPI's WebSocket routing works differently from HTTP routers.

### Activity REST Endpoint

A router at `src/policy_factory/server/routers/activity.py` with prefix `/api/activity` and tag `activity`. Requires JWT authentication.

**GET `/api/activity/`** — Returns recent events for the activity feed.
- Accepts optional query parameters: `event_type` (filter by type), `layer` (filter by layer slug), `category` (filter by category: cascade, heartbeat, idea, system), `limit` (default 50, max 200), `offset` (for pagination).
- Returns events in reverse chronological order (newest first).
- Each event in the response includes: the database-generated ID, event type, timestamp, and the full event data object.
- This endpoint serves both the activity feed page (step 022) and the WebSocket reconnection replay. For replay, the frontend passes `since_id` as a query parameter and receives only events newer than that ID (in chronological order, not reverse).

**GET `/api/activity/replay`** — Returns events since a given ID for WebSocket reconnection replay.
- Accepts required query parameter: `since_id` (the last event ID the client received before disconnection).
- Returns events with ID > since_id in chronological order (oldest first, so the client can process them in sequence).
- Limited to 500 events. If more than 500 events were missed, the client should do a full refresh of its stores instead.

## Acceptance Criteria

- [ ] All event categories defined in the architecture spec have corresponding typed dataclass events: cascade lifecycle (7 types), layer processing (6 types), agent streaming (1 type), heartbeat (3 types), ideas (5 types), system (4 types)
- [ ] Each event dataclass has an `event_type` field set automatically (not passed by caller) and a `to_dict()` method returning a JSON-serialisable dictionary
- [ ] Datetime fields in events are serialised to ISO 8601 format
- [ ] The EventEmitter supports subscribing multiple handlers
- [ ] Emitting an event calls all subscribed handlers
- [ ] Async handlers are properly awaited by the emitter
- [ ] Unsubscribing a handler prevents it from receiving further events
- [ ] A failing handler does not prevent other handlers from receiving the event
- [ ] The ConnectionManager validates the JWT from the query parameter on WebSocket connection
- [ ] A WebSocket connection with a valid JWT is accepted and tracked
- [ ] A WebSocket connection with an invalid, expired, or missing JWT is rejected with an appropriate close code
- [ ] Broadcasting sends the event as JSON to all connected clients
- [ ] Broadcasting to zero connected clients does not error
- [ ] A disconnected client is automatically cleaned up during broadcast (no errors from sending to a closed connection)
- [ ] The broadcast handler persists each event to SQLite and includes the database-generated event ID in the WebSocket broadcast payload
- [ ] The events table is created during schema initialisation with appropriate indexes
- [ ] Events can be stored and retrieved from the database
- [ ] Event retrieval supports filtering by since_id, event_type, layer_slug, and category
- [ ] GET `/api/activity/` returns recent events in reverse chronological order with optional filtering and pagination
- [ ] GET `/api/activity/replay?since_id=N` returns events with ID > N in chronological order
- [ ] All activity endpoints require JWT authentication
- [ ] The WebSocket endpoint is registered at `/ws` in the app factory
- [ ] The ConnectionManager, EventEmitter, and broadcast handler are accessible via dependency injection
- [ ] The event emitter is a singleton shared across the application
