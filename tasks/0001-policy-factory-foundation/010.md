# 010: Frontend layer detail view

## Context

The stack overview page (step 009) shows the five-layer policy stack at a glance. Clicking a layer card navigates to `/layers/:layerSlug` — currently a placeholder page. This step replaces that placeholder with the full layer detail view, the primary drill-down page where users inspect a single layer's content.

The layer detail view is a read-heavy page that displays: the narrative summary (README.md rendered as prose) at the top, all items in the layer as cards with frontmatter metadata, incoming feedback memos from layers above with accept/dismiss actions, and an aggregated critic assessment summary when available. A refresh button in the header triggers a layer regeneration (which starts a cascade from that layer).

This step also extends the layer store from step 009 to support fetching items for a specific layer (`GET /api/layers/:slug/items`), the narrative summary (`GET /api/layers/:slug/summary`), and feedback memos (pending memos targeting this layer). Since the feedback memo mixin and endpoints don't exist yet (they arrive in step 017), the memo fetch returns an empty list for now — but the UI component structure for displaying and acting on memos is built in this step so it's ready when the backend catches up.

The page also renders a critic assessment summary section. Since the critic system arrives in steps 015–016, this section initially shows a placeholder or "no assessments yet" state. The layout and component are created now; real data flows in once the critic runner and its API are built.

## Deliverables

### Markdown Rendering Component

A component at `ui/src/components/atoms/Markdown.tsx` (or molecules, depending on complexity) that renders markdown content as styled HTML:

- Accepts a markdown string as input and renders it as formatted HTML.
- Supports standard markdown features: headings, paragraphs, bold, italic, links, lists (ordered and unordered), blockquotes, inline code, and code blocks.
- Supports GitHub-flavored markdown (tables, task lists, strikethrough) via a remark plugin.
- Links open in new tabs when pointing to external URLs.
- The rendered output is styled consistently with the design system — font sizes, line heights, heading weights, link colours, and code block backgrounds all use theme tokens.
- This component is used for the narrative summary on this page and will be reused for item body rendering in step 011.

### Loading, Error, and Empty State Molecules

If not already created in a prior step, create three reusable molecule components:

**LoadingState** — a centered spinner or skeleton with an optional message. Accepts a `compact` prop for inline usage vs. full-page usage.

**ErrorState** — an error icon with a message and an optional retry button. Accepts `message`, `onRetry`, and `compact` props.

**EmptyState** — an icon with a title, optional subtitle, and optional action button. Used when a layer has no items or no narrative summary. Accepts `title`, `subtitle`, `icon`, `actionLabel`, `onAction`, and `compact` props.

These follow the cc-runner molecule patterns and will be reused across all subsequent pages.

### Layer Store Extension

Extend the layer store from step 009 (`ui/src/stores/layerStore.ts`) with per-layer detail data:

**Additional state:**
- The currently viewed layer's items (list of item objects with filename, title, status, last modified date, last modified by).
- The currently viewed layer's narrative summary (markdown string).
- The currently viewed layer's pending feedback memos (list; empty for now until step 017 provides the backend).
- The currently viewed layer's slug (to know which layer's detail data is loaded).
- Loading and error states specific to the detail fetch (separate from the layer listing loading state).

**Additional actions:**
- **Fetch layer detail**: given a layer slug, calls `GET /api/layers/:slug/items` and `GET /api/layers/:slug/summary` in parallel via the authenticated HTTP client. Populates the detail state. Called on mount of the layer detail page and when the slug route param changes.
- **Refresh layer detail**: re-fetches without showing loading state (keeps stale data visible during fetch).
- **Clear layer detail**: resets the detail state when navigating away from the detail view to prevent stale data flash when navigating to a different layer.

The store does not need to cache detail data for multiple layers simultaneously — it only holds detail data for the currently viewed layer. Navigating to a different layer triggers a new fetch.

### Layer Detail Page

Replace the placeholder `LayerDetailPage` from step 009 with a full implementation at `ui/src/pages/LayerDetailPage.tsx`:

**Route:** `/layers/:layerSlug`

**Page header:**
- The layer's display name, styled prominently with the layer's identity colour as an accent (e.g., coloured left border, coloured underline, or coloured icon).
- A back link to the home page (`/`).
- A refresh button in the header's right area. Clicking it triggers a layer regeneration cascade. For now, it calls `POST /api/cascade/trigger` with the layer slug. If the cascade endpoint is not yet available (it arrives in step 017), the button should handle the failure gracefully — show a brief notification that the feature is not yet available, rather than an unhandled error. The button shows a loading state while the request is in flight.

**Narrative summary section:**
- Rendered at the top of the page content area, below the header.
- Uses the Markdown component to render the README.md content as formatted HTML.
- If no narrative summary exists yet (empty string returned by the API), shows a subtle empty state message (e.g., "No summary generated yet") rather than a blank section.
- The section has a visual heading (using the `layers.summary` i18n key) to label it.

**Feedback memos section:**
- Displayed between the narrative summary and the items list.
- Each pending feedback memo is rendered as a dismissable card with: the source layer name, the memo content, and two action buttons — accept and dismiss.
- Accepting a memo would call the feedback memo update endpoint (not yet available — step 017). For now, the accept/dismiss actions can call a placeholder handler that logs the action or shows a "not yet available" notification.
- Dismissed memos are removed from the displayed list immediately (optimistic UI).
- If there are no pending memos, this section is hidden entirely (not shown as an empty section).
- The section has a visual heading (using the `layers.feedbackMemos` i18n key) with a count badge.

**Item cards section:**
- All items in the layer displayed as a vertical list of cards.
- Each item card shows:
  - **Title** — from the frontmatter, prominently displayed.
  - **Status** — as a Badge atom (if the frontmatter has a status field).
  - **Last modified** — human-readable relative time (same formatting approach as step 009 layer cards).
  - **Last modified by** — the user email or agent name that last changed this item.
  - **Reference count** — a subtle indicator of how many cross-layer references this item has (optional; the reference data is available via the references endpoint but fetching it for every item on the list view may be too expensive — the implementer should decide whether to include this or defer it to the item detail view).
- Clicking an item card navigates to `/layers/:layerSlug/:itemSlug` (the item detail view — step 011).
- Items are sorted alphabetically by title by default.
- If the layer has no items, show the EmptyState molecule with an appropriate message.

**Critic assessment summary section:**
- Displayed below the items list.
- This section shows an aggregated view of the most recent critic assessments for this layer (from the last cascade run).
- Since the critic system (steps 015–016) and its API are not yet built, this section initially renders a placeholder state: "No critic assessments available yet" or similar.
- The layout and component are created now so that when critic data becomes available via the API, it can be wired in without restructuring the page.
- The section has a visual heading (using the `layers.criticSummary` i18n key).

**Loading and error handling:**
- While the detail data is being fetched, the page shows the LoadingState molecule.
- If the fetch fails, the page shows the ErrorState molecule with a retry button.
- If the layer slug from the route params is invalid (not one of the 5 known layers), the page shows an error message or redirects to the home page.

**Responsive behaviour:**
- At narrow viewports (below 768px), item cards become full-width with reduced padding.
- The narrative summary section remains readable at all viewport widths.
- The header actions (back link, refresh button) remain accessible on small screens.
