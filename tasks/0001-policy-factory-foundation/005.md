# 005: JWT authentication backend

## Context

Policy Factory is a multi-user system shown to politicians — it needs authentication. cc-runner has no auth at all (single-user local tool), so this is entirely new infrastructure. This step builds the complete backend auth system: JWT token generation/validation, bcrypt password hashing, the auth and users API routers, FastAPI auth dependencies, and WebSocket JWT authentication.

The auth system has a deliberate first-user bootstrap flow: the very first person to register automatically becomes admin. After that, registration is closed — only admins can create new user accounts (credentials shared out-of-band, no email infrastructure). This is appropriate for a small trusted group.

Step 004 created the SQLite auth mixin (user CRUD with pre-hashed passwords). This step adds the auth service layer (hashing, JWT creation/validation), the auth dependencies (for protecting routes), and the two routers (auth endpoints and user management endpoints). The frontend auth flow comes in step 006.

## Deliverables

### Auth Service Module

A new module at `src/policy_factory/auth.py` (or `src/policy_factory/server/auth.py`) providing the auth business logic, separate from both the store layer and the router layer:

**Password hashing:**
- A function to hash a plaintext password using bcrypt, returning the hash string. This is called during registration and user creation.
- A function to verify a plaintext password against a stored bcrypt hash, returning a boolean.

**JWT operations:**
- A function to create a JWT access token. The token payload should include: user ID, email, role, issued-at timestamp, and expiration timestamp. The token is signed with a secret key loaded from an environment variable (`JWT_SECRET_KEY`). Token expiry should default to a reasonable duration (e.g., 24 hours) configurable via environment variable.
- A function to decode and validate a JWT. Returns the token payload if valid. Raises or returns an error for: expired tokens, invalid signatures, malformed tokens, missing required claims.
- The JWT secret key must be loaded from the environment. If not set, the server should fail to start with a clear error message (not silently fall back to a default key, which would be a security vulnerability). For development convenience, the `env.example` file should document this variable.

**PyJWT dependency:**
- Add `pyjwt` to the project dependencies in `pyproject.toml` (if not already added in step 004).

### Auth Dependencies

New FastAPI dependencies added to the dependency injection module:

**`get_current_user`:**
- Extracts the JWT from the `Authorization: Bearer <token>` header.
- Decodes and validates the token.
- Looks up the user in the database by the user ID from the token (to ensure the user still exists — a deleted user's token should not be valid).
- Returns the user record (without the hashed password).
- Raises HTTP 401 if: no Authorization header, malformed header (not "Bearer" scheme), invalid/expired token, user not found in database.

**`require_admin`:**
- Depends on `get_current_user` (chains the dependency).
- Checks that the returned user has the "admin" role.
- Raises HTTP 403 if the user is not an admin.
- Returns the user record (so the admin's identity is available to the endpoint).

### Auth Router

A router at `src/policy_factory/server/routers/auth.py` with prefix `/api/auth`:

**POST `/api/auth/login`:**
- Accepts email and password in the request body.
- Looks up the user by email. If not found, returns 401 with a generic error message (not "user not found" — don't reveal whether an email is registered).
- Verifies the password against the stored hash. If wrong, returns 401 with the same generic error message.
- On success, creates a JWT and returns it along with the user's basic info (id, email, role). Does not return the password hash.

**POST `/api/auth/register`:**
- Accepts email and password in the request body.
- **First-user flow**: if no users exist in the database (count_users returns 0), the registration is open to anyone. The first user is automatically assigned the "admin" role. Returns a JWT so the user is immediately logged in.
- **Subsequent registrations**: requires a valid admin JWT (uses the `require_admin` dependency). The new user is assigned the "user" role. Returns the created user info (not a JWT — the admin creates accounts for others, who then log in separately).
- Validates the email format (basic validation — contains @, not empty).
- Validates the password (minimum length, not empty).
- Checks if the email is already registered. If so, returns 409 (conflict).
- Hashes the password with bcrypt before storing.

**POST `/api/auth/refresh`:**
- Requires a valid (not yet expired) JWT via the `get_current_user` dependency.
- Creates and returns a new JWT with a fresh expiry time.
- This extends the user's session without requiring re-login.

### Users Router

A router at `src/policy_factory/server/routers/users.py` with prefix `/api/users`:

All endpoints in this router require the `require_admin` dependency — only admins can manage users.

**GET `/api/users/`:**
- Returns the list of all users (id, email, role, created_at). Password hashes are never included.

**POST `/api/users/`:**
- Accepts email and password in the request body.
- Creates a new user with the "user" role (admin-created accounts are always regular users — there's no way to create additional admins via this endpoint, which is intentional for security).
- Validates email format and password requirements.
- Checks for duplicate email. Returns 409 if already registered.
- Hashes the password before storing.
- Returns the created user info.

**DELETE `/api/users/:id`:**
- Deletes the user with the given ID.
- An admin cannot delete themselves — returns 400 with a clear error message if the admin's own user ID is passed.
- Returns 404 if the user ID doesn't exist.
- Returns 204 on successful deletion.

### WebSocket Authentication

Update the WebSocket endpoint in `app.py` to require JWT authentication:

- The client passes the JWT as a query parameter on the connection URL (e.g., `/ws?token=<jwt>`).
- During the WebSocket handshake, the server extracts the token from the query parameter, decodes and validates it, and verifies the user exists in the database.
- If the token is missing, invalid, expired, or the user doesn't exist, the server rejects the connection (closes the WebSocket with an appropriate close code before entering the receive loop).
- If the token is valid, the connection is accepted and tracked by the ConnectionManager as it was before.
- The authenticated user identity is available for future use (e.g., logging which user is connected) but for now all authenticated users see all broadcasts — no per-user filtering.

### App Factory Update

Update `create_app()` to:
- Include the auth router and users router in the registered routers.
- Update the WebSocket endpoint with JWT authentication.

### Environment Configuration

Update `env.example` to document:
- `JWT_SECRET_KEY` — the secret key for signing JWTs. Must be set in production. Should be a long random string.
- `JWT_EXPIRY_HOURS` — optional, defaults to 24. How long JWT tokens are valid.

## Acceptance Criteria

- [ ] Login with correct email and password returns a valid JWT containing user ID, email, and role
- [ ] Login with incorrect password returns 401 with a generic error message
- [ ] Login with non-existent email returns 401 with the same generic error message (no email enumeration)
- [ ] The JWT contains an expiry time and can be decoded to extract user claims
- [ ] An expired JWT is rejected by the `get_current_user` dependency with 401
- [ ] A malformed JWT (invalid signature, corrupted payload) is rejected with 401
- [ ] The `get_current_user` dependency extracts the user from a valid `Authorization: Bearer <token>` header
- [ ] The `get_current_user` dependency returns 401 when no Authorization header is present
- [ ] The `require_admin` dependency allows admin users through and returns their identity
- [ ] The `require_admin` dependency rejects non-admin users with 403
- [ ] A deleted user's still-valid JWT is rejected (user lookup fails after token decode)
- [ ] The first user to register is automatically assigned the admin role
- [ ] The first user registration returns a JWT so they are immediately logged in
- [ ] Registration after the first user, without auth, returns 403
- [ ] Registration after the first user, with a non-admin JWT, returns 403
- [ ] Registration after the first user, with an admin JWT, creates a regular user
- [ ] Duplicate email registration returns 409
- [ ] Token refresh with a valid token returns a new JWT with a fresh expiry
- [ ] Token refresh with an expired token returns 401
- [ ] GET `/api/users/` returns the user list when called by an admin (without password hashes)
- [ ] GET `/api/users/` returns 403 when called by a non-admin
- [ ] POST `/api/users/` creates a new user when called by an admin
- [ ] POST `/api/users/` returns 403 when called by a non-admin
- [ ] DELETE `/api/users/:id` deletes the user when called by an admin
- [ ] DELETE `/api/users/:id` returns 403 when called by a non-admin
- [ ] An admin cannot delete themselves (returns 400)
- [ ] DELETE `/api/users/:id` returns 404 for a non-existent user
- [ ] A WebSocket connection with a valid JWT query parameter is accepted
- [ ] A WebSocket connection with an invalid or expired JWT is rejected
- [ ] A WebSocket connection with no token query parameter is rejected
- [ ] The server fails to start if `JWT_SECRET_KEY` environment variable is not set
- [ ] `pyjwt` is listed as a dependency in `pyproject.toml`
- [ ] Passwords are hashed with bcrypt before storage — plaintext passwords are never persisted
- [ ] The health endpoint (`/api/health/check`) remains accessible without authentication
