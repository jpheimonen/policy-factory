# 001: Project scaffold and dev tooling

## Context

This is the very first step — nothing exists yet. We need to establish the complete project structure, build system, dev workflow, and minimal runnable application before any features can be built. Everything in subsequent steps depends on this foundation being solid.

Policy Factory follows the cc-runner architectural pattern: a Python backend package serving both a REST API and built frontend assets from a single process, with a Vite+React frontend that proxies to the backend during development. The Makefile orchestrates the dev workflow with automatic port detection so multiple instances can coexist.

This step delivers: a Python package that starts a FastAPI server with a health endpoint, a React app that renders a blank shell, `make dev` that launches both concurrently with working proxy, and all the config plumbing (pyproject.toml, vite config, tsconfig, .env loading).

## Deliverables

### Backend Python Package

- A Python package at `src/policy_factory/` with an `__init__.py`
- A CLI entry point module that:
  - Loads .env files from multiple locations in order of preference: current working directory, project root (development mode), and user config directory (`~/.config/policy-factory/.env`). First found wins.
  - Provides a `server` subcommand with `--host` (default `127.0.0.1`) and `--port` (default `8765`) arguments
  - When port is not explicitly specified, auto-detects an available port starting from the default, trying up to 20 ports
  - When port is explicitly specified, fails with a clear error if the port is busy
  - Initializes dependencies and starts uvicorn
  - The `cli()` function serves as the `console_scripts` entry point
- A port utilities module with `is_port_available(host, port)` and `find_available_port(host, start_port, max_attempts)` functions
- A FastAPI app factory function `create_app()` that:
  - Uses FastAPI lifespan for startup initialization
  - Registers all API routers
  - Defines an SPA fallback catch-all route that serves `index.html` for non-API paths, and serves static assets for exact file matches (favicons, JS/CSS bundles)
  - Mounts the static assets directory from the built frontend
  - Uses `importlib.resources.files()` to locate static assets relative to the package (works both in editable installs and wheel installs)
- A dependency injection module with module-level singletons, an `init_deps()` function called during lifespan startup, and getter functions for each dependency. Initially just a placeholder store and ws_manager.
- A server routers package with a single health router:
  - Prefix `/api/health`
  - A GET endpoint at `/check` that returns a basic health status response (application name, status ok)
- An empty `static/dist/` directory placeholder within the package (will be populated by the frontend build)

### Dev Workflow (Makefile)

- `make dev` (aliased to `make run`):
  - Finds an available port by running the port utility via `uv run python -c "..."` one-liner
  - Prints the port being used
  - Launches the backend (via `uv run policy-factory server --port $PORT`) and frontend (via `cd ui && VITE_BACKEND_PORT=$PORT bun run dev`) concurrently
  - `trap 'kill 0' EXIT` ensures both processes are killed when either exits or the user hits Ctrl+C
- `make build`:
  - Installs frontend dependencies with `bun install`
  - Runs the frontend production build with `bun run build` (which triggers `tsc -b && vite build`)
  - The build output goes into `src/policy_factory/static/dist/`
- `make clean`:
  - Removes `ui/node_modules` and `src/policy_factory/static/dist`
- `make install`:
  - Runs `make build` then `uv pip install -e .` for local editable install
