# 013: Conversation event dispatch

## Context

The frontend event dispatch system needs to route the new conversation events to the conversation store. This connects the WebSocket event stream to the Zustand store handlers, enabling real-time streaming updates in the UI. The dispatcher also handles cross-store side effects, such as triggering layer content refresh when the AI edits files.

## Related Code

Existing event dispatch:

- `ui/src/hooks/useEventDispatch.ts` — Central dispatcher with switch statement routing events to stores by event_type, uses getState() to access store actions without subscribing, handles cross-store side effects for completion events

Event types (from step 006):

- `ui/src/types/events.ts` — TypeScript event interfaces including new conversation events

Stores involved:

- `ui/src/stores/conversationStore.ts` — (from step 012) conversation store with handler methods
- `ui/src/stores/layerStore.ts` — layer store with refresh() method for reloading content
- `ui/src/stores/activityStore.ts` — activity store that logs most events

## Reuse Opportunities

- Follow the exact switch case pattern used for other event types
- Use getState() access pattern to avoid subscription overhead
- Follow the cross-store side effect pattern (like cascade_completed triggering layer refresh)
- Follow the activity store logging pattern, skipping high-frequency events

## Deliverables

### Event Type Cases

Add switch cases to useEventDispatch for all conversation event types:

**conversation_started**
- Route to conversationStore.handleConversationStarted(event)
- Log to activity store

**conversation_text_chunk**
- Route to conversationStore.handleConversationTextChunk(event)
- Skip activity store logging (high-frequency streaming event, same as agent_text_chunk)

**conversation_file_edit**
- Route to conversationStore.handleConversationFileEdit(event)
- Log to activity store
- Trigger cross-store side effect: call layerStore.refresh() to reload content that may have changed

**conversation_turn_complete**
- Route to conversationStore.handleConversationTurnComplete(event)
- Log to activity store
- Trigger cross-store side effect: call layerStore.refresh() to ensure UI shows any file changes

**conversation_cascade_pending**
- Route to conversationStore.handleConversationCascadePending(event)
- Log to activity store

**conversation_turn_error**
- Route to conversationStore.handleConversationTurnError(event)
- Log to activity store

### Cross-Store Side Effects

When conversation events indicate file changes occurred:
- conversation_file_edit: trigger layerStore.refresh() to reload layer/item content
- conversation_turn_complete: trigger layerStore.refresh() since files may have been edited

This ensures the item/layer detail views update when the AI makes edits during conversation, without requiring manual refresh.

### Activity Store Integration

Log conversation events to the activity store following the existing pattern:
- Include all conversation events except conversation_text_chunk
- conversation_text_chunk is skipped to avoid flooding the activity feed with streaming fragments (same rationale as agent_text_chunk)
