# 017: Backend integration tests

## Context

The conversation feature, philosophy layer, and values prompt fix all require integration tests to verify end-to-end behavior. Integration tests use mocked agent sessions to avoid external API calls while still exercising the full code path from HTTP request through database operations and event emission.

## Related Code

Existing integration test patterns:

- `tests/test_integration_cascade.py` — Pattern for testing long-running processes with mocked agent runners, event emission, and status verification
- `tests/test_integration_ideas.py` — Pattern for testing API endpoints with authenticated clients, database persistence, and background task triggering
- `tests/test_integration_websocket.py` — Pattern for testing WebSocket event broadcasting and client subscriptions
- `tests/conftest.py` — Shared fixtures including store, data_dir, auth_headers, integration_client, mock agent factories

New components to test (from previous steps):

### Test Classes

**TestConversationAPI**

Tests for the REST API endpoints:
- POST /api/conversations creates conversation with valid layer_slug
- POST /api/conversations returns 400 for invalid layer_slug
- POST /api/conversations requires authentication (401 without token)
- GET /api/conversations?layer_slug=X returns conversations for that layer
- GET /api/conversations?layer_slug=X&filename=Y returns item-level conversations
- GET /api/conversations/{id} returns conversation with messages
- GET /api/conversations/{id} returns 404 for non-existent ID
- POST /api/conversations/{id}/messages creates user message
- POST /api/conversations/{id}/messages returns 202 (accepted for processing)
- DELETE /api/conversations/{id} removes conversation and all messages
- DELETE /api/conversations/{id} returns 404 for non-existent ID

**TestConversationTurnWithMockedAgent**

Tests for the conversation runner with mocked agent:
- Full turn stores both user message and assistant response
- Agent file write triggers ConversationFileEdit event
- Agent file delete triggers ConversationFileEdit event with action="deleted"
- Multiple file edits in single turn are all recorded in files_edited
- Assistant message includes files_edited metadata
- Git commit is created after turn with file edits
- Git commit message includes conversation context (layer, item)
- Agent timeout results in error message stored

**TestConversationCascadeIntegration**

Tests for cascade triggering from conversations:
- Edit to philosophy layer creates pending cascade with starting_layer="philosophy"
- Edit to values layer creates pending cascade with starting_layer="values"
- Edit to situational-awareness creates pending cascade
- Edit to strategic-objectives does NOT create pending cascade
- Edit to tactical-objectives does NOT create pending cascade
- Edit to policies does NOT create pending cascade
- Multiple edits to same foundational layer do not create duplicate pending cascades
- Edit to lower foundational layer updates pending cascade starting_layer
- POST /api/cascade/trigger-pending starts cascade from pending record
- POST /api/cascade/dismiss-pending clears pending without running cascade
- Pending cascade persists across multiple conversation turns

**TestPhilosophyLayerInCascade**

Tests for philosophy layer cascade integration:
- Full cascade from philosophy processes all 6 layers in order
- Philosophy content appears in context when generating values
- Cascade from values does NOT include philosophy in processing (only layers below starting point)
- Critics evaluate philosophy layer with all 6 ideological perspectives
- Synthesis is generated for philosophy layer

**TestConversationWebSocketEvents**

Tests for WebSocket event broadcasting:
- ConversationStarted event broadcasts with correct conversation_id
- ConversationTextChunk events stream during agent response
- ConversationFileEdit events broadcast for each file operation
- ConversationTurnComplete event broadcasts when turn finishes
- ConversationCascadePending event broadcasts for foundational edits
- Events include correct db_id for frontend deduplication
- Events are only broadcast to subscribed clients

**TestValuesPromptFix**

Verification that updated values prompt produces correct output format:
- Mock agent with values prompt generates items with tension-pair format
- Generated values have "X vs. Y" style titles (not single-topic)
- Generated values include arguments from opposing perspectives

### Fixtures for Conversation Tests

**conversation_client fixture**
- Pre-authenticated TestClient with conversation-specific setup
- Creates initial test conversations for retrieval tests

**mock_conversation_agent fixture**
- Configured mock that captures tool calls (file reads/writes)
- Returns deterministic responses
- Can be parameterized to simulate different behaviors

**captured_events fixture**
- EventEmitter wrapper that records all emitted events
- Provides assertion helpers for checking event sequences

### Test Isolation

Each test should:
- Use fresh temporary database (via store fixture)
- Use fresh temporary data directory (via data_dir fixture)
- Not depend on state from other tests
- Clean up any side effects
