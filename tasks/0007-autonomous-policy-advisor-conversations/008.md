# 008: Cascade pending mechanism

## Context

When conversation edits affect foundational layers (philosophy, values, situational-awareness), a cascade should be queued to regenerate all layers above. However, unlike the normal cascade queue which auto-processes when the previous cascade completes, conversation-triggered cascades should wait for explicit user approval.

This "pending cascade" mechanism shows a notification to the user and persists until they either trigger the cascade or dismiss it. Multiple conversation edits to foundational layers should update the existing pending cascade rather than creating duplicates.

This step must come before the conversation runner (step 009) because the runner calls the store methods defined here.

- The trigger_cascade() function can be reused to actually start the cascade when the user approves

- The QueueEntry dataclass pattern can be followed for the pending cascade entry

## Deliverables

### Database Storage

Add a pending_conversation_cascade table to schema.py:
- id: unique identifier
- conversation_id: which conversation triggered this (for UI context)
- starting_layer: the lowest affected foundational layer
- affected_layers: JSON list of all foundational layers that were edited
- created_at: when the pending cascade was created
- updated_at: when it was last updated (for subsequent edits)

### Store Methods

Add to CascadeStoreMixin (or create new ConversationCascadeStoreMixin):

**create_or_update_pending_cascade**
- If no pending cascade exists, create one with the given starting_layer
- If a pending cascade already exists, update it:
  - If the new starting_layer is lower in hierarchy than existing, update starting_layer
  - Add new affected layers to the affected_layers list
  - Update the updated_at timestamp
- Return the pending cascade record

**get_pending_cascade**
- Return the current pending cascade if one exists, None otherwise
- Only one pending cascade should exist at a time (single-user system)

**clear_pending_cascade**
- Delete the pending cascade record (used after trigger or dismiss)

**get_pending_cascade_by_conversation**
- Return pending cascade associated with a specific conversation_id

### API Endpoints (Added to Existing Cascade Router)

Add new endpoints to the existing `routers/cascade.py` file (NOT to the conversation router):

**POST /api/cascade/trigger-pending**
- Retrieve the pending cascade record
- Clear the pending cascade record
- Call trigger_cascade() with the stored starting_layer
- Return the new cascade_id
- If no pending cascade exists, return 404

**DELETE /api/cascade/pending**
- Clear the pending cascade record without triggering
- Return 204 No Content for success (idempotent — returns 204 even if no pending cascade exists)

### Event Emission

When create_or_update_pending_cascade creates or updates the record:
- Emit ConversationCascadePending event with conversation_id and starting_layer
- This event is already defined in step 006

### Behavior Differences from Normal Queue

The pending cascade mechanism differs from the normal cascade queue:
- Normal queue: auto-dequeues and starts when previous cascade completes
- Pending cascade: persists until explicit user action (trigger or dismiss)
- Normal queue: can have multiple entries
- Pending cascade: only one at a time, subsequent edits update it
- Normal queue: processed in FIFO order
- Pending cascade: single record, always the current pending work
