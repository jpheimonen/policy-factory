# 009: Conversation runner

## Context

The conversation runner is the core orchestration module for processing a single conversation turn. It assembles the complete prompt (system prompt, full stack context, conversation history, user message), executes the agent, detects file edits from tool calls, commits changes to git, and determines whether to queue a pending cascade.

This is the heart of the conversation feature — it bridges the REST API, the agent system, the event system, and the cascade integration.

This step depends on step 008 (cascade pending mechanism) for the store methods that track pending cascades.

## Deliverables

### New Module

Create `src/policy_factory/conversation/runner.py` with a conversation turn runner.

### Public Function Signature

The main entry point called by the API layer (step 011):

**run_conversation_turn**
- conversation_id: str — which conversation this turn belongs to
- user_content: str — the user's message text
- store: PolicyStore — for message persistence and cascade pending
- emitter: EventEmitter — for streaming events
- data_dir: Path — the policy data directory for file operations

This function is async and returns None (results are streamed via events and persisted to the store).

### Context Assembly

Implement a function to gather full policy stack context:
- Read all layers from bottom (philosophy) to top (policies)
- For each layer, include the README.md narrative and all item content
- Format consistently with existing layer context patterns

### Prompt Assembly

Implement prompt assembly that combines:
- The conversation system prompt (loaded from prompts/conversation/system.md)
- Full policy stack context
- Conversation history (all previous messages in this conversation)
- The current user message
- If discussing a specific item, include that item's content prominently
- If discussing a layer, include that layer's summary prominently

### Agent Execution

Execute the conversation turn:
- Create AgentSession with role="conversation" and appropriate config
- Pass the assembled prompt to session.run()
- Stream AgentTextChunk events via EventEmitter during execution (reusing existing streaming pattern but with conversation event types)
- Emit ConversationStarted event when turn begins
- Emit ConversationTextChunk events during streaming
- Emit ConversationTurnComplete or ConversationTurnError when finished

### File Edit Detection

After agent execution completes:
- Inspect the AgentResult for tool calls that modified files (write_file, delete_file)
- Extract the list of affected files (layer_slug/filename paths)
- Emit ConversationFileEdit event for each file modification
- Include the files_edited list in the stored assistant message

### Git Commit

If any files were edited:
- Call commit_changes with a message indicating conversation context
- Commit message should reference the conversation ID and summarize the action

### Cascade Queueing

After git commit, check if any edited files belong to foundational layers:
- Use FOUNDATIONAL_LAYERS constant from layers.py (defined in step 001)
- If foundational layer was edited, call create_or_update_pending_cascade (from step 008)
- The store method handles event emission (ConversationCascadePending)

### Message Persistence

Throughout the turn:
- Store the user message when the turn starts
- Store the assistant message when the turn completes, including files_edited metadata
- Update conversation's last_active_at timestamp

### Error Handling

Handle failures gracefully:
- If agent times out or errors, store partial assistant message with error flag
- Emit ConversationTurnError event with error details
- Do not commit partial file changes (files may be in inconsistent state)
- Allow retry from the API layer

### Context Truncation

If the assembled prompt exceeds 150,000 tokens:
- Truncate oldest messages from conversation history until under 120,000 tokens
- Keep the most recent messages intact
- Log a warning about truncation
- If stack content alone exceeds limits, this is a fatal error (cannot truncate stack)
