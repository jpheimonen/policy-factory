# 005: Upper-layer seed endpoints

## Context

This step adds three new POST endpoints for seeding the upper layers: strategic objectives, tactical objectives, and policies. Each endpoint follows the same flow as the existing SA seed endpoint but with two key differences: (1) prerequisite validation — all layers below the target must have at least one item before seeding is allowed, and (2) no cascade trigger after completion — seeded content is independent, not a cascade step.

Like the SA seed, the upper-layer seed agents use FILE_TOOLS to write markdown files directly to the data directory. The server clears existing items before running the agent, then commits the agent-written files to git after completion. Unlike the SA seed, these endpoints do not accept optional human context or trigger a cascade.

This step depends on all four preceding steps:
- Step 001 — Agent roles `strategic-seed`, `tactical-seed`, `policies-seed` are registered
- Step 002 — `gather_context_below` and `check_prerequisites` utilities are available
- Step 003 — Seed status endpoint covers all 5 layers (not strictly required but logically prior)
- Step 004 — Prompt templates `strategic.md`, `tactical.md`, `policies.md` exist in `prompts/seed/`

## Reuse Opportunities

- The `SeedResponse` model already has the right shape: `success`, `cascade_id` (will be `None` for upper seeds since no cascade is triggered), and `message`. No new response model needed.

- The three new endpoints are nearly identical — they differ only in target layer slug, agent role name, prompt template name, and human-readable label. A shared private helper function should extract the common flow (prerequisite check → clear → gather context → build prompt → run agent → commit). Each endpoint becomes a thin wrapper that passes its specific parameters. This is the same DRY pattern the codebase uses elsewhere.

- The `mock_agent_session` and `mock_agent_session_with_side_effect` context managers in `test_integration_seeding.py` (lines 183–222) patch `AgentSession` at the source module level, which works for all agent-based endpoints regardless of where they're defined.

- The `_make_mock_sa_agent_result` helper (line 140) demonstrates the pattern for mocking FILE_TOOLS agents: write files to the data directory as a side effect, then return an `AgentResult`. A similar mock helper is needed for the upper-layer seeds, parameterized by target layer slug.

- The existing test fixtures (`data_dir`, `store`, `emitter`, `client`, `auth_headers`) in both test files are reusable without modification.

## Deliverables

### Three new POST endpoints

Add three endpoints to the seed router, all under the `/api/seed/` prefix:

| URL path | Target layer slug | Agent role | Prompt template name | Agent label |
|----------|-------------------|------------|---------------------|-------------|
| `/api/seed/strategic-objectives` | `strategic-objectives` | `strategic-seed` | `strategic` | `Strategic Objectives seed agent` |
| `/api/seed/tactical-objectives` | `tactical-objectives` | `tactical-seed` | `tactical` | `Tactical Objectives seed agent` |
| `/api/seed/policies` | `policies` | `policies-seed` | `policies` | `Policies seed agent` |

Each endpoint requires authentication (same `get_current_user` dependency as existing endpoints). None accept a request body. All return `SeedResponse`.

**Endpoint behavior (same for all three):**

1. **Validate prerequisites** — Check that all layers below the target have at least one item. If any prerequisite layer is empty, return a failure response listing the empty layers. Do NOT proceed to clear items or run the agent.

2. **Clear existing items** — Delete all existing items in the target layer directory. Log warnings for any deletion failures but continue.

3. **Gather context from layers below** — Use the shared utility from step 002 to collect formatted content from all layers below the target.

4. **Build the prompt** — Call `build_agent_prompt("seed", template_name, current_date=current_date, context_below=gathered_context)` where `template_name` is the prompt file name from the table above (e.g., `"strategic"`). The keyword arguments `current_date` and `context_below` must match the `{current_date}` and `{context_below}` placeholders defined in the prompt templates from step 004.

5. **Record agent run** — Create an agent run record in the store with no cascade ID and the target layer slug.

6. **Execute the agent** — Run the agent session and handle the result. On agent failure, complete the agent run record with the error and return a failure response.

7. **Commit to git** — Commit the agent-written files with a descriptive message naming the layer. Log warnings on commit failure.

8. **Return success** — Return a success response with a descriptive message. Do NOT set `cascade_id` (leave as `None`).

### Shared helper function

Extract the common logic of steps 1–8 into a private async helper that takes the target layer slug, agent role, template name, and label as parameters. Each of the three endpoint functions becomes a thin wrapper calling this helper with the appropriate arguments. This avoids tripling ~60 lines of nearly identical code.

### Unit tests in `test_seed_api.py`

Add tests for all three endpoints. For each endpoint:

- **Authentication required** — POST without auth returns 401.
- **Prerequisite failure returns error** — With empty prerequisite layers, the endpoint returns a failure response naming the missing layers.
- **Prerequisite failure does not clear target layer** — When prerequisites fail, any existing items in the target layer must remain untouched.

Prerequisite coverage specifics:
- Strategic objectives requires values AND SA to be populated (2 prerequisite layers)
- Tactical objectives requires values, SA, AND strategic objectives (3 prerequisite layers)
- Policies requires all 4 layers below (values, SA, strategic, tactical)

Test at least one case per endpoint where only some prerequisites are met (e.g., values populated but SA empty → strategic seed should still fail).

### Integration tests in `test_integration_seeding.py`

Add a mock helper that creates files in a target layer directory as a side effect (same pattern as `_make_mock_sa_agent_result` but parameterized by layer slug).

Add a helper that populates all prerequisite layers for a given target layer by writing a minimal markdown item to each layer directory below the target.

Add integration test classes — one per endpoint or a shared parameterized class:

**For each endpoint, test:**

- **Creates items in the correct directory** — Populate prerequisites, mock agent to write files, verify items exist in the target layer directory after the endpoint returns.
- **Clears existing items before seeding** — Create items in the target layer directory, run the seed endpoint (with prerequisites met), verify old items are removed.
- **Commits to git** — Count git commits before and after seeding, verify a new commit was made with a message referencing the layer name.
- **Records agent run** — Verify an agent run record exists in the store after seeding, with the correct agent type and target layer.
- **Returns error on agent failure** — Mock a failing agent, verify the endpoint returns a failure response with an error message.
- **Does NOT trigger cascade** — Mock `trigger_cascade` and verify it is NOT called by any of the three new endpoints. This is a critical behavioral difference from the SA seed.
- **Prerequisite failure does not clear items** — With missing prerequisites, verify the endpoint returns a failure response AND existing items in the target layer are preserved.

**Additional per-endpoint tests:**

- **Tactical seed fails when only values and SA populated** — Strategic objectives empty should cause failure even though values and SA are fine.
- **Policies seed fails when any single prerequisite is empty** — E.g., values, SA, and strategic are populated but tactical is empty → policies seed should fail.
