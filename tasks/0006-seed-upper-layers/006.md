# 006: Admin page seed card redesign

## Context

The admin page's seed status card currently shows only 2 status items (values and SA) with 3 buttons in a flat action bar (Seed Values, Seed SA Layer, Full Cascade). Now that the API returns status for all 5 layers (step 003) and endpoints exist for seeding all 5 (steps 004ŌĆō005), the card needs to show all 5 layers with per-layer seed buttons and prerequisite-based disabling.

The card changes from a flat layout to a vertical layer list. Each row shows the layer's seeded/not-seeded status and a seed button. The UI enforces seeding order: upper-layer seed buttons are disabled until all layers below are populated. The Full Cascade button remains at the bottom of the card, outside the per-layer rows.

This step depends on:
- Step 003 ŌĆö The seed status API returns a list-based response with 5 layer entries
- Step 005 ŌĆö The seed endpoints for all 5 layers exist

### API path mapping

Each layer maps to a specific seed endpoint. The values and SA paths are pre-existing; the three upper-layer paths come from step 005:

| Layer slug | Seed endpoint path |
|-----------|-------------------|
| `values` | `/api/seed/values` |
| `situational-awareness` | `/api/seed/` |
| `strategic-objectives` | `/api/seed/strategic-objectives` |
| `tactical-objectives` | `/api/seed/tactical-objectives` |
| `policies` | `/api/seed/policies` |

This mapping must be maintained in the component ŌĆö either as a constant lookup object or derived from the slug with a special case for `situational-awareness` (which uses `/api/seed/` rather than `/api/seed/situational-awareness`).

## Deliverables

### Updated TypeScript type for seed status

Replace the current flat `SeedStatus` interface with one matching the new list-based API response from step 003. The new type must represent a list of layer entries, each with slug, display name, seeded boolean, and item count.

### Consolidated loading state

Replace the two individual boolean loading states (`seedValuesLoading`, `seedSaLoading`) with a single structure that tracks loading state per layer slug. This supports 5 layers without adding 5 separate `useState` calls. The component also needs a derived value indicating whether ANY seed operation is currently running, used to disable all other seed buttons during an operation.

### Generic seed handler

Replace the two individual handlers with a single function that takes a layer slug and API path. The behavior is identical to the existing handlers: set the layer's loading flag, POST to the API, schedule a status refresh after 1 second, and clear the loading flag in the finally block. The Full Cascade handler (`handleFullCascade`) remains separate and unchanged since it calls a different API endpoint and has different behavior.

### Vertical layer list in the seed card

Replace the seed card body with a list that iterates over the layers from the seed status response and renders a row for each. Each row displays:

- **Left side**: A status dot (green when seeded, grey when not), the layer's display name from the API, and the item count when seeded (e.g., "(3 items)").
- **Right side**: A seed button for that layer.

The rows appear in the same order as the API response (hierarchical: values at top, policies at bottom).

**Prerequisite logic for button disabling**: A layer at position N in the list has its prerequisites met when every layer at positions 0 through N-1 all have `count > 0`. This is derived directly from the layers array ŌĆö no separate API call needed. Specifically:
- Values (position 0): always enabled ŌĆö no prerequisites
- Situational Awareness (position 1): always enabled ŌĆö no prerequisites. The existing SA seed endpoint gathers values content when available but handles its absence gracefully with a fallback string, so values is not a hard prerequisite. This matches the backend behavior ŌĆö the SA seed endpoint has no prerequisite validation.
- Strategic Objectives (position 2): enabled only when positions 0 and 1 both have count > 0
- Tactical Objectives (position 3): enabled only when positions 0, 1, and 2 all have count > 0
- Policies (position 4): enabled only when positions 0, 1, 2, and 3 all have count > 0

**Button disabled conditions**: A seed button is disabled when EITHER:
1. The layer's prerequisites are not met (any required layer below has count 0), OR
2. Any seed operation is currently running (any entry in the loading map is true)

**Button text**: Shows a loading message when that specific layer is being seeded, otherwise shows a "Seed {layer}" label using the display name.

The Full Cascade button remains below the layer rows, visually separated. Its behavior is unchanged from the current implementation.

### New styled components

Follow the existing styling conventions: use theme tokens for spacing, colors, fonts, and border radii. Match the visual density of the existing `StatusItem` component.
