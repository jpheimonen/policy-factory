# 007: E2E tests

## Context

This step adds Playwright E2E tests verifying the redesigned seed status card on the admin page. The tests focus on UI behavior: the 5-layer display, prerequisite-based button disabling, and the Full Cascade button. These tests do NOT run seed agents — they verify the UI renders correctly based on the seed status API response.

The key testing challenge is controlling layer state. Tests need layers to be empty or populated to verify prerequisite logic. Rather than invoking seed agents (which are slow and require AI calls), the tests use the existing layers API to create minimal items directly. The `POST /api/layers/{slug}/items` endpoint creates markdown files in a layer directory without involving any agent — it only requires an authenticated request with a filename, frontmatter, and body.

This step depends on all previous steps being complete (the full UI and API must be working).

## Reuse Opportunities

- **`setupAndLogin(page)`** — Every test calls this to handle registration and login. The same pattern as the existing admin panel tests.

- **`getAdminToken(page)`** — Used to get a JWT token for authenticated API calls to the layers endpoint. The existing helper returns a token string that can be passed as a Bearer token in `page.request.post()` calls.

- **`page.request` API** — Playwright's built-in request context allows making HTTP calls to the server alongside page interactions. Tests can use `page.request.post("/api/layers/{slug}/items", { headers: { Authorization: "Bearer {token}" }, data: { ... } })` to create items in layer directories. This is the cleanest approach — no filesystem manipulation, no shell commands, uses the same API the application uses.

- **Existing test structure** — The 4 existing tests demonstrate the pattern: `test.describe` block, each test calls `setupAndLogin`, navigates to `/admin`, makes assertions. The new block follows the same structure.

- **Test data cleanup** — The global setup clears the database but does NOT clean layer directories. Tests that create items via the layers API need either: (a) a `beforeEach` / `afterEach` hook that deletes created items via `DELETE /api/layers/{slug}/items/{filename}`, or (b) unique filenames per test to avoid conflicts. Option (a) is cleaner — a shared helper that deletes all items from all layers after each test ensures a clean slate.

## Deliverables

### Shared test helper for populating layers

Add a helper function within the test file (or in `helpers.ts` if it would benefit other test files) that creates a minimal markdown item in a specified layer via the layers API. The helper takes a Playwright page context, a JWT token, a layer slug, and optionally a filename. It calls `POST /api/layers/{slug}/items` with a minimal valid payload — a filename, simple frontmatter with a title, and a short body string. This helper is called by tests that need specific layers to be populated before asserting on UI state.

### Shared cleanup helper

Add a helper that removes all test-created items from a layer. It calls `GET /api/layers/{slug}/items` to list items, then calls `DELETE /api/layers/{slug}/items/{filename}` for each. This is called in an `afterEach` hook or at the end of tests that create items, ensuring layer directories don't accumulate stale test data across test runs.

### New test describe block for the seed status card

Add a `test.describe("Seed Status Card", ...)` block to `admin-panel.spec.ts` with the following tests. All tests navigate to `/admin` after login.

**Display tests (empty state):**

1. **Seed card displays all 5 layers** — Verify the seed card is visible and contains the display names for all 5 layers: "Values", "Situational Awareness", "Strategic Objectives", "Tactical Objectives", "Policies". Use text locators to find each name within the card.

2. **Layers are in hierarchical order** — Verify that the 5 layer names appear in the correct order: Values first, then Situational Awareness, Strategic Objectives, Tactical Objectives, and Policies last. Use element bounding box y-coordinates or DOM order to assert ordering — Values must appear above SA, SA above Strategic Objectives, and so on.

3. **Empty layers show not-seeded indicators** — With no items in any layer, verify all 5 status dots indicate the not-seeded state (grey, not green). The `StatusDot` component uses a boolean prop for its active/inactive state, which maps to a CSS color. Assert that no green status indicators are present when all layers are empty.

**Prerequisite and disabled state tests:**

4. **Values and SA seed buttons are enabled when all layers empty** — With no layers populated, verify the seed buttons for Values and Situational Awareness are not disabled. These two layers have no prerequisites and should always be clickable.

5. **Upper-layer seed buttons are disabled when prerequisites empty** — With no layers populated, verify the seed buttons for Strategic Objectives, Tactical Objectives, and Policies are all disabled. All three have unmet prerequisites when the layers below are empty.

6. **Strategic objectives button enables when values and SA populated** — Use the layer population helper to create at least one item in both the `values` and `situational-awareness` layers. Refresh the admin page. Verify the Strategic Objectives seed button is no longer disabled.

7. **Tactical objectives button remains disabled when strategic empty** — With values and SA populated but strategic-objectives empty, verify the Tactical Objectives seed button is still disabled. This confirms the prerequisite chain — having layers 0 and 1 populated is not enough for layer 3.

8. **Policies button enables only when all 4 layers below populated** — Use the population helper to create at least one item in all 4 layers below policies (values, situational-awareness, strategic-objectives, tactical-objectives). Refresh the admin page. Verify the Policies seed button is no longer disabled.

**Populated state tests:**

9. **Seeded layers show green status and count** — Use the population helper to create 2 items in the `values` layer. Navigate to the admin page. Verify the Values row shows a green status indicator and displays a count reflecting the number of items.

10. **Layer count updates after adding items** — Navigate to the admin page and note the count for a layer. Use the population helper to create additional items. Refresh the page. Verify the count has increased.

**Full Cascade button:**

11. **Full Cascade button is present** — Verify the Full Cascade button is visible in the seed card, visually below the layer rows. Use a role-based locator matching the Full Cascade button text from the i18n keys.

**Mutual exclusion during seeding:**

12. **All seed buttons disabled during any seed operation** — This test is optional and may be unreliable in E2E context since it depends on timing. If implemented: click a seed button (e.g., Values, which has no prerequisites), then immediately check that all other seed buttons become disabled while the operation is in progress. This test should be skipped (`test.skip`) if it proves flaky, since the behavior is already covered by unit tests in step 006.

### Layer data cleanup strategy

Tests that create items via the layers API must clean up after themselves. The recommended approach is an `afterEach` hook in the seed status card describe block that deletes all items from all 5 layers using the cleanup helper. This ensures each test starts with empty layers (the default state). Tests that need populated layers create items at the start of their test body, and the `afterEach` hook removes them.

The global setup already handles database cleanup (users, events, etc.) but does not touch layer directories. The `afterEach` hook fills this gap for seed card tests specifically.
