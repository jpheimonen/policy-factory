# 003: Seed status API refactor

## Context

The `GET /api/seed/status` endpoint currently returns a flat object with four hardcoded fields covering only the values and SA layers (`values_seeded`, `values_count`, `sa_seeded`, `sa_count`). This shape cannot accommodate the three new upper layers. This step refactors the response into a list-based structure that reports the seeding status of all 5 layers in hierarchical order.

This is a **breaking change** to the API contract. The frontend consumes this endpoint in `AdminPage.tsx` тАФ it reads the flat fields to render status dots and text. The frontend is updated to consume the new shape in step 006, so between this step and step 006 the admin page seed card will be temporarily broken. That's acceptable since both steps land in the same PR. No action is needed in this step to fix the frontend тАФ step 006 handles that entirely.

## Reuse Opportunities

- The endpoint already uses `list_items(data_dir, slug)` to count items per layer тАФ the same function works for all 5 layers with no modification. The refactor is purely about looping over all layers instead of hardcoding two.

- `LAYERS` from `policy_factory.data.layers` is already imported in `seed.py` indirectly (via `list_items`, `delete_item`). Adding the `LAYERS` import is trivial. Each `LayerInfo` in that list carries `slug` and `display_name` тАФ both needed in the response.

- The existing test fixtures (`data_dir`, `auth_headers`, `client`) in both test files work unchanged. The tests just need their assertions rewritten to navigate the list-based response instead of flat fields.

## Deliverables

### Response model refactor

Replace the `SeedStatusResponse` model with a list-based structure. The response should contain a single `layers` field holding a list of per-layer status entries. Each entry must include:

- **slug** тАФ The layer's identifier string (e.g., `"values"`, `"situational-awareness"`)
- **display_name** тАФ Human-readable name (e.g., `"Values"`, `"Situational Awareness"`)
- **seeded** тАФ Boolean: whether the layer has at least one item
- **count** тАФ Integer: the number of items in the layer

Create a new Pydantic model for the individual layer entry. The `SeedStatusResponse` wraps a list of these entries.

The old flat fields (`values_seeded`, `values_count`, `sa_seeded`, `sa_count`) are removed entirely тАФ no backward compatibility shim.

### Endpoint refactor

Update `get_seed_status` to iterate over the `LAYERS` list from `policy_factory.data.layers`. For each layer, call `list_items` to get the item count and determine the seeded boolean. Build the response list in the same order as `LAYERS` (bottom-to-top: values first, policies last).

The endpoint's URL (`/status`), HTTP method (`GET`), and authentication requirement (`get_current_user` dependency) remain unchanged.
