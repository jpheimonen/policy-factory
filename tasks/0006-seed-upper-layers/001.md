# 001: Agent config and store changes

## Context

This is the foundational step. Every subsequent step — prompt templates, seed endpoints, frontend — depends on the new agent roles being registered and the model resolution supporting `None`. This step makes two changes: (1) adds three new seed agent roles, and (2) switches existing Claude SDK roles from a hardcoded model string to `None` so they use the CLI's default model.

The agent run store also needs updating because `create_agent_run` currently requires a string model name, but roles using the CLI default won't have a model name to pass.

## Related Code

Existing files to modify:

- `src/policy_factory/agent/config.py` — All 6 parallel dictionaries (`AgentRole` literal, `_DEFAULT_MODELS`, `_ENV_VAR_MAP`, `_ALLOWED_TOOLS_BY_ROLE`, `_TOOL_SET_BY_ROLE`, `_USE_SEARCH_BY_ROLE`) plus the 3 resolver functions. Every new role requires an entry in each dictionary. The existing `seed` role entry is the closest template for the new roles (MCP file tools, no WebSearch, no Google Search grounding) except the new roles use `None` for model instead of a Claude model string.

- `src/policy_factory/store/agent_run.py` — The `AgentType` literal, the `AgentRun` dataclass, and the `create_agent_run` method. The `AgentType` literal is separate from `AgentRole` in config.py — they have overlapping but not identical members. The `model` field on `AgentRun` and the `model` parameter on `create_agent_run` are both typed as `str` and need to become `str | None`.

- `tests/test_agent_config.py` — 59 existing tests covering `resolve_model`, `resolve_allowed_tools`, `resolve_tool_set`, `resolve_use_search`, and `AgentConfig`. Tests for existing roles that assert specific model strings (like asserting `generator` returns `claude-sonnet-4-20250514`) will need updating to assert `None` instead.

## Reuse Opportunities

- The existing `seed` role entry across all 6 dictionaries is the pattern to follow for the 3 new roles. The only differences: the new roles use `None` for model (not a Claude model string), and they get `[MCP_SERVER_REF]` for allowed tools (no `"WebSearch"`).

- The `_ENV_VAR_MAP` naming convention is `POLICY_FACTORY_MODEL_{ROLE_UPPER}` with hyphens converted to underscores. Follow this for the new roles.

- The existing test structure in `test_agent_config.py` has separate test classes for each resolver function. New role assertions should be added to the existing test classes rather than creating new ones.

## Deliverables

### Agent Config Changes

Add three new roles — `strategic-seed`, `tactical-seed`, `policies-seed` — to all 6 dictionaries in `config.py`:

- **AgentRole literal**: Add the 3 new role strings.
- **_DEFAULT_MODELS**: Set to `None` for all 3 new roles. Also change the 4 existing Claude SDK roles (`generator`, `critic`, `heartbeat-sa-update`, `seed`) from `"claude-sonnet-4-20250514"` to `None`. The dict value type changes from `str` to `str | None`.
- **_ENV_VAR_MAP**: Add env var names following the existing convention.
- **_ALLOWED_TOOLS_BY_ROLE**: All 3 new roles get `[MCP_SERVER_REF]` (file tools only, no WebSearch).
- **_TOOL_SET_BY_ROLE**: All 3 new roles get `TOOL_SET_FULL`.
- **_USE_SEARCH_BY_ROLE**: All 3 new roles get `False`.

Update `resolve_model()` so its return type is `str | None` — when the default is `None` and no env var override is set, it returns `None`.

### Agent Run Store Changes

- Add the 3 new seed role names to the `AgentType` literal. Also add `values-seed` to close the existing gap — `AgentRole` in config.py already includes `values-seed` but `AgentType` in the store does not. Adding it here prevents future issues when the values seed records an agent run.
- Change the `model` field on the `AgentRun` dataclass from `str` to `str | None`.
- Change the `model` parameter on `create_agent_run` from `str` to `str | None`.

### Test Updates

- Update existing `resolve_model` tests for `generator`, `critic`, `heartbeat-sa-update`, and `seed` to expect `None` instead of `"claude-sonnet-4-20250514"`.
- Add `resolve_model` tests for the 3 new roles: default returns `None`, env var override returns the override value.
- Add `resolve_allowed_tools` tests for the 3 new roles: returns MCP server reference without WebSearch.
- Add `resolve_tool_set` tests for the 3 new roles: returns full tool set.
- Add `resolve_use_search` tests for the 3 new roles: returns false.
- Verify that Gemini-backed roles are unaffected (still return their hardcoded model strings).
- Add store tests verifying `create_agent_run` accepts `None` model and that the run can be retrieved with `model=None`.
